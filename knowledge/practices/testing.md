# テストプラクティス

> **プロジェクト CLAUDE.md の指定が優先**。ここは汎用デフォルト。
> IPA知識ベース参照: `ipa/sm.md` §テスト管理

## 1. テスト駆動開発（TDD）— ソフトウェア開発の原則

適用対象: API・UI・ユーティリティ・ビジネスロジック等、**入出力が明確な**コード

### 手順
1. 期待される入出力に基づき、**テストを先に書く**
2. テストを実行し、**失敗を確認**する（Red）
3. テストが正しいことを確認したらコミット
4. テストをパスさせる**最小限の実装**を書く（Green）
5. 実装中はテストを変更しない
6. 全テスト通過後にリファクタリング（Refactor）
7. Red → Green → Refactor を繰り返す

### 注意
- テストが先、実装が後。順序を逆にしない
- テストを通すためにテストを変えない（実装を変える）
- 1テスト = 1つの振る舞いを検証

## 2. 予測モデル検証 — TDDではなく仮説検証型

適用対象: 機械学習・統計予測・推薦等、**正解が事前に分からない**もの

### なぜTDDが適用できないか
- 予測モデルの「正しい出力」は事前に定義できない
- 精度指標（RMSE, AUC等）の許容値は実験で決まる
- データ分布の変化で期待値が変わる

### 仮説検証型ワークフロー
1. **仮説を立てる**: 「この特徴量を追加すればAUCが0.02以上改善する」
2. **ベースラインを記録**: 現行モデルの指標を固定
3. **実験する**: 仮説に基づく変更を実施
4. **評価する**: ベースラインとの比較（統計的有意性を確認）
5. **判断する**: 採用/棄却/追加実験の3択
6. **記録する**: 実験条件・結果・判断理由をドキュメント化

### テスト可能な部分はテストする
- データパイプライン（入力形式・欠損処理・型変換）→ TDD
- 特徴量エンジニアリング（計算ロジック）→ TDD
- モデルの入出力インターフェース（型・shape）→ TDD
- 予測精度・ビジネスKPI → 仮説検証型

## 3. バグ修正手順

1. **再現**: バグを再現するテストを書く（失敗を確認）
2. **原因特定**: デバッグで根本原因を特定
3. **修正**: テストが通る最小限の修正を実施
4. **回帰確認**: 既存テスト全件が通ることを確認
5. **水平展開**: 同じパターンのバグが他にないか検索

## 4. カバレッジ戦略

- **新規コード**: テストを書いてからコードを書く（TDD）
- **既存コード修正**: 修正対象にテストがなければ先に追加
- **リファクタリング**: 変更前にテストで現状の振る舞いを固定
- **優先度**: ビジネスクリティカルなパス > エッジケース > ユーティリティ

## 5. テストの3層構造（`quality/review.md` 連携）

| 層 | 対象 | タイミング |
|---|---|---|
| ユニットテスト | 関数・モジュール単位 | コード変更の都度 |
| 統合/E2Eテスト | API・画面フロー | PR作成時・デプロイ前 |
| 本番動作確認 | 実環境での稼働 | デプロイ後 |
