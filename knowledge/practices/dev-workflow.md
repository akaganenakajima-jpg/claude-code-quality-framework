# 開発ワークフロー

> **プロジェクト CLAUDE.md の指定が優先**。ここは汎用デフォルト。
> 品質プロセス連携: `quality/process.md`

## 全体フロー（PDCA対応）

```
受入 → 調査 → 計画(Plan) → 実装(Do) → 検証(Check) → 改善(Act)
```

## 1. 受入（タスクを受けたとき）

- タスクの目的と完了条件を確認する
- 不明点があれば先に質問する（推測で進めない）
- 関連する知識ファイルを参照する（CLAUDE.md §知識ベース自動参照）

## 2. 調査（実装前の必須ステップ）

- 同じ機能が既にコードベースにないか検索する
- 使用ライブラリに同じ機能がないか確認する
- 変更対象のコードを読んで設計意図を理解する
- 影響範囲を特定する（どこが壊れる可能性があるか）

## 3. 計画（Plan）

- 受入基準を明文化する（「何が達成されたら完了か」）
- リスク判定を実施する（`quality/risks.md`）
- 実装方針を決める（どのファイルをどう変えるか）
- テスト方針を決める（何をどうテストするか）

## 4. 実装（Do）

### ソフトウェア開発の場合（TDD）
1. テストを先に書く → 失敗を確認
2. テストが通る最小限のコードを書く
3. リファクタリング
4. 詳細手順 → `testing.md`

### 予測モデル・実験の場合（仮説検証型）
1. 仮説を立てる
2. ベースラインを記録
3. 実験を実施
4. 結果を評価
5. 詳細手順 → `testing.md` §予測モデル検証

### 共通
- 1機能/1バグ修正 = 1コミット
- 品質ゲートを順守する（`quality/gates.md`）

## 5. 検証（Check）

- ユニットテスト → 統合テスト → 本番動作確認（3層）
- 修正後は `git diff HEAD` で変更内容を確認
- 変更の要点をユーザーに報告する
- 詳細 → `quality/review.md`

## 6. 改善（Act）

- 問題が出たら根本原因を分析する
- 是正処置を実施する
- 同じパターンの問題が他にないか確認する（水平展開）
- 詳細 → `quality/nonconformity.md`
