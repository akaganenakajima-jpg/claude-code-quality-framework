# Git ワークフロー

> **プロジェクト CLAUDE.md の指定が優先**。ここは汎用デフォルト。
> IPA知識ベース参照: `ipa/pm.md` §構成管理

## 1. ブランチ戦略（GitHub Flow）

- `main` は常にデプロイ可能な状態を保つ
- 変更は feature ブランチで行い、PR を経由して main にマージする
- ブランチ名: `feature/xxx`, `fix/xxx`, `refactor/xxx`
- 長寿ブランチ禁止（1-2日でマージ or クローズ）

## 2. コミット規約

- **粒度**: 1機能 or 1バグ修正 = 1コミット（巨大コミット禁止）
- **メッセージ**: 「なぜ」を中心に書く（「何を」はdiffで分かる）
- **形式**: 動詞で始める（Add / Fix / Update / Remove / Refactor）
- **巨大になる場合**: 論理的な単位で分割コミット

## 3. 変更前の安全策

- 大きな変更の前に現状をコミットして戻れるようにする
- リスクの高い変更は Worktree（隔離環境）で作業する
- `git diff HEAD` で変更内容を確認してからコミット

## 4. PR（プルリクエスト）

- タイトル: 変更の要約（70文字以内）
- 本文: 背景・変更内容・テスト計画
- セルフレビュー: 提出前に自分のdiffを確認
- テスト通過: CI/テストが全て通ってからマージ

## 5. ロールバック

- デプロイ前にロールバック手順を確認する
- 問題発生時は即座に前バージョンに戻せる状態を維持
- ロールバック実施後に原因調査（ロールバック自体が修正ではない）

## 6. リスクレベル別の運用

| リスク | 対応 |
|---|---|
| 🟢 低 | 通常フロー（ブランチ→PR→マージ） |
| 🟡 中 | 先にコミットしてから着手 |
| 🟠 高 | 影響範囲を洗い出してから着手 |
| 🔴 最高 | Worktree（隔離環境）で作業 |
