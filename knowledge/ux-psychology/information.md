# 情報設計（Information Design）— UX心理学ナレッジベース

> 情報をどう構造化し、どう提示するかに関わる8つの心理学原則。
> UIの「わかりやすさ」を科学的根拠に基づいて設計するためのリファレンス。

---

## §1. 認知負荷（Cognitive Load）

### 1.1 心理学的メカニズム

Sweller (1988) の認知負荷理論は、人間の作業記憶が処理できる情報量に限界があることを前提とする。認知負荷は3種に分類される。

| 種類 | 英語名 | 説明 | UI設計との関係 |
|---|---|---|---|
| 内在的負荷 | Intrinsic Load | タスク自体の複雑さに由来する負荷 | タスクの分割・段階化で軽減 |
| 外在的負荷 | Extraneous Load | 不適切な情報提示が生む不要な負荷 | レイアウト・表現の最適化で除去 |
| 学習促進負荷 | Germane Load | 理解を深めるために使われる有益な負荷 | チュートリアル・ヒントで支援 |

Miller (1956) は短期記憶の容量を「7±2チャンク」と提唱したが、Cowan (2001) の研究により、実用的な限界は **4チャンク** であることが示された。これはUIにおけるグルーピングの上限設計に直結する。

### 1.2 UIへの適用

| 手法 | 実装例 | 効果 |
|---|---|---|
| チャンキング | 電話番号を `090-1234-5678` と区切る | 記憶・入力の負担を軽減 |
| ウィザードフォーム | 長いフォームを3〜5ステップに分割 | 各画面の判断項目を4以下に抑制 |
| 視覚的グルーピング | カード・セクション区切り・余白 | 関連情報のまとまりを瞬時に認識 |
| 装飾的要素の排除 | 意味のないアイコン・背景パターンを削除 | 外在的負荷の低減 |
| インラインバリデーション | 入力直後にエラーを表示 | 記憶に頼らずその場で修正可能 |

### 1.3 Apple HIG との接点

- **Clarity（明瞭さ）原則**: テキストは読みやすく、アイコンは正確で、装飾は控えめに、機能主導のデザインを推奨
- **SF Symbols**: テキストラベルをアイコンで代替することで視覚的に処理可能な情報量を増加
- **Typography**: San Francisco フォントの Dynamic Type により、ユーザーの読みやすさを個別最適化

### 1.4 アンチパターン

- **過剰な単純化**: 必要な情報まで削ってしまい、ユーザーが正しい判断に必要なコンテキストを失う
- **1画面に4つ以上の独立した意思決定**: 各決定が干渉し合い、判断精度が著しく低下する
- **情報密度の過不足**: 密度が高すぎると外在的負荷増大、低すぎるとスクロール負荷で内在的負荷増大

### 1.5 実務活用

- フォーム設計時: 1ステップあたりの入力フィールド数を4以下に制限する
- ダッシュボード設計時: ウィジェット数を1画面7個以下、各ウィジェット内のKPIを3個以下に
- ナビゲーション設計時: 第一階層のメニュー項目を5±2個に収める
- 判断材料: 「このUIで同時に処理すべき情報は何チャンクか？」を常に数える

→ 関連: `choice.md` §7（決断疲れ — 認知負荷の蓄積で判断力が低下）, `motivation.md` §4（ツァイガルニク効果 — 未完了タスクの認知負荷）

## §2. 段階的開示（Progressive Disclosure）

### 2.1 心理学的メカニズム

Nielsen (2006) が体系化した段階的開示の原則は、ユーザーに**必要な情報だけを適切なタイミングで提示**し、詳細は要求に応じて開示するアプローチである。認知負荷理論を情報提示に応用した実践的手法であり、以下の心理学的前提に基づく。

- 人間は一度にすべてを処理できない（作業記憶の限界）
- 情報の全量を見せると「どこから始めるべきか」の判断自体が負荷になる
- 段階的に開示することで、各時点での認知負荷を管理可能な水準に保つ

### 2.2 UIへの適用

| パターン | 実装例 | 適用場面 |
|---|---|---|
| アコーディオン/展開 | FAQ、設定画面の詳細オプション | 情報量が多く優先度に差がある場合 |
| ツールチップ/ポップオーバー | 専門用語の説明、フィールドのヘルプテキスト | 補足情報を必要な人だけに提供 |
| マルチステップフォーム | 会員登録、購入フロー、申請手続き | 入力項目が10以上ある場合 |
| 「もっと見る」ボタン | 商品レビュー、検索結果の追加読み込み | 初期表示量を適正に保つ |
| コンテキストメニュー | 右クリック、長押しによる詳細操作 | 上級者向け機能の隠蔽 |

### 2.3 Apple HIG との接点

- **Navigation Hierarchy**: NavigationStack による階層的な画面遷移で、情報を段階的にドリルダウン
- **Disclosure Group**: SwiftUI の標準コンポーネントとして展開/折りたたみを提供
- **Sheet/Popover**: 補足情報をモーダルまたはポップオーバーで表示し、メイン画面の情報量を抑制

### 2.4 アンチパターン

- **重要情報の過剰隠蔽**: 安全上の警告や必須情報を「もっと見る」の中に隠してしまう
- **クリック地獄**: 目的の情報に到達するまで4回以上のクリック/タップが必要になる
- **不一致な開示基準**: 同じ階層の情報が、あるものは表示されあるものは隠されていて基準が不明

### 2.5 実務活用

- 設定画面: 「基本設定」はデフォルト表示、「詳細設定」はアコーディオンで隠す
- エラーメッセージ: 簡潔なメッセージを表示し、「詳細を見る」で技術情報を開示
- 判断基準: 「80%のユーザーが必要とする情報か？」で初期表示/開示を分ける

→ 関連: `information.md` §1（認知負荷 — 段階的開示の理論的基盤）, `motivation.md` §6（フット・イン・ザ・ドア — 小さなステップから始める原則）

## §3. 視覚的階層（Visual Hierarchy）

### 3.1 心理学的メカニズム

ゲシュタルト心理学（Wertheimer, 1923）の知覚組織化の法則が視覚的階層の基盤となる。

| 法則 | 説明 | UIでの活用 |
|---|---|---|
| 近接の法則（Proximity） | 近いものはグループとして知覚される | 関連要素を近くに配置、余白でセクション分離 |
| 類似の法則（Similarity） | 似たものは同じグループと知覚される | 同種の操作に統一したスタイルを適用 |
| 図と地（Figure-Ground） | 前景と背景を自動的に分離して知覚する | モーダル表示時の背景暗転、フォーカス状態 |
| 閉合の法則（Closure） | 不完全な形でも補完して全体を認識する | カードUI、アイコンデザイン |
| 連続の法則（Continuity） | 滑らかに続くパターンを追従する | プログレスバー、タイムライン |

サイズ・色・コントラスト・位置・余白の5つの要素が読み取り順序（視覚的ウェイト）を決定する。

### 3.2 UIへの適用

| 要素 | 高優先度 | 低優先度 |
|---|---|---|
| サイズ | 大きいフォント/ボタン | 小さいフォント/リンク |
| 色 | ブランドカラー、高彩度 | グレー、低彩度 |
| コントラスト | 暗背景に白テキスト | 薄いグレーテキスト |
| 位置 | 左上（F-pattern）、中央上部（Z-pattern） | 右下、フッター領域 |
| 余白 | 広い余白で孤立させる | 密集したレイアウト |

**F-pattern**: テキスト主体のページ（記事、検索結果）で目線が左上から右へ、次に下へ移動する。
**Z-pattern**: ビジュアル主体のページ（LP、広告）で目線がZ字に移動する。

### 3.3 Apple HIG との接点

- **Typography Scale**: Large Title (34pt) → Title 1 (28pt) → Body (17pt) → Caption (12pt) の4段階で明確な情報階層を構成
- **SF Pro Weight System**: Ultralight〜Black の9ウェイトで視覚的ウェイトを細かく制御
- **Semantic Colors**: `label`, `secondaryLabel`, `tertiaryLabel`, `quaternaryLabel` の4段階で自動的に階層を表現

### 3.4 アンチパターン

- **均一な強調**: すべてを太字・大きく・カラフルにすると、何も目立たなくなる
- **偽の階層**: サイズや色に意味の裏付けがなく、視覚的なノイズになっている
- **アクション競合**: Primary / Secondary / Tertiary の区別なくボタンが並ぶ

### 3.5 実務活用

- ボタン設計: 1画面に Primary ボタンは1つのみ、Secondary は2つまで、それ以外は Tertiary（テキストリンク）
- 見出し設計: H1→H2→H3 のフォントサイズ比率を 1.5:1.25:1.0 程度に設定
- カラー設計: ブランドカラーの使用を全体面積の10%以下に制限し、アクセントとして機能させる

→ 関連: `information.md` §6（系列位置効果 — 配置の優先順位）, `impression.md` §1（美的ユーザビリティ — 視覚的美しさと使いやすさの関連）

#### §3.1 視覚的アンカー（Visual Anchor）

視覚的アンカーとは、画面内で最初に目を引き、視線の流れの起点となるフォーカルポイントである。

- **ヒーローイメージ**: LPやプロダクトページの大面積画像が視線を固定し、そこから情報を読み始めるガイドになる
- **アイコン/イラスト**: リスト項目やカード内の視覚的アンカーが情報の取っ掛かりを提供する
- **コントラストカラー**: 背景色と大きくコントラストを持つアクセント要素が自然に視線を集める
- **空間的孤立**: 周囲に余白を持たせた要素は図と地の原則により前景として強く認識される

→ 関連: `choice.md` §1（アンカー効果 — 数値的アンカリングとの比較。視覚的アンカーは注意の起点、数値アンカーは判断基準の起点）

## §4. 選択的注意（Selective Attention）

### 4.1 心理学的メカニズム

Cherry (1953) のカクテルパーティー効果は、騒がしい環境でも特定の会話に集中できる能力を示した。人間の注意資源は限られており、目標に関連する情報を選択的に処理し、無関連な情報をフィルタリングする。

Simons & Chabris (1999) の「見えないゴリラ」実験は、注意の盲点（Inattentional Blindness）を実証した。特定のタスクに集中しているとき、予期しない刺激は文字通り「見えない」。これはUIにおいて、ユーザーが目的外の情報を体系的に無視することを意味する。

### 4.2 UIへの適用

| 手法 | 実装例 | 効果 |
|---|---|---|
| 関連アクションのハイライト | コンテキストに応じてボタンの有効/無効を切替 | ユーザーの目標に沿った選択肢を強調 |
| ノイズの削減 | 現在のタスクに不要な要素を非表示/縮小 | 注意資源を目的に集中させる |
| コンテキストメニュー | 選択したオブジェクトに応じた操作のみ表示 | 不要な選択肢によるフィルタリング負荷を除去 |
| 変化の検出 | アニメーション、色の変化、バッジ表示 | 選択的注意の「変化への感度」を活用 |
| 聴覚・触覚フィードバック | 通知音、Haptic Feedback | 視覚的注意の外にある情報を別チャネルで伝達 |

### 4.3 Apple HIG との接点

- **Deference（控えめさ）原則**: UIはコンテンツを支え、コンテンツと競合しない。装飾的なクロム要素を最小化
- **Focus System**: tvOS/macOS のフォーカスシステムにより、現在の注意対象を明確に視覚化
- **Contextual Menus**: 長押しで表示されるメニューが、コンテキストに応じた選択肢のみを提示

### 4.4 アンチパターン

- **注意の分散**: バナー、ポップアップ、アニメーションが同時に動き、どこに集中すべきかわからない
- **無関連な選択肢の提示**: 現在のタスクに不要なナビゲーションやプロモーションが注意を奪う
- **重要通知の埋没**: 既存のノイズが多すぎて、本当に重要な通知がフィルタリングされてしまう

### 4.5 実務活用

- 通知設計: 緊急度を3段階（Critical/Warning/Info）に分類し、視覚的差別化を明確に
- モーダル設計: ユーザーの注意を完全に引き付ける必要がある場合のみ使用し、濫用を避ける
- 判断基準: 「今、ユーザーの目標は何か？この要素はその目標達成に貢献するか？」

→ 関連: `information.md` §5（バナーブラインドネス — 選択的注意の帰結としての広告無視）

## §5. バナーブラインドネス（Banner Blindness）

### 5.1 心理学的メカニズム

Benway & Lane (1998) の研究により、ユーザーがバナー広告のような外観を持つ要素を**体系的に無視**することが実証された。これは選択的注意の学習された応用であり、ウェブ利用経験を通じてユーザーは「広告らしい領域は無視して構わない」というスキーマを獲得する。

アイトラッキング研究（Nielsen Norman Group, 2007）では以下の要素がバナーブラインドネスを誘発することが確認されている。

- ページ上部の横長の矩形領域
- 鮮やかな色のボーダーで囲まれた領域
- アニメーションやフラッシュ要素
- 本文コンテンツと視覚的に分離された「島」状の要素
- 「広告」「PR」「スポンサー」の近くに配置された要素

### 5.2 UIへの適用

| 方針 | 実装 | 理由 |
|---|---|---|
| ネイティブUI準拠 | 重要な通知をOS標準のアラートスタイルで表示 | 広告と誤認されない |
| インライン配置 | お知らせをコンテンツフロー内に自然に配置 | 本文の一部として認知される |
| テキスト主体 | 画像やバナーより本文テキストでの告知 | テキストはコンテンツとして処理される |
| 段階的強調 | 最初は控えめ、無視された場合に段階的に強調 | 初回の広告誤認を回避 |
| パーソナライズ | ユーザー名や状態に応じた動的メッセージ | 広告の「汎用性」と区別される |

### 5.3 Apple HIG との接点

- **Consistency（一貫性）**: プラットフォームの慣例に従った通知スタイルはコンテンツとして認識される
- **標準コンポーネント**: `UIAlertController`、`NotificationCenter` などのシステム標準を使用することで広告と区別される
- **App Store Review Guidelines**: 広告と機能の混同を禁止しており、バナーブラインドネスの回避にも寄与

### 5.4 アンチパターン

- **重要通知をバナースタイルで表示**: セキュリティ警告やシステムメンテナンス通知が広告と誤認され無視される
- **過剰なプロモーション**: アプリ内バナーが多すぎてすべてが無視対象になり、本当の通知まで見逃す
- **フルスクリーンインタースティシャル**: ユーザーの目標達成を阻害し、即座に閉じられる

### 5.5 実務活用

- 重要告知: バナー形式を避け、インラインテキストまたはモーダルダイアログで表示
- プロモーション: コンテンツと同じ視覚言語（フォント、色、レイアウト）を使い、情報として自然に溶け込ませる
- 判断基準: 「この要素を広告だと思ったユーザーは何%いるか？」を常にテストで検証

→ 関連: `information.md` §4（選択的注意 — バナーブラインドネスの認知的基盤）

## §6. 系列位置効果（Serial Position Effect）

### 6.1 心理学的メカニズム

Ebbinghaus (1885) が発見した系列位置効果は、リスト内の項目の記憶がその位置に依存することを示す。

| 効果 | 位置 | メカニズム | 記憶の持続 |
|---|---|---|---|
| 初頭効果（Primacy） | リストの最初 | リハーサル回数が多く長期記憶に転送 | 長期間持続 |
| 新近効果（Recency） | リストの最後 | 短期記憶にまだ保持されている | 短期的 |
| 中間効果 | リストの中央 | どちらの恩恵も受けない | 最も忘れやすい |

この記憶曲線はU字型を描き、先頭と末尾の項目が中央の項目より有意に高い再生率を示す。

### 6.2 UIへの適用

| 配置戦略 | 実装例 | 根拠 |
|---|---|---|
| 先頭に最重要項目 | ナビゲーションの先頭にホーム/主要機能 | 初頭効果で長期記憶に定着 |
| 末尾に重要アクション | タブバーの右端にプロフィール/設定 | 新近効果で直近の認知に残る |
| 中央は補助機能 | 検索、ブックマーク等の補助的タブ | 使用頻度が中程度の機能を配置 |
| リスト冒頭の強調 | 検索結果の上位3件を視覚的に差別化 | 初頭効果を意図的に強化 |
| 最終印象の演出 | 完了画面にポジティブなメッセージ | 新近効果で体験の印象を向上 |

### 6.3 Apple HIG との接点

- **Tab Bar Design**: 最も使用頻度の高い機能を両端に配置し、中央を補助機能に割り当てるガイドラインはこの効果に基づく
- **Navigation Bar**: 左端に「戻る」、右端にアクションボタンを配置する慣例は両端の認知的優位性を活用
- **List Design**: セクションヘッダーの最初の項目が自然に注目を集める

### 6.4 アンチパターン

- **重要アクションのリスト中央配置**: 「購入」「確認」などの重要ボタンがリストの中間にあると見逃される
- **ナビゲーション中央に主要機能**: タブバーの3番目（5タブの場合）にメイン機能を置くと利用率が低下
- **均一なリスト表示**: 先頭・末尾の強調がないと系列位置効果を活用できない

### 6.5 実務活用

- タブバー設計: 5タブの場合、1番目と5番目に最重要機能を配置
- メニュー設計: 「ログアウト」「設定」など記憶に残したい項目をリスト末尾に
- コンテンツ設計: 記事やメールの冒頭と結末に最も伝えたいメッセージを配置
- 判断基準: 「このリストで最も重要な項目は先頭か末尾にあるか？」

→ 関連: `information.md` §3（視覚的階層 — 位置による優先度の表現）

## §7. デフォルトバイアス（Default Bias）

### 7.1 心理学的メカニズム

Johnson & Goldstein (2003) の臓器提供研究は、デフォルト設定の圧倒的な影響力を実証した。オプトイン（自分から申し込む）方式の国では臓器提供率が4〜28%であるのに対し、オプトアウト（自分から辞退する）方式の国では85〜100%であった。

このバイアスには複数の心理学的要因が関与する。

| 要因 | 説明 |
|---|---|
| 現状維持バイアス（Status Quo Bias） | 変更にはコストがかかると知覚し、現状を維持する傾向 |
| 省力化（Effort Minimization） | デフォルトを変更する認知的・操作的コストを避ける |
| 暗黙の推奨（Implied Recommendation） | 「設計者がこれを推奨している」と解釈する |
| 損失回避（Loss Aversion） | デフォルトを変更することで何かを失うリスクを避ける |

### 7.2 UIへの適用

| 手法 | 実装例 | 効果 |
|---|---|---|
| 推奨オプションのプリセレクト | プラン選択で「おすすめ」を初期選択 | 選択の負荷軽減+コンバージョン向上 |
| スマートデフォルト | 地域に応じた通貨・言語の自動設定 | 設定変更の手間を排除 |
| フォームの事前入力 | 過去の入力や推定値による自動入力 | 入力コストの削減 |
| プライバシー設定のデフォルト | プライバシー保護をデフォルトON | ユーザーの利益を守る倫理的設計 |
| 通知設定の初期値 | 重要通知のみON、マーケティングはOFF | 通知疲れの防止と信頼構築 |

### 7.3 Apple HIG との接点

- **Settings Design**: 「ほとんどのユーザーに適した設定をデフォルトにする」を明示的にガイドライン化
- **Form Pre-fill**: AutoFill、キーチェーンとの連携によりフォームの事前入力を推奨
- **Privacy by Default**: App Tracking Transparency ではトラッキング拒否がデフォルトの立場を採用

### 7.4 アンチパターン

- **ダークパターン（事前チェック）**: 「マーケティングメールを受け取る」がデフォルトON — ユーザーの信頼を損なう
- **オプトアウトの隠蔽**: 解除手順を複雑にし、デフォルトバイアスを悪用する
- **不利益なデフォルト**: ユーザーの利益に反する設定をデフォルトにする（高額プランの初期選択など）
- **GDPR/個人情報保護法違反**: EU圏ではマーケティング同意のデフォルトONは法的に禁止

### 7.5 実務活用

- 倫理的判断基準: 「このデフォルトはユーザーの利益になるか、それとも事業者の利益だけか？」
- プラン選択: ユーザーの利用パターンデータに基づき、最もコスパの良いプランをデフォルトに
- 設定画面: デフォルト値の選定根拠をドキュメント化し、レビュー可能にする
- A/Bテスト: デフォルト変更の影響は非常に大きいため、変更前後で必ず効果測定を行う

→ 関連: `choice.md` §6（ナッジ — デフォルト設計はナッジの最も強力な手法の一つ）

## §8. ドハティ閾値（Doherty Threshold）

### 8.1 心理学的メカニズム

Doherty & Thadani (1982) はIBMでの研究で、コンピュータの応答時間が **400ミリ秒** を下回ると、ユーザーの生産性が飛躍的に向上することを発見した。この閾値を境に、ユーザーとシステムの相互作用が「待機→応答」の断続的なものから、**連続的なフロー状態**に変化する。

| 応答時間 | ユーザーの知覚 | 行動への影響 |
|---|---|---|
| 0〜100ms | 即座の応答 | 直接操作の感覚。フロー状態が最も維持される |
| 100〜400ms | わずかな遅延 | 許容範囲。大部分のユーザーが連続操作を維持 |
| 400ms〜1s | 明確な遅延 | 集中力の低下が始まる。コンテキストの一部を忘れ始める |
| 1〜10s | 長い待機 | 注意が逸れる。進捗インジケーターが必須 |
| 10s以上 | 離脱リスク | タスクの放棄率が急増。ユーザーは別のタブや行動に移る |

Nielsen (1993) も同様に、0.1秒・1秒・10秒の3つの閾値を応答時間設計の基準として提唱した。

### 8.2 UIへの適用

| 手法 | 実装例 | 効果 |
|---|---|---|
| スケルトンスクリーン | コンテンツ読み込み中にレイアウト骨格を表示 | 知覚される待機時間を短縮 |
| オプティミスティックUI | 「いいね」のリクエスト送信前に即座にUI反映 | 応答をゼロに近づける |
| プリフェッチ | 次に表示されそうなデータを事前に取得 | 遷移時の待機を排除 |
| プログレスバー | 長時間処理の進捗を視覚化 | 待機の不確実性を解消 |
| 即時フィードバック | タップ時のハイライト、Haptic Feedback | 操作が受け付けられたことを即座に伝達 |
| 遅延のマスキング | アニメーションで遷移を演出しつつ裏で読み込み | 待機時間を演出に置換 |

### 8.3 Apple HIG との接点

- **Responsive Touch Feedback**: すべてのタッチ操作に即座の視覚的フィードバックを要求
- **Loading Indicators**: `UIActivityIndicatorView`、`ProgressView` によるシステム標準の進捗表示
- **Animation**: 画面遷移のアニメーション（0.3秒程度）が読み込み時間をマスクする設計パターン
- **Launch Screen**: アプリ起動時にスケルトンUIを表示し、体感的な起動速度を向上

### 8.4 アンチパターン

- **偽のインスタントローディング**: 実際の処理が完了していないのに完了したように見せ、後からエラーを表示する（信頼の毀損）
- **不正確なプログレスバー**: 進捗が90%で長時間停止するなど、実際の処理と乖離した表示
- **過剰なアニメーション**: 待機時間のマスクが目的だったアニメーションが、かえって操作を遅く感じさせる
- **スケルトンの長時間表示**: スケルトンスクリーンが5秒以上続くと、逆にイライラの原因になる

### 8.5 実務活用

- パフォーマンスバジェット: ページの初回描画（FCP）を400ms以内、操作応答（FID）を100ms以内に設定
- 測定指標: Core Web Vitals（LCP, FID, CLS）をドハティ閾値の観点で監視
- 段階的改善: まず知覚を改善（スケルトン、オプティミスティックUI）→ 次に実際の速度を改善（キャッシュ、CDN）
- トレードオフ: オプティミスティックUIの失敗時のロールバックUXも事前に設計する

→ 関連: `impression.md` §5（労働の錯覚 — 待機時間に「処理の努力」を見せることで価値を感じさせる手法）
