# 動機設計 — UX心理学リファレンス

> ユーザーの内発的・外発的動機を活性化し、エンゲージメントと継続利用を促すための心理学法則。
> 各セクション末尾に「実務活用」を収録。

---

## §1. 目標勾配効果（Goal Gradient Effect）

> ゴールに近づくほど、人はより速く・より積極的に行動する。

### 1.1 心理学的メカニズム

Clark Hull（1932）が動物実験で発見した法則。迷路を走るラットは、餌（報酬）に近づくほど走行速度が加速する。この「ゴールへの接近が動機を高める」現象は、人間の行動にも広く当てはまる。

Ran Kivetz ら（2006）のコーヒーロイヤルティカード研究が有名。10杯で1杯無料のスタンプカードにおいて、残りスタンプ数が少なくなるほど購入頻度が上がることを実証した。さらに「12杯中2杯済み」のカードは「10杯中0杯済み」のカードより完了率が高かった（Endowed Progress Effect）。

**核心**: 人間は「進捗が見える」状態でゴールに近い場合、動機が非線形に加速する。進捗の可視化と適切なゴール距離設計が鍵となる。

### 1.2 UIへの適用

| パターン | 実装例 | 効果 |
|---|---|---|
| プログレスバー | プロフィール完成度「あと2項目で完了」 | 完了率を 20〜30% 向上させる |
| ステップインジケーター | 購入フロー「ステップ 3/4」 | 離脱率を低減、現在位置の明示 |
| マイルストーン可視化 | 学習アプリ「次のバッジまであと3レッスン」 | 中間目標で動機を分散・維持 |
| 初期進捗の付与 | 新規登録時「すでに20%完了！」 | Endowed Progress Effect で着手率向上 |
| カウントダウン表示 | 「あと50ポイントで次のランク」 | 残り量の減少が加速感を生む |

### 1.3 Apple HIG との接点

- **Activity Rings（Apple Watch）**: 円形プログレスがゴールへの接近を直感的に伝える。リングが閉じる直前にハプティクスで加速感を演出
- **Progress Indicators**: HIG はプログレスバーに「残り時間/量の推定」を併記することを推奨。目標勾配効果を最大化する設計
- **Workout アプリ**: ゴールの 90% 到達時に通知を送り、最後の追い込みを促進

### 1.4 アンチパターン

| アンチパターン | 問題点 | 改善策 |
|---|---|---|
| フェイク進捗（初期20%表示） | 操作的だと気づかれると信頼を失う | 実際の完了項目に基づく真の進捗を表示 |
| ゴールが遠すぎる | 「0/1000ポイント」は絶望感を生む | 中間マイルストーンを設置（100pt刻み等） |
| 線形的で変化のない進捗表示 | 単調な進行は動機を維持しない | 節目ごとにビジュアル変化やお祝い演出を追加 |
| 進捗リセット | 苦労して貯めた進捗のリセットは離脱原因 | 累積型の設計。リセットするなら明確な理由と代替報酬を |

### 1.5 実務活用

- **オンボーディング設計**: 全体ステップ数を明示し、最初の1〜2ステップを極端に簡単にする（名前入力だけで30%完了）
- **ロイヤルティプログラム**: ランク制度では次のランクまでの距離を常に表示。ランク間距離は均等ではなく後半を短くする
- **フォーム最適化**: 長いフォームはステップ分割し、各ステップの完了を視覚フィードバックで強化
- **KPI計測**: 進捗バー表示あり/なしでタスク完了率を A/B テストする

→ 関連: `motivation.md` §4（ツァイガルニク — 未完了タスクが緊張を生む）, `choice.md` §8（サンクコスト — 投じた労力が継続動機になる）

## §2. 変動型報酬（Variable Reward）

> 予測不能な報酬パターンが、最も強い習慣形成とエンゲージメントを生む。

### 2.1 心理学的メカニズム

B.F. Skinner（1957）のオペラント条件付け研究が基盤。強化スケジュールの中で、**変動比率スケジュール**（Variable-Ratio Schedule）が最も高い反応率と消去抵抗を示す。スロットマシンが典型例 — 次にいつ当たるか分からないからこそ、レバーを引き続ける。

Nir Eyal（2014）は『Hooked』で **Hook Model** を提唱:
1. **Trigger**（きっかけ）: 外部トリガー（通知）→ 内部トリガー（退屈・不安）
2. **Action**（行動）: 最小限の行動（スワイプ、タップ）
3. **Variable Reward**（変動報酬）: 予測不能な報酬
4. **Investment**（投資）: 次回の体験を良くする行動（プロフィール充実、フォロー等）

変動報酬の3類型:
- **Tribe（社会的報酬）**: いいね、コメント、フォロワー増加
- **Hunt（獲得の報酬）**: 新しいコンテンツ、セール情報、レアアイテム
- **Self（自己達成の報酬）**: スキルアップ実感、新しい機能のアンロック

### 2.2 UIへの適用

| パターン | 実装例 | 効果 |
|---|---|---|
| 無限スクロールフィード | SNS のタイムライン、ニュースフィード | 予測不能なコンテンツが探索行動を維持 |
| ガチャ・ランダムリワード | ログインボーナスのランダム報酬 | 固定報酬より継続率が 15〜40% 高い |
| パーソナライズドレコメンド | 「あなたへのおすすめ」 | 予測不能だが関連性が高い → Hunt 報酬 |
| ソーシャルフィードバック | いいね数・コメント通知 | Tribe 報酬として強力な再訪動機 |
| サプライズ演出 | 予期しないバッジ獲得、隠し機能の発見 | Self 報酬 + デライト効果 |

### 2.3 Apple HIG との接点

- **Dynamic Content**: Today ウィジェットや For You タブは変動報酬の設計原理を体現
- **Notification Design**: HIG は通知の「予測可能性」と「価値」のバランスを重視。過度な通知は変動報酬ではなくノイズになる
- **App Store の Today タブ**: 毎日変わるキュレーションが Hunt 報酬として機能

### 2.4 アンチパターン

| アンチパターン | 問題点 | 改善策 |
|---|---|---|
| 中毒的パターン設計 | ユーザーの幸福度を下げる依存を生む | 使用時間リマインダー、セッション区切りを導入 |
| 通知疲れ | 過剰な通知は変動報酬ではなく嫌悪刺激になる | 頻度制限、ユーザーによる通知粒度のカスタマイズ |
| スロットマシン的メカニクス | 射幸心を煽り、倫理的問題を招く | 報酬の透明性を確保し、上限・冷却期間を設ける |
| 報酬価値の低下 | 同じ報酬の繰り返しは飽きを生む | 報酬プールの定期更新、段階的な報酬品質向上 |

### 2.5 実務活用

- **倫理的設計**: 変動報酬は強力だが、Dark Pattern になりやすい。「ユーザーにとっての価値」を常に自問する
- **通知戦略**: 通知の開封率が下がり始めたら頻度を減らす。量より質で変動報酬の効果を維持
- **A/Bテスト**: 固定報酬 vs 変動報酬でリテンション率を比較。変動報酬群で離脱が増えたら中毒パターンを疑う
- **3類型のバランス**: Tribe・Hunt・Self を偏りなく提供する。SNS が Tribe に偏ると社会的比較の害が生じる

→ 関連: `motivation.md` §3（ゲーミフィケーション — 変動報酬の体系的適用）, `impression.md` §7（ユーザーデライト — 予期しない喜び）

## §3. ゲーミフィケーション（Gamification）

> ゲームの設計要素を非ゲーム文脈に適用し、ユーザーの行動を促進・持続させる手法。

### 3.1 心理学的メカニズム

Sebastian Deterding ら（2011）の定義: 「ゲームデザイン要素をゲーム以外の文脈で使用すること」。単にポイントやバッジを付けることではなく、ゲームが人を惹きつける**構造的メカニズム**を応用することが本質。

心理学的基盤:
- **自己決定理論**（Deci & Ryan, 1985）: 自律性・有能感・関係性の3つの基本的欲求が内発的動機を支える
- **フロー理論**（Csikszentmihalyi, 1990）: スキルと難易度のバランスが最適な没入状態を生む
- **達成動機理論**（McClelland, 1961）: 達成・権力・親和の欲求がゲーム要素と対応する

主要なゲーム要素:
- **ポイント（Points）**: 行動の定量的フィードバック
- **バッジ（Badges）**: 達成の視覚的証明
- **リーダーボード（Leaderboards）**: 社会的比較と競争
- **チャレンジ（Challenges）**: 明確な目標と制約
- **ストリーク（Streaks）**: 連続行動の維持と損失回避の活用

### 3.2 UIへの適用

| パターン | 実装例 | 効果 |
|---|---|---|
| 実績システム | GitHub Contributions、Duolingo ストリーク | 長期的な行動継続を促進 |
| デイリーストリーク | 連続ログイン日数のカウント・表示 | 損失回避と組み合わせて継続率を向上 |
| XP/レベル制 | 学習アプリでのレベルアップ、スキルツリー | 有能感の満足と成長実感の可視化 |
| チームチャレンジ | チーム対抗のステップ数競争 | 関係性欲求を満たし社会的動機を活性化 |
| プログレッションシステム | 初心者→中級→上級のランクアップ | 長期目標とマイルストーンの両方を提供 |

### 3.3 Apple HIG との接点

- **Game Center**: 実績、リーダーボード、チャレンジの標準フレームワーク
- **HealthKit / Activity**: リングの完成、ムーブゴール達成、月間チャレンジ — ヘルスケア領域のゲーミフィケーション成功例
- **Apple Watch Awards**: 特別な日のバッジ（元日、アースデイ等）が希少性と収集欲を刺激

### 3.4 アンチパターン

| アンチパターン | 問題点 | 改善策 |
|---|---|---|
| 浅いゲーミフィケーション | 意味のないバッジは「お子様扱い」と感じられる | バッジに実質的な価値（機能アンロック等）を持たせる |
| 外発的動機による内発的動機の阻害 | 報酬目当てになると報酬がないと行動しなくなる | 内発的動機（学び、成長実感）を主軸にし、外発的報酬は補助に |
| リーダーボードの逆効果 | 上位と差が開くと大多数が離脱する | 同レベル帯でのグループ化、自己ベスト比較を併用 |
| 強制的な競争 | 競争を好まないユーザーを疎外する | 協力型チャレンジと個人目標を選択可能にする |
| ストリーク脅迫 | 「ストリークが途切れます！」の脅迫は反感を買う | ストリーク復活券、週末パスなど柔軟な救済措置 |

### 3.5 実務活用

- **Octalysis フレームワーク**: Yu-kai Chou のフレームワークで8つの Core Drive を分析し、偏りのないゲーミフィケーション設計を行う
- **段階的導入**: 初期はシンプル（ポイント + 進捗バー）→ 慣れたら複雑な要素（チャレンジ、リーダーボード）を追加
- **計測指標**: DAU/MAU 比率、ストリーク継続率、バッジ取得率を追跡し、どの要素が効いているか分析
- **ターゲット層の見極め**: Bartle のプレイヤータイプ（Achiever / Explorer / Socializer / Killer）でユーザーセグメントに合った要素を設計

→ 関連: `motivation.md` §1（目標勾配 — プログレッションの心理基盤）, `motivation.md` §2（変動報酬 — 報酬設計の理論）, `impression.md` §7（ユーザーデライト — 期待を超える体験）

## §4. ツァイガルニク効果（Zeigarnik Effect）

> 未完了のタスクは完了したタスクより記憶に残り、心理的緊張を生んで完了への動機を高める。

### 4.1 心理学的メカニズム

Bluma Zeigarnik（1927）の実験: 被験者に複数のタスクを与え、一部を途中で中断させた。結果、中断されたタスクは完了したタスクより約 2 倍記憶に残った。

心理学的説明 — Kurt Lewin の場の理論に基づく。未完了タスクは「緊張システム」を形成し、この緊張は完了によってのみ解消される。この未解決の緊張が:
- 当該タスクへの注意を持続させる
- 再開への動機を生む
- 無関係な活動中にも侵入的に想起される（反すう）

**前提条件**: タスクへの関与度（Commitment）が一定以上であること。興味のないタスクの中断は効果が弱い。

### 4.2 UIへの適用

| パターン | 実装例 | 効果 |
|---|---|---|
| プロフィール完成度表示 | 「プロフィール70%完了」LinkedIn式 | 未完了の視覚化が完了行動を促進 |
| 未完了オンボーディング | 「あと3ステップで設定完了」のリマインド | 初期設定の完了率を 40% 以上改善 |
| コレクション・収集要素 | 「5/12バッジ獲得済み」の一覧表示 | 収集欲と完了欲が相乗的に働く |
| 下書き保存リマインダー | 「未完成の下書きが2件あります」 | 再訪・完了動機を生成 |
| シリーズコンテンツ | 「エピソード 3/8 視聴済み」 | 次エピソードへの自動的な関心 |

### 4.3 Apple HIG との接点

- **Progress Indicators**: HIG はタスクの進行状況を常にユーザーに伝えることを推奨。ツァイガルニク効果の設計基盤
- **Badges / Notifications**: 未完了タスクのバッジ表示（メールの未読数等）は緊張を可視化
- **Continuity（Handoff）**: デバイス間での作業継続機能は「中断した作業を再開したい」という心理を活用

### 4.4 アンチパターン

| アンチパターン | 問題点 | 改善策 |
|---|---|---|
| 未完了項目の過剰表示 | 「未完了が50件」は圧倒感と不安を生む | 表示数を制限し、優先度順にフィルタリング |
| 罪悪感を煽る表現 | 「まだ完了していません！」は不快 | ポジティブなフレーミング「あと少しで達成！」 |
| 解消不可能な緊張 | 完了方法が分からない/困難すぎる | 明確な次のアクションを提示、スキップオプション |
| 強制的な完了圧力 | ダイアログを閉じられない、スキップ不可 | 「後で」「スキップ」オプションを必ず提供 |

### 4.5 実務活用

- **メール・通知設計**: 「未完了のアクションがあります」系のリマインドメールは開封率が高い。ただし頻度制限必須
- **オンボーディング**: 初回セットアップを5ステップに分割し、最初の2ステップを完了させてから中断させる（関与度を高めてからツァイガルニク効果を発動）
- **eコマース**: カート放棄メール（「カートに商品が残っています」）はこの効果の直接的応用
- **計測**: 「中断→再開」率、「表示→完了」率を追跡し、緊張が適切に完了動機に変換されているか検証

→ 関連: `motivation.md` §1（目標勾配 — ゴールが近いほど加速する効果との相乗）, `choice.md` §8（サンクコスト — 既に投じた労力が完了を後押し）

## §5. 好奇心ギャップ（Curiosity Gap）

> 「知っていること」と「知りたいこと」の間のギャップが、情報探索行動を強力に動機づける。

### 5.1 心理学的メカニズム

George Loewenstein（1994）の情報ギャップ理論: 好奇心は「知識のギャップを認知した時」に生じる感情。このギャップは不快感を伴い、人はそれを「閉じる」（情報を得る）ことで解消しようとする。

重要な前提条件:
- ギャップが**存在すること**を認知する必要がある（全く知らない分野には好奇心が湧きにくい）
- ギャップが**適度**であること（大きすぎると諦め、小さすぎると興味が湧かない）
- **解消可能**であるという期待が必要（クリックすれば分かる、読めば分かる）

好奇心の種類:
- **知覚的好奇心（Perceptual Curiosity）**: 新奇な刺激への反応。短期的
- **認識的好奇心（Epistemic Curiosity）**: 知識を得たいという欲求。持続的

### 5.2 UIへの適用

| パターン | 実装例 | 効果 |
|---|---|---|
| ティーザーコンテンツ | 記事冒頭の3行表示 + 「続きを読む」 | クリック率を 20〜50% 向上 |
| ぼかしプレビュー | プレミアム機能のぼかし表示 | 「中が見えそうで見えない」がギャップを生成 |
| 通知プレビュー | 「○○さんがあなたの投稿に…」（途中で切れる） | アプリ起動率の向上 |
| パーソナライズドインサイト | 「あなたの先月の利用傾向を分析しました」 | 自己参照効果 + 好奇心ギャップ |
| カウントダウン・ティーザー | 「3日後に新機能公開」 | 期待感と再訪動機の醸成 |

### 5.3 Apple HIG との接点

- **Notification Preview Text**: ロック画面の通知プレビューは好奇心ギャップの典型。内容をチラ見せしてアンロック・タップを促す
- **Widget Teasers**: ウィジェットの限定情報表示がアプリ起動のトリガーになる
- **App Store Preview**: スクリーンショットのキャプションが好奇心を刺激する設計

### 5.4 アンチパターン

| アンチパターン | 問題点 | 改善策 |
|---|---|---|
| クリックベイト | 期待を裏切る内容は信頼を破壊する | タイトルの約束を本文で必ず果たす |
| 過度なティーザー | 「続きは有料」の連発はフラストレーションを生む | 無料で十分な価値を提供した上で有料コンテンツを提示 |
| 情報の出し惜しみ | 必要な情報をわざと隠すのは不誠実 | ユーザーに「隠している」と感じさせない自然な設計 |
| ギャップが大きすぎる | 何について話しているか全く分からない | 文脈を十分に提供してからギャップを作る |

### 5.5 実務活用

- **メール件名の最適化**: 好奇心ギャップを活用した件名（「あなたのアカウントに変化がありました」）は開封率が高い。ただしクリックベイトとの境界に注意
- **コンテンツマーケティング**: 記事タイトルで問いを投げかけ、本文で回答する構造。問いは読者が「答えを知りたい」と思えるレベルに調整
- **プロダクトローンチ**: 段階的な情報公開（ティーザー → 詳細1 → 詳細2 → 発売）で好奇心を持続
- **A/Bテスト**: ティーザー表現のバリエーション（ぼかし量、表示文字数、切り取り位置）でクリック率を最適化

→ 関連: `motivation.md` §2（変動報酬 — 予測不能な情報が好奇心を刺激）, `impression.md` §6（プライミング — 先行情報がギャップの文脈を形成）

## §6. フット・イン・ザ・ドア（Foot-in-the-Door）

> 小さな要求に応じた人は、その後の大きな要求にも応じやすくなる。

### 6.1 心理学的メカニズム

Freedman & Fraser（1966）の古典的実験: 最初に「安全運転のステッカーを貼ってください」という小さな依頼に応じた住民は、後日「庭に大きな看板を設置させてください」という大きな依頼に 76% が同意（対照群は 17%）。

心理学的メカニズム:
- **自己認知理論**（Bem, 1972）: 小さな行動をとった自分を「そういう人間だ」と認識する
- **一貫性の原理**（Cialdini, 2001）: 以前の行動と矛盾しない行動をとろうとする
- **コミットメント**: 小さなコミットメントが大きなコミットメントの心理的障壁を下げる

段階的エスカレーション（Graduated Escalation）が効果的。急激な飛躍は失敗する。

### 6.2 UIへの適用

| パターン | 実装例 | 効果 |
|---|---|---|
| フリーミアムモデル | 無料利用 → 有料プランへの段階的誘導 | 無料での成功体験が課金障壁を下げる |
| ゲストチェックアウト | 購入完了後「アカウント作成しませんか？」 | 購入という小さなコミットメント後に大きな要求 |
| 段階的権限要求 | 機能使用時に必要な権限のみリクエスト | 一括要求より許可率が 30〜50% 高い |
| マイクロコンバージョン | 「まずはお気に入り登録」→「カートに追加」→「購入」 | 各ステップが次のステップの心理的準備 |
| 無料トライアル | 14日間無料 → 自動継続（明示的同意付き） | 使い始めた行動が継続の動機になる |

### 6.3 Apple HIG との接点

- **Onboarding Flows**: HIG は初回起動時に全機能を説明するのではなく、最低限の設定から始めることを推奨
- **Progressive Account Setup**: Apple ID の段階的情報収集は FITD の体現
- **Permission Requests**: HIG は「使用時に権限を求める」パターンを推奨。一括要求を避ける

### 6.4 アンチパターン

| アンチパターン | 問題点 | 改善策 |
|---|---|---|
| ベイト・アンド・スイッチ | 「無料」で釣って隠れた費用を請求 | 料金体系を事前に透明に開示 |
| 急すぎるエスカレーション | 登録直後に有料プラン強要 | 十分な無料体験期間を設け、価値を実感してから提案 |
| ダークパターン的同意取得 | 小さな同意を積み重ねて望まない契約に誘導 | 各ステップで明確な説明とオプトアウトを提供 |
| 段階なしの大要求 | 初回訪問でクレジットカード情報を要求 | まずは価値提供（コンテンツ閲覧、機能試用）から |

### 6.5 実務活用

- **SaaS オンボーディング**: 「メールアドレスだけで開始」→「プロフィール設定」→「チーム招待」→「有料プラン選択」の段階設計
- **コンバージョンファネル**: 各段階の離脱率を計測し、エスカレーションが急すぎるポイントを特定
- **メールマーケティング**: 「まずはこの記事を読んでみてください」→「無料ウェビナーに参加しませんか」→「有料コースのご案内」
- **UXライティング**: 各段階のCTAは次のステップのハードルの低さを強調（「1分で完了」「いつでもキャンセル可能」）

→ 関連: `information.md` §2（段階的開示 — 情報量の段階的提示）, `bias.md` §7（リアクティブオンボーディング — 行動に応じた導入）

## §7. 誘惑バンドリング（Temptation Bundling）

> 「やりたいこと」と「やるべきこと」を組み合わせることで、やるべき行動の実行率を高める。

### 7.1 心理学的メカニズム

Katherine Milkman ら（2014）の実験: 被験者を3群に分割 — (1) 魅力的なオーディオブックをジムでのみ聴ける群、(2) 自由に聴ける群、(3) 対照群。結果、(1) のジム通い頻度が有意に高かった。

理論的背景:
- **即時報酬バイアス**: 人間は将来の利益より目の前の快楽を優先する
- **誘惑バンドリング**: 即時的な快楽（やりたいこと）を、長期的に有益な行動（やるべきこと）の条件にすることで、バイアスを逆手に取る
- **Self-Control Theory**（Baumeister & Vohs, 2007）: 意志力は有限資源。バンドリングは意志力に頼らない行動変容を可能にする

条件:
- 「やりたいこと」が十分に魅力的であること
- 「やるべきこと」が過度に苦痛でないこと
- バンドリングが自然・合理的に感じられること

### 7.2 UIへの適用

| パターン | 実装例 | 効果 |
|---|---|---|
| 学習 + エンタメ | 語学アプリのゲーム化されたレッスン | 学習の苦痛を遊びの快楽で相殺 |
| タスク完了 + 報酬 | 「この作業を完了すると限定コンテンツ解放」 | タスク完了率の大幅向上 |
| 運動 + コンテンツ | フィットネスアプリ内での音楽・動画提供 | 運動継続率の改善 |
| 退屈な設定 + 進捗報酬 | セキュリティ設定完了でバッジ獲得 | セキュリティ設定の完了率向上 |
| 日常タスク + ソーシャル | チームでの目標達成チャレンジ | 社会的報酬がタスクの動機を強化 |

### 7.3 Apple HIG との接点

- **Apple Fitness+**: トレーナーの励ましや音楽（快楽）+ エクササイズ（やるべきこと）の直接的なバンドリング
- **Apple Music + Workout**: ワークアウト中の音楽連携で運動体験を向上
- **Screen Time**: 制限アプリの使用時間を「やりたいこと」として、学習・運動と条件付ける設計パターン

### 7.4 アンチパターン

| アンチパターン | 問題点 | 改善策 |
|---|---|---|
| 強制的なバンドリング | ユーザーの選択肢を奪う | バンドリングはオプトイン。ユーザーが組み合わせを選べるように |
| 無関係なペアリング | 「アンケート回答で割引」等の不自然な組合せ | 行動の文脈に合った自然なペアリングを設計 |
| 報酬の質が低い | 「やりたいこと」の魅力が弱いと効果がない | 報酬の魅力度をユーザーテストで検証 |
| やるべきことが過酷すぎる | どんなに良い報酬でもバランスが崩れる | 行動の難易度を段階的に上げる設計 |

### 7.5 実務活用

- **ヘルスケアアプリ**: 服薬リマインダー + お気に入りコンテンツ通知のバンドリング
- **教育プラットフォーム**: 難しい課題の後にお楽しみコンテンツ（面白いクイズ、ミニゲーム）を配置
- **企業内ツール**: 月次レポート作成（苦痛）+ チームランキング表示（楽しみ）の組み合わせ
- **効果測定**: バンドリングあり/なし群で「やるべきこと」の実行率を比較。バンドリング解除後の行動も追跡（習慣化したか）

→ 関連: `motivation.md` §3（ゲーミフィケーション — ゲーム要素を報酬として活用）

## §8. 心理的リアクタンス（Psychological Reactance）

> 自由を脅かされると、人はその自由を回復しようとして反対の行動をとる。

### 8.1 心理学的メカニズム

Jack Brehm（1966）の**リアクタンス理論**: 人は行動の自由を持つと認識しており、その自由が外部から制限・脅威にさらされると、自由を回復するための動機（リアクタンス）が生じる。

リアクタンスの発生条件:
- **自由の認知**: 当該行動が自由に行えると信じていること
- **自由の脅威**: 外部からの制限、強制、圧力があること
- **脅威の強さ**: 制限が強いほどリアクタンスも強い
- **自由の重要性**: 当人にとって重要な自由ほどリアクタンスが大きい

リアクタンスの表出:
- **直接的回復**: 禁止された行動をあえて行う（広告ブロッカーの導入等）
- **間接的回復**: 類似の自由行動を強化する
- **攻撃性**: 制限者への怒り、低評価レビュー、SNS での批判
- **態度変容**: 制限されたものの価値を過大評価する（禁止されたものほど欲しくなる）

### 8.2 UIへの適用

| パターン | 実装例 | 効果 |
|---|---|---|
| 明確な拒否オプション | 「いいえ、結構です」ボタンの常時表示 | 選択の自由を保証し、リアクタンスを防止 |
| 透明性のある選択肢 | 料金・条件の事前開示 | 「騙された」感を排除し、信頼を構築 |
| ソフトな提案言語 | 「〜してみませんか？」vs「〜してください」 | 命令調を避け、自律性を尊重 |
| ユーザーコントロール | 通知頻度・種類のカスタマイズ | 自分で制御できる感覚がリアクタンスを抑制 |
| 元に戻す機能 | Undo サポート、設定リセット | 「取り返しがつかない」恐怖を除去 |

### 8.3 Apple HIG との接点

- **User Control**: HIG の根幹原則。ユーザーが常にコントロールしている感覚を維持する
- **Undo Support**: 破壊的操作の前に確認、実行後の取り消しを推奨
- **Non-Intrusive Suggestions**: Siri の提案は「押し付け」ではなく「オファー」として表示される設計

### 8.4 アンチパターン

| アンチパターン | 問題点 | 改善策 |
|---|---|---|
| 閉じられないポップアップ | 最強のリアクタンストリガー。アプリ削除に直結 | 明確な「×」ボタンを常に表示 |
| 隠された解約導線 | 解約を困難にすると怒りと悪評を生む | 解約プロセスを登録と同等に簡単にする |
| 罪悪感を煽るコピー | 「本当にやめるんですか？」「見逃しますよ」 | 事実ベースの中立的な表現を使う |
| 強制的なアップセル | ページ遷移のたびに有料プラン広告 | 適切なタイミング（価値を実感した後）にのみ提示 |
| 選択肢を制限するデザイン | 「同意」ボタンだけ大きく、「拒否」が極小 | 両方の選択肢を同等の視認性で提示 |

### 8.5 実務活用

- **通知戦略**: オプトインを基本とし、頻度・種類をユーザーが完全にコントロールできる設計。プッシュ通知の無断送信は最悪のリアクタンス源
- **価格戦略**: 「期間限定」「在庫わずか」の希少性訴求が過度だとリアクタンスを誘発。事実に基づく控えめな表現が効果的
- **解約フロー設計**: 解約理由の聴取はOKだが、多段階の引き止めは逆効果。最大1回の提案（割引等）に留める
- **ダークパターン監査**: 自社UIの全モーダル・ポップアップ・通知を棚卸し、「ユーザーの自由を制限していないか」の観点で監査
