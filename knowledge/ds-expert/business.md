## §1. 問題設定

### 1.1 ビジネスKPIとML指標の対応

**ビジネスKPI → ML指標への変換**:
| ビジネスKPI | ML問題 | ML指標 | 注意点 |
|-------------|--------|--------|--------|
| 売上増加 | 推薦・需要予測 | CTR、CVR、RMSE | オフライン指標とオンライン指標の乖離 |
| 解約率低減 | 二値分類（Churn） | Recall（解約者を逃さない） | Precision-Recallのトレードオフ |
| 不正損失削減 | 異常検知・分類 | Precision（誤検知コスト） | 不均衡データ対応 |
| 顧客満足度 | 感情分析・NPS予測 | F1、相関係数 | ラベルの主観性 |
| オペレーション効率 | 最適化・自動化 | 処理時間、コスト削減額 | 導入前後の比較設計 |

**指標選択の原則**:
- ビジネスの意思決定に直結する指標を最上位に置く
- ML指標はビジネスKPIの代理指標（proxy）であることを認識する
- Proxy指標がビジネスKPIと乖離する場合を常に警戒する

**Goodhartの法則**: 「指標が目標になると、良い指標でなくなる」
- 例: CTR最適化 → クリックベイトの増加 → ユーザ体験悪化
- 対策: 複数指標のバランス（ガードレール指標の設定）

### 1.2 ROI計算（モデル導入の費用対効果）

**MLプロジェクトのコスト構造**:
```
総コスト = 初期コスト + 運用コスト
初期コスト = データ整備 + 開発 + インフラ構築 + テスト
運用コスト = 推論インフラ + 監視 + 再学習 + 保守 (月額)
```

**ROI計算**:
```
ROI = (累積効果額 - 累積コスト) / 累積コスト × 100%

NPV = Σ_{t=0}^{T} (効果額ₜ - コストₜ) / (1+r)ᵗ
```

**効果額の推定方法**:
| 効果の種類 | 推定方法 |
|-----------|---------|
| 売上増加 | A/Bテスト × 対象ユーザ数 × 年間 |
| コスト削減 | (人手の工数 - ML後の工数) × 単価 |
| 損失回避 | (不正検知率向上分) × 平均損失額 |
| 機会費用 | 自動化で浮いた時間 × その時間の価値 |

**MLプロジェクトのリスク要因**:
- データ品質が想定を下回る確率
- モデル精度が実用水準に達しない確率
- 導入後のドリフトによる精度劣化速度
- 規制変更による利用制限リスク

**段階的投資判断**: PoC → MVP → 本番の各段階でGo/No-Go判断。サンクコストに囚われない。

### 1.3 意思決定フレームワーク

**MLモデルによる意思決定支援の設計**:
```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ ML予測/スコア│ ──→ │ ビジネスルール│ ──→ │ 最終意思決定 │
│ (確率, ランク)│     │ (閾値, 制約)  │     │ (自動/人間)  │
└─────────────┘     └──────────────┘     └─────────────┘
```

**自動化レベル**:
| レベル | 説明 | 例 |
|--------|------|-----|
| 0 | 情報提示のみ | ダッシュボードに予測値表示 |
| 1 | 推薦（人間が判断） | 解約リスク顧客リストの提示 |
| 2 | 条件付き自動化 | 高確信度のみ自動処理、低確信度は人間 |
| 3 | 完全自動化 | リアルタイム不正検知（即ブロック） |

**閾値設計（コスト感応的）**:
```
期待コスト(τ) = C_FP × FP率(τ) × N_neg + C_FN × FN率(τ) × N_pos
最適閾値 τ* = argmin 期待コスト(τ)
```

- C_FP: 偽陽性1件のコスト（例: 正常取引を停止する損失）
- C_FN: 偽陰性1件のコスト（例: 不正を見逃す損失）
- コスト非対称性を反映した閾値が最適

**公平性の考慮**:
- デモグラフィック・パリティ: 各グループで予測陽性率が等しい
- 機会均等: 各グループで真陽性率が等しい
- 個人公平性: 類似した個人には類似した予測
- トレードオフ: 公平性指標間、精度と公平性間の妥協点を明示する

### 1.4 実務活用

- **EC企業**: 推薦モデルのROI = (A/Bテスト増分売上 × 年間) - (開発・運用コスト)。Goodhartの法則対策としてカート追加率+リピート率をガードレールに設定
- **金融機関**: 不正検知の閾値をコスト感応的に設計（誤検知1件=¥500, 見逃し1件=¥50,000 → 低閾値が最適）
- **SaaS**: 解約予測モデルのROI。介入コスト（割引クーポン等）を含めた損益分岐点を計算
- **製造業**: 品質検査自動化の段階的導入。Level 2（高確信度のみ自動→低確信度は人間）でスタート

## §2. 実験設計

### 2.1 A/Bテスト（大規模）

**サンプルサイズ設計**:
```
n = (Z_{α/2} + Z_β)² × 2σ² / δ²
```
- α: 第1種の過誤率（通常0.05）
- β: 第2種の過誤率（通常0.2 → 検出力80%）
- δ: 検出したい最小効果量（Minimum Detectable Effect）
- σ²: アウトカムの分散

**分散削減手法（大規模A/Bテストの精度向上）**:
| 手法 | 原理 | 効果 |
|------|------|------|
| CUPED | 実験前データで共変量調整 | 分散30-50%削減 |
| 層別化 (Stratification) | ユーザ属性でブロック分け | 分散10-30%削減 |
| クラスタ設計 | ユーザ間の干渉を考慮 | バイアス除去 |
| Delta method | 比率指標の分散推定 | 正確な信頼区間 |

**CUPED (Controlled-experiment Using Pre-Experiment Data)**:
```
Y_adj = Y - θ × X_pre
θ = Cov(Y, X_pre) / Var(X_pre)
```
- X_pre: 実験前の同じ指標（過去30日のCTR等）
- 相関が高いほど分散削減効果が大きい

**ネットワーク効果への対処**:
- ユーザ間の干渉がある場合（SNS、マーケットプレイス）
- クラスタランダム化: 地域/グループ単位でランダム割り当て
- Ego-cluster: 各ユーザとその友人を1クラスタとして扱う

**多重比較問題**:
- 複数のバリアント（A/B/C/D...）を同時テスト → 偽陽性率が増大
- Bonferroni補正: α/m（保守的すぎる場合あり）
- Benjamini-Hochberg: FDR制御（推奨）
- α-spending: 中間分析ごとにαを配分

**Peeking問題（早期打ち切りバイアス）**:
- データ蓄積中に繰り返しp値を見ると偽陽性率が増大
- 対策: 固定サンプルサイズ設計 or 逐次テスト or ベイズ的手法

### 2.2 多腕バンディット

**A/Bテストとの比較**:
| 特性 | A/Bテスト | バンディット |
|------|-----------|-------------|
| 目的 | 最適なバリアントの特定 | 累積報酬の最大化 |
| 割り当て | 固定（50:50等） | 動的（成績に応じて変化） |
| regret | 実験期間中は高い | 低い（逐次的に最適化） |
| 統計的推論 | 明確な検定結果 | 事後的な推論が複雑 |

**探索-活用のトレードオフ**:
- 探索が少なすぎる → 局所最適に陥る
- 探索が多すぎる → 最適な腕を十分に活用できない

**主要アルゴリズム**（`advanced-modeling.md` §4.4 参照）:
- ε-greedy: シンプルだが最適性に欠ける
- UCB: 楽観主義の原則に基づく確定的な探索
- Thompson Sampling: ベイズ的に最適な探索-活用バランス（推奨）

**文脈付きバンディット**: ユーザの属性（文脈）に応じて最適なアクションを選択。
- ECのパーソナライゼーション: ユーザ属性→最適なレイアウト/推薦ロジック
- 広告: ユーザ属性→最適なクリエイティブ

### 2.3 逐次テスト

**問題**: 固定サンプルサイズ設計では実験期間が長くなりすぎる場合がある。

**SPRT (Sequential Probability Ratio Test)**:
```
Λₙ = Π_{i=1}^{n} p₁(xᵢ)/p₀(xᵢ)

上限B = (1-β)/α を超えたら H₁ を採択
下限A = β/(1-α) を下回ったら H₀ を採択
A < Λₙ < B なら継続
```

**Group Sequential Design**:
- 中間分析のタイミングと各時点のαを事前に計画
- O'Brien-Fleming: 初期は厳しく、後半で緩く → 実質的にサンプルサイズのロスが少ない
- Pocock: 各中間分析で同一のα → サンプルサイズのロスがやや大きい

**Always-Valid p-values**:
- 任意の時点で有効な推論が可能
- Confidence Sequence: 任意の停止時刻で有効な信頼区間
- e-process: 逐次的な仮説検定の新しい枠組み

**Bayesian Stopping Rule**:
```
P(δ > 0 | data) > 0.95  →  対立仮説を支持して停止
P(|δ| < ε | data) > 0.95  →  帰無仮説（実質的同等）を支持して停止
```
- 実務的に自然な停止基準
- ROPEベース: 事実上の同等性を判断可能

### 2.4 オフライン実験（反実仮想推定）

**問題**: オンライン実験が不可能/コストが高い場合に、ログデータから新方策を評価。

**IPS (Inverse Propensity Scoring)**:
```
V̂(π) = (1/n) Σ_{i=1}^{n} (π(aᵢ|xᵢ) / π₀(aᵢ|xᵢ)) × rᵢ
```
- π₀: ログ収集時の方策（既知である必要がある）
- 高分散問題: 重みがクリッピングが必要な場合あり

**Doubly Robust**:
```
V̂_DR(π) = (1/n) Σ [q̂(xᵢ,aᵢ) + (π(aᵢ|xᵢ)/π₀(aᵢ|xᵢ))(rᵢ - q̂(xᵢ,aᵢ))]
```
- q̂: 報酬の回帰モデル
- IPSと回帰推定の良いとこ取り

### 2.5 実務活用

- **EC**: A/Bテスト（CUPED適用で実験期間半減）→ 新推薦アルゴリズムの効果検証
- **メディア**: Thompson Samplingで記事見出しの最適化（CTR最大化）
- **SaaS**: Group Sequential Design で新UI機能の早期検出/早期中止
- **広告**: 文脈付きバンディットでクリエイティブ×ターゲティングの最適組み合わせ探索
- **金融**: オフライン評価（IPS）で新しい信用スコアリングモデルの事前検証

## §3. コミュニケーション

### 3.1 技術結果の非技術者への説明

**STAR-T フレームワーク**（データサイエンス結果の伝達）:
| 要素 | 内容 | 例 |
|------|------|-----|
| **S**ituation | ビジネスの背景・課題 | 「月間解約率が5%に上昇している」 |
| **T**ask | 分析の目的 | 「解約要因を特定し、介入施策を提案する」 |
| **A**nalysis | 分析手法と結果 | 「機械学習モデルで上位3要因を特定」 |
| **R**esult | ビジネスインパクト | 「介入で解約率1.5%削減、年間¥3,000万の効果」 |
| **T**akeaway | 推奨アクション | 「まず〇〇セグメントへのクーポン施策を実施」 |

**用語変換テーブル**:
| 技術用語 | 非技術者向け表現 |
|----------|----------------|
| AUC 0.85 | 100人中85人を正しく識別できる精度 |
| 特徴量重要度 | 予測に最も影響する要因 |
| 過学習 | 過去のデータに合わせすぎて新しいデータに対応できない |
| 正則化 | モデルが複雑になりすぎないようにする仕組み |
| 信頼区間 | 真の値がこの範囲にある可能性が95% |
| p値 < 0.05 | 偶然だけでは説明しにくい差がある |
| 因果効果 | この施策を行ったから起きた変化（たまたまではない） |

**数字の提示の仕方**:
- 絶対数と割合の両方を示す: 「5%（500人中25人）」
- 比較対象を明示: 「現在の5%から3.5%に改善（30%削減）」
- 金額換算: 「解約率1%改善 = 年間¥2,000万の売上維持」

### 3.2 不確実性の伝え方

**不確実性の源泉**:
| 種類 | 説明 | 伝え方 |
|------|------|--------|
| データの不確実性 | サンプルサイズ、測定誤差 | 信頼区間、サンプルサイズ |
| モデルの不確実性 | 仮定、簡略化 | 感度分析、複数モデル比較 |
| 予測の不確実性 | 将来の変動 | 予測区間、シナリオ分析 |

**可視化手法**:
- **信頼区間のバンド表示**: 中央値 + 80%/95%区間を帯で表示
- **シナリオ分析**: ベスト/ベース/ワーストの3シナリオ提示
- **ファンチャート**: 時間軸で広がる扇形の不確実性表示
- **リスクマトリクス**: 発生確率 × 影響度のヒートマップ

**不確実性コミュニケーションの原則**:
1. **正直さ**: わからないことは「わからない」と伝える
2. **段階的開示**: まず結論、詳細は求められてから
3. **行動指向**: 不確実性があっても推奨アクションは明示
4. **自然頻度**: 確率より頻度表現が直感的（「100人中15人」> 「15%」）
5. **アンカリング回避**: 点推定値だけ先に見せない（区間と同時に提示）

**意思決定者への質問**:
- 「どの程度の確信度があれば行動に移しますか？」
- 「間違った場合のコストは？（偽陽性 vs 偽陰性）」
- 「追加データを取得するコストと時間はどの程度ですか？」

### 3.3 ダッシュボード設計

**情報階層（ピラミッド原則）**:
```
┌─────────────────┐
│   KPI サマリー    │  ← エグゼクティブ（3秒で把握）
├─────────────────┤
│   トレンド・比較   │  ← マネージャー（1分で状況理解）
├─────────────────┤
│   詳細分析・ドリル  │  ← アナリスト（深掘り）
└─────────────────┘
```

**ダッシュボード設計の原則**:
| 原則 | 説明 |
|------|------|
| 目的明確化 | 誰が、何のために見るか |
| 情報量制限 | 1画面に7±2の指標まで |
| ゲシュタルト | 関連情報をグループ化 |
| F型パターン | 重要情報を左上に配置 |
| 色の一貫性 | 赤=悪化、緑=改善を統一 |
| アクション導出 | 見て終わりでなく次の行動につながる |

**グラフ種類の選択**:
| 目的 | 推奨グラフ | 避けるべき |
|------|-----------|-----------|
| 時系列トレンド | 折れ線グラフ | 3Dグラフ |
| カテゴリ比較 | 棒グラフ（横 or 縦） | 円グラフ（5カテゴリ超） |
| 分布 | ヒストグラム、箱ひげ図 | — |
| 相関 | 散布図 | — |
| 構成比 | 積み上げ棒、ツリーマップ | 円グラフ（微差の比較） |
| 地理分布 | コロプレスマップ | — |

**インタラクティブ機能**:
- フィルタ: 期間、セグメント、地域
- ドリルダウン: サマリー→詳細への遷移
- ツールチップ: ホバーで詳細数値表示
- アラート: 閾値超過時の自動通知

**ツール**:
| ツール | 特徴 | 対象ユーザ |
|--------|------|-----------|
| Tableau | GUI中心、美しい可視化 | ビジネスユーザ |
| Looker | SQL/LookMLベース | データチーム |
| Streamlit | Pythonコードベース | データサイエンティスト |
| Grafana | 時系列監視特化 | エンジニア |
| Evidence | SQLベースBI、Markdownライク | データチーム |

### 3.4 レポーティングのベストプラクティス

**分析レポートの構成**:
1. **エグゼクティブサマリー**（1段落）: 結論と推奨アクション
2. **背景と目的**（2-3段落）: なぜこの分析を行ったか
3. **手法の概要**（非技術的に）: 何をしたか
4. **主要な発見**（3-5点）: ビジネスインパクトの大きい順
5. **推奨アクション**（優先順位付き）: 次に何をすべきか
6. **制約事項と次のステップ**: 分析の限界と今後の計画
7. **付録**（技術詳細）: 手法の詳細、追加の図表

**効果的なデータストーリーテリング**:
- **対比**: 「現状A → 施策後B」のbefore/after構造
- **漸進的開示**: 全体像→ドリルダウンの順序
- **アノテーション**: グラフ上に重要ポイントを注釈
- **ナラティブ**: 数字の羅列でなくストーリーの流れ

### 3.5 実務活用

- **経営会議**: STAR-Tフレームワークでモデル導入効果を説明。ROIは金額換算で提示
- **マーケティング**: A/Bテスト結果をシナリオ分析（ベスト/ワースト）付きで報告
- **プロダクト**: KPIダッシュボードをピラミッド構造で設計。日次自動更新+週次レビュー
- **リスク管理**: 不確実性を信頼区間バンドで可視化。意思決定の閾値を事前合意
- **研究開発**: Streamlitで分析プロトタイプをインタラクティブに共有→フィードバック収集
