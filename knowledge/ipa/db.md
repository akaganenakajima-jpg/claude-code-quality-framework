# データベース設計・運用 — 開発リファレンス

# 1. 概念データモデリング

## 1.1 ER図の基本要素
- **エンティティ**: 矩形。独立エンティティ（自身のPKで一意）/ 依存エンティティ（親のPKを含む複合PKで一意）
- **リレーションシップ**: 1:1 / 1:N / M:N（M:Nは中間テーブルで解消）
- **属性**: PK（主キー）/ FK（外部キー）/ 非キー属性
- **カーディナリティ**: 最小参加(0 or 1) × 最大参加(1 or N)
- **原理**: 現実世界の業務構造をデータ構造として抽象化する。業務ルール=制約条件

## 1.2 スーパータイプ/サブタイプ

### 分類軸
| | 排他的 | 共存的 |
|---|---|---|
| **完全** | 全インスタンスが必ず1つのサブタイプに属する | 全インスタンスが1つ以上のサブタイプに属する |
| **不完全** | 属さないインスタンスが存在しうる | 属さないインスタンスが存在しうる |

- **排他的**: サブタイプ識別子（区分コード等）で1つに決まる。例: 正社員/契約社員
- **共存的**: 同時に複数のサブタイプに属しうる。例: 店舗販売商品 かつ ネット販売商品

### テーブル実装の3パターン
| パターン | 方法 | 長所 | 短所 |
|---|---|---|---|
| 1テーブル統合 | 全属性を1テーブル | 結合不要 | NULL多発・制約困難 |
| スーパー+サブ分離 | 共通テーブル+サブテーブル | 正規化・柔軟 | 常に結合が必要 |
| サブタイプのみ | サブに共通属性も含む | 結合不要 | UNION必要・冗長 |

### 設計判断の指針
- 同じPKを持つテーブル群 = スーパー/サブ関係の可能性大
- 業務要件から「排他的 or 共存的」「完全 or 不完全」を判定
- パターン暗記ではなく業務要件からの読み取りが本質
- 将来のサブタイプ追加可能性が高い場合はスーパー+サブ分離が安全

## 1.3 ER図→関係スキーマ変換
- 1:1 → どちらかにFK追加（参照頻度が高い側に置くのが定石）
- 1:N → N側にFK追加
- M:N → 中間テーブル（連関エンティティ）作成。PKは両側のFKの複合キー
- **連関エンティティに独自属性がある場合**: 数量、日付等の「関係そのものの属性」

# 2. 正規化理論

## 2.1 関数従属性
- **完全関数従属**: {A,B}→C で、A→C もB→C も成り立たない
- **部分関数従属**: {A,B}→C で、A→C が成り立つ
- **推移的関数従属**: A→B かつ B→C（Bは候補キーでない）ならば A→C は推移的
- **アームストロングの公理**: 反射律/増加律/推移律（これだけで全ての関数従属性を導出可能）
- **合併律**: A→B かつ A→C ならば A→{B,C}
- **分解律**: A→{B,C} ならば A→B かつ A→C

## 2.2 各正規形
| 正規形 | 条件 | 排除対象 |
|---|---|---|
| **1NF** | 全属性が原子値 | 繰返しグループ |
| **2NF** | 1NF + 部分関数従属なし | 複合PKの一部→非キーの従属 |
| **3NF** | 2NF + 推移的関数従属なし | 非キー→非キーの従属 |
| **BCNF** | 全ての決定因子が候補キー | 候補キーでない決定因子 |
| **4NF** | BCNF + 自明でない多値従属性なし | 独立した多値従属性 |
| **5NF** | 4NF + 結合従属性が候補キーのみ | 暗黙の結合従属性 |

## 2.3 理論的性質
- **3NFまで**: 情報無損失かつ関数従属性保存の分解が**常に可能**
- **BCNF**: 情報無損失の分解は可能だが、関数従属性保存は保証されない（トレードオフ）
- **4NF/5NF**: 実務で問題になるケースは稀。多くの場合 3NF=BCNF=4NF=5NF
- **多値従属性**: A→→B（Aが決まるとBの集合が決まる。CとBは独立）
  - 例: {教授}→→{担当科目}, {教授}→→{研究テーマ} が独立なら4NF違反

## 2.4 非正規化（デノーマライゼーション）
- **目的**: 結合コスト削減による性能向上
- **手法**: 冗長列の追加/テーブル統合/集計テーブル/マテリアライズドビュー
- **原理**: 正規化=更新整合性優先、非正規化=参照性能優先。トレードオフを理解した上で判断
- **よくある設計ミス**: 「正規化すれば常に良い」という思い込み。OLTP=正規化、OLAP=非正規化が定石
- **非正規化の導入基準**: 実行計画で結合コストがボトルネックと確認できた場合に限定する。推測で非正規化しない

# 3. SQL高度技法

## 3.1 ウィンドウ関数
```sql
SELECT col, AGG() OVER (
    PARTITION BY partition_col
    ORDER BY order_col
    ROWS BETWEEN start AND end
) FROM table;
```

### 関数種類
| カテゴリ | 関数 | 用途 |
|---|---|---|
| 集約系 | SUM/AVG/COUNT/MAX/MIN | 累計・移動平均 |
| 順位系 | ROW_NUMBER/RANK/DENSE_RANK/NTILE | 連番・順位付け |
| 分析系 | LAG/LEAD/FIRST_VALUE/LAST_VALUE | 前後行参照 |

### フレーム指定
- `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` — 先頭から現在行（累計）
- `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW` — 直近3行（移動平均）
- `ROWS`はフレーム行数指定、`RANGE`は値範囲指定

### RANK vs DENSE_RANK vs ROW_NUMBER
- RANK: 同順位があると次の順位を飛ばす（1,2,2,4）
- DENSE_RANK: 飛ばさない（1,2,2,3）
- ROW_NUMBER: 必ず連番（1,2,3,4）— 同順位でも一意

### 実務Tips
- ウィンドウ関数はWHERE句の後に評価される。フィルタ後の結果に対して動作する点に注意
- 複数のウィンドウ定義を使い回す場合は `WINDOW w AS (...)` で名前付き定義を活用

## 3.2 再帰CTE (WITH RECURSIVE)
```sql
WITH RECURSIVE cte AS (
    SELECT id, parent_id, name, 1 AS depth
    FROM org WHERE parent_id IS NULL        -- アンカーメンバ
    UNION ALL
    SELECT o.id, o.parent_id, o.name, c.depth + 1
    FROM org o JOIN cte c ON o.parent_id = c.id  -- 再帰メンバ
)
SELECT * FROM cte;
```
- **用途**: BOM（部品表）展開、組織階層、グラフ探索、経路探索
- **原理**: アンカーで初期行を生成→再帰メンバで繰返し拡張→空集合になったら終了
- **注意**: 無限ループ防止のため最大再帰深度を設定（MAXRECURSION）
- **実務例**: カテゴリツリーのパンくずリスト生成、依存関係の推移閉包算出

## 3.3 MERGE文
```sql
MERGE INTO target t USING source s ON (t.id = s.id)
WHEN MATCHED THEN UPDATE SET t.name = s.name
WHEN NOT MATCHED THEN INSERT (id, name) VALUES (s.id, s.name);
```
- INSERT/UPDATE/DELETEを1文で実行（upsert）
- ETLパイプラインやマスタ同期処理で多用される

## 3.4 高度な集約
- `GROUP BY ROLLUP(a, b)`: 小計+総計を自動生成
- `GROUP BY CUBE(a, b)`: 全組合せの集計
- `GROUP BY GROUPING SETS((a),(b))`: 指定した組合せのみ
- `GROUPING()`: NULL行が集計行かどうかを判定（0=実データ、1=集計行）

## 3.5 EXISTS vs IN
- **EXISTS**: 相関副問合せ。行が1つでも見つかればtrue（NULLに安全）
- **IN**: 値リストとの照合。NULLが含まれるとNOT INが期待通り動かない
- **よくある設計ミス**: NOT IN (NULL含むリスト) → 常にUNKNOWN → 結果0行。NOT EXISTSを使うべき場面でNOT INを使ってしまうバグ

## 3.6 トリガー
```sql
CREATE TRIGGER trg AFTER INSERT ON orders
FOR EACH ROW
BEGIN
    UPDATE inventory SET stock = stock - NEW.quantity WHERE id = NEW.product_id;
END;
```
- BEFORE/AFTER × INSERT/UPDATE/DELETE × FOR EACH ROW/STATEMENT
- **注意**: トリガーの連鎖（カスケード）による予期せぬ動作に注意
- **設計指針**: トリガーの多用はデバッグ困難の原因。アプリケーション層で処理できるならそちらを優先

# 4. トランザクション管理

## 4.1 分離レベル
| レベル | ダーティリード | 非再現リード | ファントムリード |
|---|:---:|:---:|:---:|
| READ UNCOMMITTED | 発生 | 発生 | 発生 |
| READ COMMITTED | 防止 | 発生 | 発生 |
| REPEATABLE READ | 防止 | 防止 | 発生 |
| SERIALIZABLE | 防止 | 防止 | 防止 |

- **ダーティリード**: 未コミットデータを読む
- **非再現リード(ファジーリード)**: 同じ行を2回読むと値が変わっている
- **ファントムリード**: 同じ条件で2回検索すると行数が変わっている
- **実務での選択基準**: 多くのRDBMSのデフォルトはREAD COMMITTED。金融系や在庫管理など厳密な一貫性が必要な場面でSERIALIZABLEを検討

## 4.2 2相コミット (2PC)
1. **投票フェーズ**: 調停者 →「コミット準備できるか？」→ 全参加者が READY or ABORT を応答
2. **決定フェーズ**: 全員READY → COMMIT指示 / 1つでもABORT → ROLLBACK指示
- **原理**: 分散DBでACID特性を保証。ネットワーク分断時はブロッキング問題あり
- **3相コミット(3PC)**: ブロッキング問題を軽減（PRE-COMMITフェーズ追加）
- **よくある設計ミス**: 2相ロック方式(2PL)と混同する。2PL=ロック制御、2PC=分散コミット
- **実務代替**: マイクロサービスではSagaパターン（補償トランザクション）が主流

## 4.3 MVCC (Multi-Version Concurrency Control)
- 各行の更新前/更新後の複数バージョンを保持
- **読取りがロック不要**: 読取りは自分の開始時点のスナップショットを参照
- **原理**: トランザクション開始時のタイムスタンプで「見えるバージョン」を決定
- MySQL InnoDB: REPEATABLE READ + MVCC（ファントムリードも実質防止）
- PostgreSQL: MVCC + SSI（Serializable Snapshot Isolation）

## 4.4 ロック
- **共有ロック(S)**: 読取り用。複数同時取得可。排他ロックとは排他
- **排他ロック(X)**: 書込み用。他のS/Xロックと排他
- **2相ロック方式(2PL)**: 成長相（ロック取得のみ）→ 縮退相（ロック解放のみ）
  - 直列化可能性を保証（デッドロック可能性あり）
- **デッドロック検出**: 待ちグラフで循環を検出→若いトランザクションをロールバック
- **デッドロック防止策**: ロック取得順序の統一、タイムアウト設定、楽観的ロック（バージョン番号）の活用

## 4.5 障害回復
- **WAL(Write-Ahead Logging)**: データ変更前にログを先に書く
- **チェックポイント**: その時点でのメモリ上の変更をディスクに反映
- **REDO(ロールフォワード)**: コミット済みだがディスク未反映のトランザクションを再実行
- **UNDO(ロールバック)**: 未コミットのトランザクションの変更を取り消し
- **ARIES**: WALベースの代表的回復アルゴリズム（分析→REDO→UNDO）

# 5. 物理設計・性能チューニング

## 5.1 インデックス
| 種類 | 特徴 | 適用場面 |
|---|---|---|
| B+木 | 範囲検索・等価検索両対応 | 最も汎用的（デフォルト） |
| ハッシュ | 等価検索のみ高速 | 主キー検索 |
| ビットマップ | カーディナリティ低い列 | DWH・分析系 |
| 関数 | 式の結果にインデックス | UPPER(name)等 |

- **クラスタ化インデックス**: インデックス順=レコード格納順。テーブルに1つのみ
- **カバリングインデックス**: インデックスだけで問合せ完結（テーブルアクセス不要）
- **複合インデックス**: 左端一致の原則（(A,B,C)でA単独、A+Bは使えるがB単独は使えない）
- **よくある設計ミス**: 全列にインデックスを張る（更新コスト増大）。SELECT対象ではなくWHERE/JOIN/ORDER BY対象列を優先

## 5.2 パーティショニング
- **レンジ**: 値の範囲で分割（日付が最も一般的）
- **リスト**: 離散値で分割（地域コード等）
- **ハッシュ**: ハッシュ値で均等分割
- **パーティションプルーニング**: 不要なパーティションをスキップして高速化
- **設計指針**: パーティションキーはWHERE句で頻繁に使われる列を選ぶ。過剰な分割は管理コスト増

## 5.3 性能チューニング手法
1. **実行計画(EXPLAIN)分析**: フルスキャン→インデックス追加
2. **統計情報更新**: ANALYZEでオプティマイザの判断精度向上
3. **非正規化**: 頻繁な結合→冗長列追加
4. **マテリアライズドビュー**: 集計結果のキャッシュ（リフレッシュ戦略が重要）
5. **テーブル分割**: 水平（行）/垂直（列）
6. **コネクションプール**: DB接続の再利用
7. **バッファキャッシュ**: ディスクI/O削減

# 6. 分散DB・NoSQL・DWH

## 6.1 CAP定理
- **C(一貫性)**: 全ノードが同じデータを返す
- **A(可用性)**: 常に応答を返す
- **P(分断耐性)**: ネットワーク分断でも動作
- **CP**: 分断時に一部ノードが応答不可（RDBMS的）
- **AP**: 分断時に古いデータを返しうる（NoSQL的）
- **原理**: ネットワーク分断は避けられない→CA同時は実現不可→CPかAPを選択
- **PACELC拡張**: 分断がない通常時もLatency vs Consistencyのトレードオフがある

## 6.2 NoSQL
| 種類 | 代表 | データモデル | 適用場面 |
|---|---|---|---|
| KVS | Redis/DynamoDB | キー→値 | セッション/キャッシュ |
| ドキュメント | MongoDB | JSON/BSON | CMS/ログ |
| カラムファミリ | Cassandra/HBase | 行キー→列ファミリ | 大規模分散 |
| グラフ | Neo4j | ノード+エッジ | SNS/推薦 |

- **BASE特性**: Basically Available, Soft state, Eventually consistent
- ACIDの対概念。結果整合性を許容して可用性・スケーラビリティを優先
- **データストア選定**: アクセスパターン（読み書き比率・クエリパターン）を先に定義し、それに合ったモデルを選ぶ

## 6.3 DWH・OLAP
- **ETL**: Extract(抽出)→Transform(変換)→Load(ロード)
- **ELT**: 近年主流。ロード後に変換（クラウドDWHの計算力を活用）
- **スタースキーマ**: 中央にファクトテーブル、周囲にディメンションテーブル
- **OLAP操作**: スライス(次元固定)/ダイス(部分立方体)/ドリルダウン(詳細化)/ドリルアップ(集約)/ピボット(軸回転)
- **スキーマ・オン・ライト** vs **スキーマ・オン・リード**: DWH vs データレイク

# 7. セキュリティ（DB固有）
- **アクセス制御**: GRANT/REVOKE、ロールベース。最小権限の原則を徹底
- **暗号化**: 透過的データ暗号化(TDE)、列レベル暗号化
- **SQLインジェクション対策**: プレースホルダ/パラメータ化クエリ（最重要）
- **監査ログ**: 誰が・いつ・何を操作したかの記録
- **バックアップ**: フル/差分/増分。RPO(目標復旧時点)に応じた戦略
- **接続管理**: アプリケーション専用ユーザーの分離、パスワードローテーション
