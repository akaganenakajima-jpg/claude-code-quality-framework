# ITサービスマネジメント — 開発リファレンス

## 1. ITサービスマネジメント概論

### ITIL4の全体像

ITIL（Information Technology Infrastructure Library）は、ITサービスマネジメントのベストプラクティス集。2019年にITIL4として大幅改訂され、従来のプロセス中心アプローチから「価値の共創」を中心に据えた包括的フレームワークへ進化した。

#### ITIL v3/2011からITIL4への主な変更点
| 観点 | ITIL v3/2011 | ITIL4 |
|------|-------------|-------|
| 中心概念 | サービスライフサイクル（5段階） | サービスバリューシステム（SVS） |
| 構成要素 | 26プロセス＋4機能 | 34プラクティス |
| アプローチ | プロセスアプローチ | ホリスティック（全体最適） |
| 対応範囲 | ウォーターフォール前提 | アジャイル・DevOps・リーン対応 |
| 価値の考え方 | プロバイダが価値を提供 | 顧客と共に価値を共創 |

### サービスバリューシステム（SVS）

SVSは「機会/需要」を入力として受け取り、「価値」を出力するシステム。以下の5つのコンポーネントで構成される。

```
[機会/需要] → [サービスバリューシステム] → [価値]
                    │
    ┌───────────────┼───────────────┐
    │               │               │
 ガイディング    ガバナンス     継続的改善
  プリンシプル
    │               │               │
    └───────┬───────┘               │
            │                       │
    サービスバリューチェーン ←── プラクティス(34)
```

**5つのコンポーネント**:
1. **ガイディングプリンシプル（指針となる原則）**: あらゆる状況で適用可能な推奨事項
2. **ガバナンス**: 組織の方向付けと統制
3. **サービスバリューチェーン（SVC）**: 価値を創出する6つの活動
4. **プラクティス**: 34の管理プラクティス
5. **継続的改善**: 全レベルでの改善活動

### サービスバリューチェーン（SVC）の6つの活動

マイケル・ポーターのバリューチェーンをサービスマネジメントに応用したモデル。

1. **プラン（Plan）**: ビジョン・戦略・方針の策定。全活動を方向付ける
2. **改善（Improve）**: 全活動・プラクティス・バリューストリームの継続的改善
3. **エンゲージ（Engage）**: ステークホルダーとの関係構築・需要理解・透明性確保
4. **デザイン＆トランジション（Design & Transition）**: 要件を満たす製品・サービスの設計と移行
5. **オブテイン/ビルド（Obtain/Build）**: サービスコンポーネントの取得・構築・統合
6. **デリバー＆サポート（Deliver & Support）**: サービスの提供と日常的なサポート

**バリューストリーム**: SVC上で「機会/需要」→「価値」の流れを、組織固有の課題に対して具体化したシナリオ群。部分最適（サイロ化）を防ぎ全体最適を実現する。

### 4つの側面（Four Dimensions of Service Management）

全てのサービスの設計・運用で考慮すべき4つの観点:

1. **組織と人材（Organizations and People）**: 組織構造、文化、スキル、コンピテンシー、コミュニケーション
2. **情報と技術（Information and Technology）**: 情報管理、技術選定、ツール、データ管理
3. **パートナーとサプライヤ（Partners and Suppliers）**: 外部組織との関係、サービス統合、調達戦略
4. **バリューストリームとプロセス（Value Streams and Processes）**: 活動の流れ、手順、管理策の設計

これらは外部要因（PESTLE: 政治・経済・社会・技術・法律・環境）の影響を受ける。

### 7つのガイディングプリンシプル

1. **価値に着目する（Focus on value）**: 全ての活動は顧客・ステークホルダーへの価値に結びつくべき
2. **現状からはじめる（Start where you are）**: ゼロからではなく既存の資産・能力を活用する
3. **フィードバックをもとに反復して進化する（Progress iteratively with feedback）**: 大きな変更を一度にせず、小さな反復で改善する
4. **協働し、可視性を高める（Collaborate and promote visibility）**: サイロを壊し、透明性を確保する
5. **包括的に考え、取り組む（Think and work holistically）**: 部分最適ではなく全体最適を追求する
6. **シンプルにし、実践的にする（Keep it simple and practical）**: 不要な複雑さを排除する
7. **最適化し、自動化する（Optimize and automate）**: まず最適化してから自動化する（順序が重要）

### ISO/IEC 20000（ITSMS）

#### 規格の概要
- ITサービスマネジメントシステム（ITSMS）に関する国際規格
- 日本版: JIS Q 20000
- ITILがベストプラクティス（推奨）であるのに対し、ISO 20000は認証規格（要求事項の全てを満たす必要あり）

#### 規格の構成
- **ISO/IEC 20000-1:2018（JIS Q 20000-1:2020）**: サービスマネジメントシステム要求事項（認証基準）
- **ISO/IEC 20000-2:2019（JIS Q 20000-2:2013）**: 適用の手引

#### 主な要求事項（箇条構成）
1. **箇条4 組織の状況**: 利害関係者のニーズ、SMSの適用範囲
2. **箇条5 リーダーシップ**: トップマネジメントのコミットメント、方針策定
3. **箇条6 計画**: リスクと機会への取り組み、目標設定
4. **箇条7 SMS の支援**: 資源、力量、認識、コミュニケーション、文書化
5. **箇条8 SMS の運用**: 運用の計画・管理、サービスポートフォリオ、関係と合意、供給と需要、サービスの設計・構築・移行、解決と実現、サービス保証
6. **箇条9 パフォーマンス評価**: 監視・測定・分析・評価、内部監査、マネジメントレビュー
7. **箇条10 改善**: 不適合と是正処置、継続的改善

#### ITILとISO 20000の関係
- ITILは「何をすべきか（What）」のガイドライン。部分的採用が可能
- ISO 20000は「何を満たすべきか（Shall）」の要求事項。全要求事項の充足が必要
- 実務ではITILで設計し、ISO 20000で認証取得するパターンが一般的
- ITIL4発行後、ISO 20000:2018も整合性を保つよう改訂されている

#### よくある誤解
- **誤**: ITILを全て実装すればISO 20000認証が取れる → **正**: ITIL準拠は認証の前提ではない。ISO 20000の要求事項を個別に満たす必要がある
- **誤**: ISO 20000はIT部門だけの話 → **正**: トップマネジメントのコミットメントと組織全体の関与が要求される
- **誤**: 一度認証を取れば終わり → **正**: 定期審査（サーベイランス）と更新審査が必要。PDCAによる継続的改善が求められる

### 実務活用
- ITIL4のSVSモデルは、運用組織のAs-Is/To-Be分析のフレームワークとして活用できる
- 4つの側面は、サービス設計レビューのチェックリストとして有効
- 7つの原則は、改善施策の優先度判断基準として日常的に使える

## 2. サービスデスク・インシデント管理

### サービスデスクの定義と役割

サービスデスクは、ユーザーとITサービスプロバイダの間の**単一連絡窓口（SPOC: Single Point of Contact）**。ITIL4ではプラクティスとして定義されている。

#### サービスデスクの組織形態
| 形態 | 特徴 | 適用場面 |
|------|------|----------|
| **ローカルサービスデスク** | ユーザーと同じ拠点に設置 | 拠点固有の問題が多い場合 |
| **中央サービスデスク** | 一箇所に集約 | コスト効率重視・標準化が進んだ組織 |
| **バーチャルサービスデスク** | 地理的に分散しつつ仮想的に統合 | グローバル組織・リモートワーク環境 |
| **フォロー・ザ・サン** | 時差を利用して24時間カバー | 24/365サービスが必要な場合 |

#### サービスデスクの機能
- インシデントの受付・記録・分類・優先度付け
- 初期診断と一次解決（ナレッジベース活用）
- エスカレーションの判断と実施
- ユーザーへの進捗報告とクローズ
- サービス要求の処理
- 顧客満足度調査の窓口

### インシデント管理

#### 定義
**インシデント**: ITサービスの計画外の中断、またはITサービスの品質低下を引き起こす事象。サービスに影響を及ぼす可能性がある事象も含む。

#### 目的
インシデント管理の目的は「根本原因の解決」ではなく、**ITサービスを可能な限り迅速に通常状態に復旧させること**。恒久対策は問題管理に委ねる。

#### インシデント管理の5つのプロセス

```
検知・記録 → 分類・優先度付け → 調査・診断 → 解決・復旧 → クローズ
```

1. **検知と記録**: ユーザー報告、監視ツール、イベント管理からの検知。全インシデントを一意のIDで記録
2. **分類と優先度付け**: カテゴリ分類（ハードウェア/ソフトウェア/ネットワーク等）と優先度決定
3. **調査と診断**: 原因の特定。ナレッジベース・既知エラーDBの参照
4. **解決と復旧**: ワークアラウンドまたは恒久的修正の適用。サービスの正常復旧確認
5. **インシデントのクローズ**: ユーザーへの復旧確認。記録の完成。満足度確認

#### インシデントの分類

**カテゴリ分類**（例）:
- ハードウェア（サーバー/ネットワーク機器/PC/プリンタ等）
- ソフトウェア（OS/アプリケーション/ミドルウェア等）
- ネットワーク（LAN/WAN/インターネット/VPN等）
- セキュリティ（不正アクセス/マルウェア/情報漏洩等）
- サービス（メール/Web/ファイルサーバー等）

**サブカテゴリ**: 上記を更に細分化（例: ソフトウェア→ERP→会計モジュール→仕訳入力機能）

#### 優先度の決定

優先度は**影響度（Impact）**と**緊急度（Urgency）**のマトリクスで決定する。

| | 緊急度: 高 | 緊急度: 中 | 緊急度: 低 |
|---|-----------|-----------|-----------|
| **影響度: 高** | P1（最優先） | P2（高） | P3（中） |
| **影響度: 中** | P2（高） | P3（中） | P4（低） |
| **影響度: 低** | P3（中） | P4（低） | P5（計画的対応） |

- **影響度**: 影響を受けるユーザー数、業務への影響範囲、財務的損失の大きさ
- **緊急度**: 対応の時間的制約、ビジネスへの即時的影響、SLA違反までの残り時間
- **優先度**: 実際にリソースを割り当てる順序を決定する指標

**SLAとの連動**: 優先度ごとに目標解決時間を設定する（例: P1=1時間、P2=4時間、P3=8時間、P4=24時間）

#### エスカレーション

**機能的エスカレーション（水平エスカレーション）**:
- 技術的にサービスデスクで解決できない場合、より専門的なチームへ引き継ぐ
- 1次→2次→3次サポート、またはベンダーサポート
- 例: サービスデスク→サーバー管理チーム→データベース管理チーム→ベンダー

**階層的エスカレーション（垂直エスカレーション）**:
- 管理者・経営層への報告・判断を仰ぐ
- リソース追加の承認、ビジネス判断が必要な場合
- SLA違反のリスクがある場合
- 例: サービスデスク担当→サービスデスクマネージャ→IT部門長→CIO

**自動エスカレーション**: 一定時間経過しても解決しない場合に自動的にエスカレーションするルール

#### 重大インシデント（Major Incident）

通常のインシデント管理プロセスとは別に、**専用の手順**で対応する。

**重大インシデントの特徴**:
- ビジネスに重大な影響を与えるインシデント
- 通常の対応プロセスでは間に合わない
- 専任のインシデントマネージャーが指揮を執る
- 経営層への即時報告が必要

**重大インシデント管理の流れ**:
1. 重大インシデントの宣言（基準に基づく判断）
2. インシデントマネージャーの任命
3. 対応チームの招集（技術者・管理者・コミュニケーション担当）
4. コミュニケーション計画の策定（ステークホルダーへの定期報告）
5. 復旧作業の実施と進捗管理
6. 暫定復旧・本復旧の確認
7. 事後レビュー（PIR: Post Incident Review）の実施

**KPI**:
- 平均復旧時間（MTTR: Mean Time To Restore）
- 初回解決率（FCR: First Contact Resolution Rate）
- 再オープン率
- SLA達成率
- ユーザー満足度

### よくある誤解
- **誤**: インシデント管理で根本原因を追究すべき → **正**: 復旧が最優先。根本原因分析は問題管理の役割。現場では「まず復旧、後で分析」を徹底する
- **誤**: 優先度が高い＝緊急度が高い → **正**: 優先度は影響度と緊急度の組み合わせで決まる
- **誤**: エスカレーションは「たらい回し」 → **正**: 適切なエスカレーションは品質向上の仕組み。オーナーシップはサービスデスクが保持する
- **誤**: 重大インシデントは優先度P1のインシデントと同じ → **正**: 重大インシデントは別プロセスで管理される。P1でも重大インシデントに該当しない場合がある

### 実務Tips
- インシデント分類体系は、監視ツールのアラートカテゴリと連動させると自動分類が可能
- 優先度マトリクスはSLAと紐付けて自動計算させることで属人化を防ぐ
- PIRの結果を問題管理や変更管理にフィードバックすることで再発防止サイクルが回る
- **War Room（ブリッジコール）の運用**: 重大インシデント時は全関係者が同一チャネルに集合し、15分間隔でステータス更新する体制を事前に整備しておく
- **インシデント記録のテンプレート化**: 「いつ・何が・誰が・どう対応・結果」を定型フォームにしておくと後続の分析精度が上がる

## 3. 問題管理

### 定義

**問題（Problem）**: 1つまたは複数のインシデントの根本原因となる、未知の原因。
**既知のエラー（Known Error）**: 根本原因が特定され、ワークアラウンドまたは恒久的解決策が文書化された問題。
**ワークアラウンド（Workaround）**: 恒久的解決策が適用されるまでの間、影響を軽減するための暫定的な回避策。

### 問題管理の目的

インシデントの根本原因を特定し、恒久的な解決策を実施することで、インシデントの再発を防止し、未然に防げないインシデントの影響を最小化すること。

### 問題管理の3つのフェーズ

#### フェーズ1: 問題の識別（Problem Identification）
- インシデントのトレンド分析から繰り返しパターンを発見
- 重大インシデントのPIRからの問題特定
- サプライヤ・パートナーからの通知
- インフラストラクチャ・アプリケーションの分析

#### フェーズ2: 問題コントロール（Problem Control）
- **問題の分析**: 根本原因分析（RCA）の実施
- **優先順位付け**: ビジネスインパクト、発生頻度、解決の複雑さを考慮
- **ワークアラウンドの文書化**: 恒久的解決までの暫定対策

**根本原因分析（RCA）の主な手法**:
| 手法 | 概要 | 適用場面 |
|------|------|----------|
| **なぜなぜ分析（5 Whys）** | 「なぜ？」を繰り返して真因に到達 | 単純な因果関係の追跡 |
| **特性要因図（フィッシュボーン）** | 原因を「人・機械・方法・材料・環境・測定」に分類 | 複合的な原因の整理 |
| **パレート分析** | 80:20の法則で影響の大きい原因に集中 | 優先順位付け |
| **FTA（故障の木解析）** | トップダウンで故障原因を論理ゲートで展開 | 複雑なシステム障害 |
| **クロノロジカル分析** | 時系列で事象を並べて因果関係を特定 | タイミング依存の障害 |
| **ケプナー・トリゴー法** | 状況把握→問題分析→決定分析→リスク分析 | 体系的な問題解決 |

#### フェーズ3: エラーコントロール（Error Control）
- 既知のエラーの管理と恒久的解決策の評価
- 変更管理への変更要求（RFC）の提出
- 解決策の有効性検証
- 既知のエラーの更新またはクローズ

### 既知のエラーデータベース（KEDB: Known Error Database）

- 既知のエラーとワークアラウンドを蓄積するデータベース
- インシデント管理がKEDBを参照することで解決時間を短縮
- サービスデスクの一次解決率向上に直結
- 定期的なレビューと更新が必要

**KEDBの構成要素**:
- 問題ID / 既知のエラーID
- 症状の説明
- 根本原因
- ワークアラウンド（暫定策）
- 恒久的解決策（ある場合）
- 影響を受けるCI（構成アイテム）
- 関連するインシデントID

### リアクティブ問題管理 vs プロアクティブ問題管理

| 観点 | リアクティブ | プロアクティブ |
|------|-------------|---------------|
| **トリガー** | インシデントの発生・再発 | トレンド分析・予防的調査 |
| **目的** | 再発防止 | 未然防止 |
| **活動** | インシデントから問題を特定し根本原因を分析 | インシデントデータの傾向分析、インフラの脆弱性調査 |
| **タイミング** | インシデント発生後 | 定期的・計画的に実施 |
| **例** | 同じDBエラーが3回発生→メモリリークが原因と特定 | 月次でインシデント分類別の増加傾向を分析→予防措置 |

### 問題管理とインシデント管理の連携

```
インシデント発生 → インシデント管理（復旧）
    │
    ├→ 単発で再発しない → クローズ
    │
    └→ 再発する/重大 → 問題管理（根本原因分析）
                            │
                            ├→ ワークアラウンド文書化 → KEDB登録
                            │
                            └→ 恒久的解決策 → RFC → 変更管理
```

### よくある誤解
- **誤**: 全てのインシデントに問題管理が必要 → **正**: 再発するインシデントや重大インシデントが主な対象。1回限りの軽微なインシデントまで全て問題化する必要はない
- **誤**: ワークアラウンドがあれば問題はクローズできる → **正**: ワークアラウンドは暫定策。恒久的解決が完了するか、コスト対効果で対応不要と判断されるまで問題はオープンのまま
- **誤**: 問題管理は技術的な活動のみ → **正**: プロセス・手順・人的要因（ヒューマンエラー）も問題管理の対象
- **誤**: リアクティブ問題管理だけで十分 → **正**: プロアクティブ問題管理がインシデント発生件数自体を減らす。成熟した組織はプロアクティブの比率を高める

### 実務Tips
- パレート分析で「上位20%の原因が80%のインシデントを生む」ことを示し、経営層へのリソース要求を正当化できる
- KEDBの充実はサービスデスクの一次解決率向上と平均解決時間短縮に直結する
- 問題管理の成果指標（再発率の低下、KEDB利用率）を定期的に測定・報告する
- **SREとの融合**: ポストモーテム（PIR）で「ブレームレス（非難なし）」の文化を醸成し、問題の正直な共有を促進する
- **問題の棚卸し**: 四半期ごとにオープン中の問題一覧をレビューし、対応不要と判断したものは理由を記録してクローズする

## 4. 変更管理

### 定義（ITIL4: Change Enablement）

ITIL4では「変更管理（Change Management）」から「変更実現（Change Enablement）」に名称変更された。目的は、リスクを適切に管理しながらITサービスへの変更を効率的に実現すること。

**変更（Change）**: サービスに影響を及ぼす可能性のある、あらゆるIT関連の追加・修正・除去。

### 変更の3つのタイプ

| タイプ | 説明 | 承認方法 | 例 |
|--------|------|----------|-----|
| **標準変更（Standard Change）** | 事前承認済みの低リスク変更。手順が確立されている | 事前承認（都度承認不要） | パスワードリセット、定例パッチ適用 |
| **通常変更（Normal Change）** | 標準でも緊急でもない変更。CABで評価・承認 | CABまたは変更権限者 | サーバー追加、アプリケーション更新 |
| **緊急変更（Emergency Change）** | 重大インシデント対応等で緊急に実施が必要な変更 | ECAB（緊急変更諮問委員会）で迅速承認 | セキュリティパッチの緊急適用 |

### 変更諮問委員会（CAB: Change Advisory Board）

**構成メンバー**:
- 変更マネージャー（議長）
- IT各部門の技術リーダー
- サービスオーナー
- アプリケーション管理者
- 運用管理者
- セキュリティ担当者
- （必要に応じて）ビジネス側代表者、サプライヤ

**CABの役割**:
- 変更要求（RFC）の評価とアドバイス提供
- 変更の影響分析のレビュー
- 変更スケジュール（変更カレンダー）の管理
- 変更実施後のレビュー

**ECAB（Emergency CAB）**:
- 緊急変更時に招集される少人数の意思決定機関
- 通常のCABプロセスを短縮して迅速に承認
- 事後レビューで通常の変更管理プロセスに記録を補完

### 変更管理プロセス

```
RFC作成 → 記録・分類 → 影響/リスク評価 → CAB審議 → 承認/却下
    │                                                    │
    │         ← 却下（理由記録）─────────────────────────┘
    │
    ↓ 承認
実施計画策定 → 実施（リリース管理と連携） → テスト → 本番適用
    │
    ↓
PIR（変更後レビュー） → クローズ
```

### RFC（Request for Change）に含める情報
- 変更の説明と理由
- 影響を受けるCI（構成アイテム）
- リスク評価結果
- 実施計画（手順・スケジュール・担当者）
- テスト計画
- **バックアウト計画（切り戻し手順）** — 全変更に必須
- 必要なリソースとコスト

### 変更スケジュール（Change Schedule）

- 承認済みの全変更を時系列で管理するカレンダー
- **フリーズ期間（Change Freeze）**: 年末年始・決算期など、変更を原則禁止する期間
- **メンテナンスウィンドウ**: 変更実施を許可する時間帯（通常は業務時間外）
- リリース管理との整合性を確保

### リリース管理との関係

変更管理が「何を変更するか」の承認プロセスであるのに対し、リリース管理は「どう展開するか」の実行プロセス。

**リリースの種類**:
| 種類 | 説明 |
|------|------|
| **メジャーリリース** | 大規模な機能追加・変更 |
| **マイナーリリース** | 小規模な機能改善・バグ修正 |
| **緊急リリース** | 緊急変更に対応するリリース |

**展開方式**:
| 方式 | 説明 | リスク |
|------|------|--------|
| **ビッグバン** | 全ユーザーに一斉展開 | 高（失敗時の影響大） |
| **段階的（フェーズド）** | グループ毎に順次展開 | 中（フィードバック反映可能） |
| **カナリアリリース** | 少数ユーザーで先行展開 | 低（問題を早期発見） |
| **ブルーグリーン** | 2環境を切り替え | 低（即時切り戻し可能） |

### 7つのR（変更評価の観点）
1. **Raised**: 誰が変更を提起したか
2. **Reason**: 変更の理由は何か
3. **Return**: 変更によるリターン（期待される利益）は何か
4. **Risk**: 変更に伴うリスクは何か
5. **Resource**: 変更に必要なリソースは何か
6. **Responsible**: 変更の責任者は誰か
7. **Relationship**: 他の変更との関連は何か

### よくある誤解
- **誤**: 緊急変更はCAB承認なしで実施できる → **正**: ECABによる承認は必要。事後にCABでのレビューも必須
- **誤**: 標準変更は変更管理の対象外 → **正**: 事前承認されているだけで、記録と監視は行う。条件を逸脱すれば通常変更として扱う
- **誤**: バックアウト計画は「できれば作る」 → **正**: 全ての通常変更・緊急変更にバックアウト計画は必須。なければCABは承認すべきでない
- **誤**: 変更管理は官僚的で開発を遅らせる → **正**: ITIL4では「Change Enablement（変更実現）」として、変更を阻止するのではなく促進する役割を強調

### 実務Tips
- 変更成功率・失敗率・緊急変更比率を月次で測定し、プロセス改善のインプットにする
- 標準変更カタログを充実させることで、CAB審議の負荷を軽減
- CI/CDパイプラインとの統合で、自動テスト通過を変更承認の条件にする
- **バックアウト計画の実効性テスト**: 本番適用前にステージング環境でバックアウト手順を実際に実行し、所要時間を計測しておく
- **変更カレンダーの共有**: 開発・運用・ビジネス部門が同じカレンダーを参照することで、変更の衝突やフリーズ期間の認識齟齬を防ぐ

## 5. サービスレベル管理

### 定義と目的

サービスレベル管理（SLM: Service Level Management）は、ビジネスニーズに基づいた明確なサービスレベル目標を設定・合意し、ITサービスの提供がその目標を達成することを確実にするプラクティス。

### SLA（Service Level Agreement: サービスレベル合意書）

**定義**: ITサービスプロバイダと顧客の間で合意された、サービスの内容・範囲・品質目標を明文化した文書。

**SLAに含める主な項目**:
- サービスの説明と範囲
- サービス時間（例: 24/365、平日9:00-18:00）
- 可用性目標（例: 99.9%）
- パフォーマンス目標（例: レスポンスタイム3秒以内）
- インシデント対応目標（例: P1=1時間以内に復旧）
- セキュリティ要件
- 報告・レビュー頻度
- ペナルティ条項（未達時の措置）
- 例外条件（計画停止、不可抗力等）

**SLAの種類**:
| 種類 | 説明 | 例 |
|------|------|-----|
| **サービスベースSLA** | 特定のサービス単位で締結 | メールサービスのSLA |
| **顧客ベースSLA** | 特定の顧客単位で締結 | A事業部向け全サービスのSLA |
| **マルチレベルSLA** | 企業レベル＋顧客レベル＋サービスレベルの階層構造 | 全社共通SLA＋部門別SLA＋個別サービスSLA |

### OLA（Operational Level Agreement: 運用レベル合意書）

**定義**: ITサービスプロバイダ内部の部門間で合意された、SLAを支援するための内部合意書。

- SLAを達成するために必要な内部チーム間の役割分担・対応時間・品質目標を規定
- 例: ネットワークチームは障害通知後30分以内に一次診断を完了する

### UC（Underpinning Contract: 外部委託契約）

**定義**: ITサービスプロバイダと外部サプライヤの間で締結される、SLAを支援するための契約。

- SLAの達成に必要な外部サービスの品質・対応時間・責任を規定
- 例: データセンター事業者との契約で、電源障害時の復旧を2時間以内と規定

### SLA・OLA・UCの関係

```
[顧客] ←── SLA ──→ [ITサービスプロバイダ]
                         │
              ┌──────────┼──────────┐
              │                     │
         ←─ OLA ─→            ←─ UC ─→
    [内部チームA]  [内部チームB]  [外部サプライヤ]
```

**重要**: SLAの目標値は、OLAとUCの合計で達成可能でなければならない。

### サービスカタログ

**定義**: 利用可能な全ITサービスの一覧と、各サービスの詳細情報を管理する文書/システム。

**2種類のカタログ**:
- **ビジネスサービスカタログ**: 顧客向け。ビジネス用語で記述されたサービス一覧
- **テクニカルサービスカタログ**: IT内部向け。技術的な構成要素とサービスの関係

### サービスレベル目標（SLO: Service Level Objective）

SLAの中で定義される個々の測定可能な目標値。

**代表的なSLO**:
| カテゴリ | SLO例 | 測定方法 |
|----------|--------|----------|
| 可用性 | 99.9%（月間ダウンタイム43.8分以内） | 監視ツールによる稼働時間計測 |
| パフォーマンス | 画面レスポンス3秒以内（95パーセンタイル） | APMツール |
| インシデント対応 | P1インシデント1時間以内に復旧 | インシデント管理ツール |
| サービス要求 | パスワードリセット30分以内に対応 | サービスデスクツール |

### サービスレビュー

定期的にSLA達成状況を顧客とレビューし、改善点を合意するプロセス。

**レビュー内容**:
- SLA達成率の報告
- 重大インシデントの振り返り
- サービス改善計画（SIP: Service Improvement Plan）の進捗
- 顧客満足度調査結果
- 次期のサービスレベル目標の見直し

### よくある誤解
- **誤**: SLAは達成不可能な高い目標を設定すべき → **正**: 達成可能かつ測定可能な目標を段階的に引き上げる
- **誤**: SLAを締結すればサービス品質は自動的に上がる → **正**: SLAは「合意」であり、達成するための仕組み（OLA・UC・プロセス）が必要
- **誤**: OLAはSLAの下位文書 → **正**: OLAはSLAとは別の合意書。SLAの達成を支援する内部合意
- **誤**: 可用性99.9%と99.99%の差は小さい → **正**: 月間ダウンタイムで43.8分vs4.3分。10倍の差がある

### 実務Tips
- SLAレビュー会議の議事録は、改善要求の正当化根拠として活用する
- **ウォーターメロンSLA**に注意: 表面は緑（=SLO達成）だが中は赤（=顧客不満足）。技術指標だけでなくユーザー体験メトリクスも含める
- **SLA交渉のコツ**: 初年度は現状ベースライン+αの目標から始め、年次レビューで段階的に引き上げる。いきなり高い目標を約束しない
- **SREのエラーバジェットとの連携**: SLO未達時にエラーバジェットが枯渇→新機能リリースを一時停止し信頼性改善に集中、という運用ルールが効果的
- 計画停止をSLAで除外する合意がある場合、稼働率計算では計画停止時間を分母・分子ともに除外する

## 6. 可用性管理

### 定義と目的

可用性管理は、ITサービスの可用性がビジネスニーズに合致した合意レベルを満たすことを確実にするプラクティス。コストとのバランスを考慮しつつ、最適な可用性設計を行う。

### 可用性の基本指標

#### MTBF（Mean Time Between Failures: 平均故障間隔）
- 復旧完了から次の故障発生までの平均稼働時間
- **MTBFが長い = 信頼性が高い**
- 計算: MTBF = 総稼働時間 / 故障回数

#### MTTR（Mean Time To Restore/Repair: 平均復旧時間）
- 故障発生から復旧完了までの平均時間
- **MTTRが短い = 保守性が高い**
- 計算: MTTR = 総ダウンタイム / 故障回数
- MTTRの内訳: 検知時間 + 診断時間 + 修復時間 + 復旧確認時間

#### MTRS（Mean Time to Restore Service: 平均サービス復旧時間）
- ITILではMTTRよりもMTRSを重視。サービスがユーザーに利用可能になるまでの時間

#### MTTF（Mean Time To Failure: 平均故障時間）
- 修理不能な機器の指標（MTBFは修理可能な機器の指標）。混同に注意

#### 稼働率（Availability）の計算

**基本公式**:
```
稼働率 = MTBF / (MTBF + MTTR)
```

**SLA観点の計算**:
```
稼働率 = (合意サービス時間 - ダウンタイム) / 合意サービス時間 × 100%
```

**計算例**:
- 月間合意サービス時間: 720時間（24h × 30日）
- 月間ダウンタイム: 43.2分（0.72時間）
- 稼働率 = (720 - 0.72) / 720 = 99.9%

**稼働率とダウンタイムの関係**:
| 稼働率 | 年間ダウンタイム | 月間ダウンタイム |
|--------|-----------------|-----------------|
| 99% | 3日15時間36分 | 7時間18分 |
| 99.9% | 8時間45分36秒 | 43分48秒 |
| 99.99% | 52分33.6秒 | 4分23秒 |
| 99.999% | 5分15.4秒 | 26.3秒 |

### 直列・並列システムの可用性計算

**直列システム**（全コンポーネントが動作しないとサービス不可）:
```
全体稼働率 = A1 × A2 × A3 × ... × An
```
例: A=0.99, B=0.99 → 全体 = 0.99 × 0.99 = 0.9801 (98.01%)

**並列システム**（冗長構成。全て停止した時のみサービス不可）:
```
全体稼働率 = 1 - (1-A1) × (1-A2) × ... × (1-An)
```
例: A=0.99, B=0.99 → 全体 = 1 - (0.01 × 0.01) = 0.9999 (99.99%)

**複合システム**:
直列と並列の組み合わせ。並列部分を先に計算し、その結果を直列に組み込む。

### 単一障害点分析（SPOF: Single Point of Failure）

システム全体の可用性を低下させる、冗長化されていないコンポーネント。

**SPOFの特定方法**:
1. システム構成図を作成
2. 各コンポーネントが停止した場合の影響を分析
3. 冗長化されていない箇所を特定
4. ビジネスインパクトに基づき対策の優先順位を決定

**SPOF対策例**:
- サーバーのクラスタリング
- ネットワークの二重化
- ストレージのRAID構成
- 電源の二重化（UPS＋自家発電）
- データセンターの地理的分散

### VBF（Vital Business Function: 重要ビジネス機能）

ビジネスの核となる重要機能。VBFを支えるITサービスに対しては、より高い可用性目標を設定する。

**VBF特定のプロセス**:
1. ビジネスプロセスの洗い出し
2. 各プロセスのビジネスインパクト評価
3. VBFの特定と優先順位付け
4. VBFを支えるITサービス・コンポーネントのマッピング
5. 可用性要件の策定

### 可用性設計の原則

**高可用性（HA: High Availability）設計**:
- **冗長化**: アクティブ-スタンバイ / アクティブ-アクティブ
- **フェイルオーバー**: 障害検知時の自動切り替え
- **負荷分散**: ロードバランサーによる分散処理
- **耐障害性（フォールトトレランス）**: コンポーネント障害があってもサービス継続

**設計時の検討事項**:
- 可用性レベルとコストのトレードオフ
- 計画停止（メンテナンスウィンドウ）の確保
- 障害検知の自動化
- 自動復旧メカニズムの導入

### よくある誤解
- **誤**: 可用性99.9%で十分 → **正**: ビジネス要件による。金融系やECサイトでは99.99%以上が求められることも多い
- **誤**: 稼働率計算に計画停止を含める → **正**: SLAで「計画停止を除外する」と合意している場合、計画停止時間は分母・分子ともに除外する
- **誤**: 冗長化すればSPOFはなくなる → **正**: 冗長化の切り替え機構自体がSPOFになる場合がある。定期的なフェイルオーバーテストが必要
- **誤**: MTBFとMTTFは同じ → **正**: MTTFは修理不能な機器の指標。MTBFは修理可能な機器の指標

### 実務Tips
- SPOF分析は新規システム設計時のレビューチェックリストとして必須
- 可用性レポートはSLAレビューの核心資料。ダウンタイムの内訳（計画/非計画）を明示する
- **フェイルオーバー演習**: 少なくとも半年に1回は計画的にフェイルオーバーを発動し、切り替え時間と手順の実効性を確認する（Game Dayの文化）
- **カスケード障害への備え**: 直列構成でのドミノ倒しを防ぐため、サーキットブレーカーパターンやタイムアウト設計を組み合わせる

## 7. キャパシティ管理

### 定義と目的

キャパシティ管理（ITIL4: キャパシティとパフォーマンス管理）は、ITサービスに必要なリソースを適切なコストで、適切なタイミングに提供できるよう計画・管理するプラクティス。

### キャパシティ管理の3つのサブプロセス

#### BCM（Business Capacity Management: 事業キャパシティ管理）
- ビジネスの将来ニーズをITキャパシティ要件に変換
- ビジネス計画・成長予測からの要件導出
- 新規サービス・ビジネス変更の影響予測
- 長期的な容量計画の策定

#### SCM（Service Capacity Management: サービスキャパシティ管理）
- 現行ITサービスのパフォーマンスとキャパシティの管理
- SLA目標の達成状況の監視
- サービスのパフォーマンスデータの分析
- ボトルネックの特定と解消

#### CCM（Component Capacity Management: コンポーネントキャパシティ管理）
- ITインフラストラクチャの個々のコンポーネントのキャパシティ管理
- CPU・メモリ・ディスク・ネットワーク帯域の使用率監視
- 閾値ベースのアラート設定
- コンポーネントレベルのチューニング

### キャパシティ計画書

将来のキャパシティ需要を予測し、対応策を文書化した計画書。

**記載内容**:
- 現在のキャパシティ使用状況
- 将来の需要予測（ビジネス計画に基づく）
- 必要な増強計画とスケジュール
- コスト見積もり
- リスクと対策

### 需要管理（Demand Management）

ITサービスへの需要パターンを理解・予測・影響し、キャパシティ管理をサポートする。

**主な活動**:
- **PBA（Pattern of Business Activity）の分析**: ビジネス活動パターンの特定（月末処理、年度末集中等）
- **需要の平準化**: ピーク時の負荷分散（バッチ処理の夜間シフト等）
- **差別化された提供**: サービスレベルに応じたリソース配分

### パフォーマンスモニタリング

**監視対象**:
| レイヤー | 監視項目 |
|----------|----------|
| インフラ | CPU使用率、メモリ使用率、ディスクI/O、ネットワーク帯域 |
| ミドルウェア | DBコネクション数、キュー長、スレッドプール使用率 |
| アプリケーション | レスポンスタイム、スループット、エラー率 |
| サービス | トランザクション処理数、同時接続ユーザー数 |

**閾値管理**: 通常は使用率70-80%で警告、90%以上で緊急対応のアラート設定

### キャパシティ管理と可用性管理の違い

| 観点 | キャパシティ管理 | 可用性管理 |
|------|----------------|-----------|
| **目的** | リソースのコスト最適化と適切な提供 | サービスの継続的稼働の確保 |
| **焦点** | 需要と供給のバランス | 障害の防止と迅速な復旧 |
| **指標** | 使用率、スループット、レスポンスタイム | 稼働率、MTBF、MTTR |

### よくある誤解
- **誤**: キャパシティ管理はリソース追加だけの話 → **正**: チューニング・需要管理・最適化も含む。リソース追加は最後の手段
- **誤**: BCM/SCM/CCMは独立したプロセス → **正**: 3層が連携して初めて効果を発揮する。BCMの予測→SCMの分析→CCMの実行
- **誤**: クラウドなら無限にスケールするのでキャパシティ管理は不要 → **正**: コスト管理の観点でむしろ重要。オートスケーリングの上限・コスト制御が必要

### 実務Tips
- パフォーマンス劣化がSLA違反に至る前に予兆を捉えるプロアクティブな管理が重要
- 需要管理で「バッチ処理の時間帯分散」を提案すると、追加投資なしでピーク問題を解決できることがある
- クラウド環境ではFinOps（クラウドコスト最適化）との連携が必須
- **負荷テストの定期実行**: リリースごとに性能テストを実施し、キャパシティの変化傾向を追跡する

## 8. IT継続性管理

### 定義と目的

ITサービス継続性管理（ITSCM: IT Service Continuity Management）は、災害や重大な障害が発生した場合に、合意されたレベルのITサービスを継続または復旧できることを確実にするプラクティス。

### BIA（Business Impact Analysis: ビジネスインパクト分析）

災害発生時のビジネスへの影響を定量的・定性的に分析するプロセス。ITSCM計画策定の前提となる最重要活動。

**BIAの実施手順**:
1. **ビジネスプロセスの洗い出し**: 全業務プロセスの一覧化
2. **影響度評価**: 各プロセスが停止した場合の影響（財務・法的・評判・安全）
3. **最大許容停止時間（MTPD）の決定**: ビジネスが致命的ダメージを受けるまでの時間
4. **復旧優先順位の決定**: MTPDに基づく復旧順序
5. **依存関係の分析**: ITサービス・システム・人・施設の依存関係マッピング

### 復旧目標の指標

#### RTO（Recovery Time Objective: 目標復旧時間）
- **定義**: 災害発生からITサービスを復旧させるまでの目標時間
- **決定要因**: BIAの結果（MTPD以内に設定する必要がある）
- **例**: RTO=4時間 → 災害発生から4時間以内にサービスを復旧させる
- **RTOが短いほどコストが高い**（ホットスタンバイ等の高コスト対策が必要）

#### RPO（Recovery Point Objective: 目標復旧時点）
- **定義**: 災害発生時にどの時点までのデータを復旧できるかの目標
- **決定要因**: データ損失の許容範囲
- **例**: RPO=1時間 → 直近1時間以内のデータは復元可能
- **RPOが短いほどコストが高い**（リアルタイムレプリケーション等）

#### RLO（Recovery Level Objective: 目標復旧レベル）
- **定義**: 復旧後のサービスレベル（通常の何%で運用するか）
- **例**: RLO=80% → 通常の80%の処理能力で暫定運用

```
[正常運用]
    │
    ↓ ★ 災害発生
    │
    │←── RPO ──→│←────── RTO ──────→│
    │             │                     │
 データ損失       最終バックアップ      サービス復旧
 許容ポイント     時点                  (RLOレベルで)
```

### BCP（Business Continuity Plan: 事業継続計画）

災害発生時にビジネスを継続するための包括的な計画。ITSCMはBCPのIT領域をカバーする。

**BCPの主な要素**:
- 対応組織体制（災害対策本部の構成と役割）
- コミュニケーション計画（連絡網・報告ルート）
- ITサービス復旧手順
- 代替手段（手作業での業務継続等）
- サプライヤ・パートナーとの連携手順
- 復旧後の通常運用への移行手順

### DRP（Disaster Recovery Plan: 災害復旧計画）

BCPのIT技術面に特化した計画。

**DRPに含める内容**:
- 復旧対象システムの優先順位（BIAに基づく）
- 各システムのRTO/RPO
- バックアップ方式と復旧手順
- DR（ディザスタリカバリ）サイトの運用手順
- ネットワーク切り替え手順
- データ整合性確認手順

### バックアップ戦略

| 方式 | 説明 | RPO | コスト |
|------|------|-----|--------|
| **フルバックアップ** | 全データを毎回バックアップ | バックアップ間隔に依存 | 高 |
| **差分バックアップ** | 前回フルバックアップからの変更分 | フル+差分の間隔 | 中 |
| **増分バックアップ** | 前回バックアップからの変更分 | 増分の間隔 | 低 |
| **リアルタイムレプリケーション** | データをリアルタイムに複製 | ほぼ0 | 最高 |

**3-2-1ルール**: 3つのコピー、2種類のメディア、1つはオフサイト保管

### DRサイトの種類

| 種類 | 説明 | RTO | コスト |
|------|------|-----|--------|
| **ホットサイト** | 本番同等の設備が常時稼働。即座に切り替え可能 | 分〜時間 | 最高 |
| **ウォームサイト** | 設備はあるがデータ・設定の最終同期が必要 | 時間〜日 | 中 |
| **コールドサイト** | 場所と電源・ネットワークのみ。機器の搬入・設定が必要 | 日〜週 | 低 |
| **モバイルサイト** | 移動式のコンテナ型DR設備 | 日 | 中 |

### DRテストの種類

| テスト | 説明 | 現実性 | リスク |
|--------|------|--------|--------|
| **デスクトップ演習（机上演習）** | 関係者が集まりシナリオを読み上げ、対応を議論 | 低 | なし |
| **ウォークスルーテスト** | 復旧手順を実際に読み合わせ、問題点を洗い出す | 低〜中 | なし |
| **シミュレーションテスト** | 災害シナリオを模擬的に実行（システムは停止しない） | 中 | 低 |
| **並行テスト** | DRサイトで実際に復旧し、本番と並行稼働させて検証 | 高 | 低 |
| **完全中断テスト** | 本番を実際に停止しDRサイトに切り替え | 最高 | 高 |

### よくある誤解
- **誤**: RTO=0が理想 → **正**: RTO=0はコストが膨大。ビジネス要件に基づいた現実的なRTOを設定する
- **誤**: バックアップがあればDRは不要 → **正**: バックアップはデータの復旧であり、サービス全体の復旧手順（DRP）とは別
- **誤**: IT継続性計画は作成すれば終わり → **正**: 定期的なテスト・更新が必須。テストしていない計画は計画ではない
- **誤**: BIAは一度やれば良い → **正**: ビジネス環境の変化に合わせて定期的（年次以上）に見直す

### 実務Tips
- BIAの結果は、IT投資の優先順位付けの根拠資料として経営層に提示できる
- RTO/RPOの設定は、バックアップ方式とDRサイト選定の判断基準
- 年次のDRテスト実施と結果の文書化は、ISO 20000/ISO 27001の要求事項でもある
- **クラウドDRの活用**: AWSのCross-Region ReplicationやAzure Site Recoveryなどのマネージドサービスを使えば、低コストでウォームサイト相当のDRを実現できる
- **Chaos Engineering**: Netflixの「Chaos Monkey」のように、意図的に障害を注入して復旧手順の実効性を検証する手法も有効

## 9. 情報セキュリティ管理

### 定義と目的

情報セキュリティ管理は、組織の情報の機密性（Confidentiality）・完全性（Integrity）・可用性（Availability）を保護するプラクティス。CIA三要素と呼ばれる。

### ISMS（Information Security Management System）との連携

**ISO/IEC 27001**: 情報セキュリティマネジメントシステムの国際規格。
**ISO/IEC 27002**: セキュリティ管理策のベストプラクティス集。

ITサービスマネジメント（ISO 20000）と情報セキュリティ（ISO 27001）は密接に連携する。

**連携ポイント**:
- インシデント管理: セキュリティインシデントは通常のインシデント管理と統合して管理
- 変更管理: セキュリティ影響評価を変更管理プロセスに組み込む
- 可用性管理: セキュリティ対策が可用性に与える影響の評価
- 構成管理: CMDBにセキュリティ属性（パッチ状態・脆弱性情報）を含める

### セキュリティインシデント

**定義**: 情報セキュリティ方針への違反、管理策の不具合、またはセキュリティに関連する可能性のある事象。

**セキュリティインシデントの分類**:
| カテゴリ | 例 |
|----------|-----|
| 不正アクセス | 権限のないシステムアクセス、アカウント乗っ取り |
| マルウェア | ウイルス、ランサムウェア、トロイの木馬 |
| 情報漏洩 | 個人情報流出、機密文書の誤送信 |
| DoS/DDoS | サービス妨害攻撃 |
| 内部不正 | 従業員による意図的な情報持ち出し |
| フィッシング | 偽メール・サイトによる認証情報詐取 |

**対応フロー**:
1. 検知・報告（SOC/SIEM/ユーザー報告）
2. トリアージ（影響度・緊急度の判定）
3. 封じ込め（被害拡大の防止）
4. 根絶（原因の除去）
5. 復旧（通常運用への復帰）
6. 事後分析（教訓の抽出・再発防止策）

### 脅威分析

**脅威の種類**:
- **意図的脅威**: サイバー攻撃、内部不正、産業スパイ
- **偶発的脅威**: 人的ミス、設定誤り、情報の誤送信
- **環境的脅威**: 自然災害、停電、火災

**リスクアセスメント**:
```
リスク = 脅威 × 脆弱性 × 影響度（資産価値）
```

**リスク対応の選択肢**:
| 対応策 | 説明 |
|--------|------|
| **リスク回避** | リスクの原因となる活動を中止 |
| **リスク軽減** | 管理策を実施してリスクを許容レベルまで低減 |
| **リスク移転** | 保険や外部委託でリスクを第三者に移転 |
| **リスク受容** | リスクを認識した上で受け入れる（対策コスト>リスクの場合） |

### 情報セキュリティ方針

**階層構造**:
1. **情報セキュリティ基本方針**: 経営層が承認する最上位方針
2. **情報セキュリティ対策基準**: 具体的な管理策の基準
3. **情報セキュリティ実施手順**: 各部門・業務での具体的手順

### よくある誤解
- **誤**: セキュリティは専門チームだけの責任 → **正**: 全従業員がセキュリティに責任を持つ。教育・啓発が重要
- **誤**: ファイアウォールがあれば安全 → **正**: 多層防御（Defense in Depth）が必要。技術的対策だけでなく人的・組織的対策も
- **誤**: セキュリティインシデントは全て機密 → **正**: 法令（個人情報保護法等）や契約で報告義務がある場合は開示が必要

### 実務Tips
- セキュリティインシデント対応手順は、通常のインシデント管理フローに統合し、判定基準で分岐させる
- 定期的な脆弱性診断とペネトレーションテストの結果を変更管理のインプットにする
- 情報セキュリティ教育は年次で全従業員に実施し、実施記録を保持する
- **ゼロトラストアーキテクチャ**: 「境界の内側は安全」という前提を捨て、全てのアクセスを検証する設計が現代のベストプラクティス

## 10. 構成管理

### 定義と目的

サービス構成管理（ITIL4: Service Configuration Management）は、サービスとそのコンポーネントに関する正確で信頼性の高い情報を、必要な時に必要な場所で利用可能にすること。

### CI（Configuration Item: 構成アイテム）

**定義**: ITサービスの提供に必要な全てのコンポーネント。構成管理の対象として管理される。

**CIの種類**:
| カテゴリ | 例 |
|----------|-----|
| ハードウェアCI | サーバー、ネットワーク機器、PC、ストレージ |
| ソフトウェアCI | OS、ミドルウェア、アプリケーション、ライセンス |
| ドキュメントCI | SLA、運用手順書、設計書、マニュアル |
| サービスCI | ITサービス定義、サービスカタログエントリ |
| 人的CI | 組織、役割、スキル情報 |

**CIの属性**:
- CI ID（一意の識別子）
- 名称・説明
- カテゴリ・タイプ
- バージョン
- 所有者・管理責任者
- ステータス（運用中/保管中/廃棄済み等）
- 場所（物理的/論理的）
- 関連するSLA・OLA

### CMDB（Configuration Management Database: 構成管理データベース）

**定義**: CIとその関係情報を格納するデータベース。ITサービスマネジメントの全プロセスに正確な構成情報を提供する。

**CMDBに格納する情報**:
- CIの属性情報
- CI間の関係（依存関係、接続関係、親子関係）
- CIのステータスとライフサイクル履歴
- 変更履歴

**CMS（Configuration Management System）**: CMDBを含む、構成情報を管理するためのツール・データベース・プロセスの統合体。ITIL v3で導入された概念で、複数のCMDBやディスカバリツールを統合する。

### CI間の関係タイプ

| 関係タイプ | 説明 | 例 |
|-----------|------|-----|
| **依存関係** | AがBの機能に依存 | アプリケーションがDBサーバーに依存 |
| **接続関係** | AとBが物理的/論理的に接続 | サーバーとスイッチの接続 |
| **親子関係** | AがBの一部 | サーバー内のCPUモジュール |
| **利用関係** | AがBを利用 | サービスがミドルウェアを利用 |
| **インストール関係** | AがBにインストール | アプリケーションがサーバーにインストール |

### 構成ベースライン

**定義**: 特定の時点での構成の正式なスナップショット。変更管理の基準点として使用される。

**用途**:
- 変更前の状態を記録し、問題発生時の切り戻し基準にする
- 監査時の比較対象
- リリース管理での展開前状態の記録

### 構成管理の主な活動

1. **計画**: 構成管理の範囲・深さ・命名規則・ツール選定
2. **識別**: 管理対象CIの特定と属性の定義
3. **記録**: CMDBへのCI情報の登録
4. **管理（コントロール）**: CIの変更管理（変更管理プロセスとの連携）
5. **検証と監査**: CMDBの情報と実態の整合性確認

### 構成監査

**目的**: CMDBに記録されている情報と実際のIT環境が一致しているかを検証する。

**監査の種類**:
- **物理監査**: 実際のハードウェア・ソフトウェアとCMDBの照合
- **機能監査**: CIが期待通りの機能を果たしているかの確認
- **定期監査**: スケジュールに基づく定期的な検証
- **変更後監査**: 変更実施後のCMDB更新確認

### よくある誤解
- **誤**: CMDBは資産管理台帳と同じ → **正**: CMDBはCIの関係情報を含む。資産管理は財務的な視点（償却・ライセンス管理）、構成管理はサービスの視点
- **誤**: CMDBには全てのIT機器を登録すべき → **正**: 管理コストとの兼ね合い。サービスに影響するCIに範囲を限定する
- **誤**: CMDBは一度構築すれば完了 → **正**: 継続的な更新と定期的な監査が必要。陳腐化したCMDBは有害
- **誤**: 自動ディスカバリツールがあればCMDBは自動的に正確になる → **正**: ディスカバリは補助手段。関係情報や非IT CIは手動での管理が必要

### 実務Tips
- インシデント発生時にCMDBで影響範囲を即座に特定できることが、迅速な対応の鍵
- 変更管理でのインパクト分析にCMDBの関係情報を活用
- 構成監査の結果を品質指標として測定し、CMDB精度の継続的改善を図る
- **IaCとの連携**: Terraform/CloudFormationなどのIaCコードをCMDBの信頼できるソースとして扱い、ドリフト検知を自動化する

## 11. 財務管理

### 定義と目的

ITサービス財務管理は、ITサービスのコストを可視化し、適切な予算策定・課金を行うことで、ITサービスの経済的価値を最大化するプラクティス。

### 主な活動

#### 予算策定（Budgeting）
- IT支出の計画と承認を得るプロセス
- 運用予算（OpEx）と資本予算（CapEx）の区分
- 前年度実績・ビジネス計画に基づく予算配分

#### 会計（Accounting）
- ITサービスのコストを正確に把握・記録するプロセス
- コストモデルの設計（直接費/間接費、固定費/変動費）
- コストセンター別・サービス別のコスト配賦

#### 課金（Charging）
- ITサービスの利用者に対してコストを請求するプロセス
- 必須ではない（組織のポリシーによる）
- 内部振替価格の設定方法

### コスト分類

| 分類軸 | 区分 | 説明 |
|--------|------|------|
| **直接費/間接費** | 直接費 | 特定のサービスに直接帰属できるコスト（サーバー費等） |
| | 間接費 | 複数サービスに共通するコスト（空調費、管理人件費等） |
| **固定費/変動費** | 固定費 | サービス利用量に関係なく発生するコスト（ライセンス年額等） |
| | 変動費 | サービス利用量に比例して変動するコスト（クラウド従量課金等） |
| **資本的支出/運用支出** | CapEx | 資産取得費用（サーバー購入、ソフトウェアライセンス等） |
| | OpEx | 日常運用費用（電気代、保守料、クラウド利用料等） |

### TCO（Total Cost of Ownership: 総保有コスト）

IT資産のライフサイクル全体にわたる総コスト。

**TCOの構成要素**:
- **取得コスト**: 購入費、ライセンス費、導入費
- **運用コスト**: 保守料、電力費、人件費、クラウド利用料
- **管理コスト**: 監視・管理ツール、教育費
- **廃棄コスト**: データ消去、機器廃棄、移行コスト

### ROI（Return on Investment: 投資収益率）

**計算式**:
```
ROI = (投資による利益 - 投資コスト) / 投資コスト × 100%
```

**ITサービスにおけるROI算出の難しさ**:
- 定量的効果（コスト削減額、売上増加額）は比較的算出しやすい
- 定性的効果（生産性向上、顧客満足度向上）は金額換算が困難
- 間接効果（リスク低減、ブランド価値向上）の評価

### 課金方式

| 方式 | 説明 | メリット | デメリット |
|------|------|----------|-----------|
| **定額課金** | 固定料金 | シンプル、予測可能 | 利用量との公平性なし |
| **従量課金** | 利用量に応じた課金 | 公平、コスト意識向上 | 予測困難、計測コスト |
| **段階課金** | 利用量のティアに応じた課金 | バランスが良い | 設計が複雑 |
| **差別課金** | サービスレベルに応じた課金 | 需要管理効果あり | 説明・管理コスト |

### コスト最適化

- **合理化**: 重複サービスの統合、不要なサービスの廃止
- **標準化**: 構成の標準化によるサポートコスト削減
- **クラウド活用**: CapExからOpExへのシフト、従量課金による最適化
- **自動化**: 手作業の自動化による人件費削減
- **契約見直し**: ベンダー契約の定期的な見直しと再交渉

### よくある誤解
- **誤**: ITコストは「下げるほど良い」 → **正**: コスト最適化はビジネス価値との兼ね合い。過度なコスト削減はサービス品質低下を招く
- **誤**: クラウドは常にオンプレミスより安い → **正**: ワークロードによる。常時稼働の大規模システムはオンプレミスのほうがTCOが低い場合がある
- **誤**: 課金は全てのIT組織で必須 → **正**: 内部IT部門がコストセンターとして運営される場合、課金は行わないことも多い

### 実務Tips
- TCO分析はシステム更改の投資判断に不可欠。ベンダー比較時の基準としても活用
- ROI算出時は定性的効果も可能な限り定量化し、経営層への説明資料を充実させる
- 月次でサービス別コストを可視化し、コスト配賦の透明性を確保する
- **FinOpsの導入**: クラウド利用が増えた組織では、エンジニア・財務・ビジネスが協力してクラウドコストを管理するFinOps体制が効果的

## 12. 継続的改善

### 定義と目的

継続的改善（ITIL4: Continual Improvement）は、組織の全レベルで改善活動を推進し、サービスの有効性と効率性を向上させるプラクティス。ITIL4ではSVSの5つのコンポーネントの1つとして位置づけられ、全ての活動に浸透すべきものとされる。

### ITIL4 継続的改善モデル

```
1. ビジョンは何か？ (What is the vision?)
    ↓
2. 現在地はどこか？ (Where are we now?)
    ↓
3. どこを目指すか？ (Where do we want to be?)
    ↓
4. どうやって到達するか？ (How do we get there?)
    ↓
5. アクションの実行 (Take action)
    ↓
6. 到達したか？ (Did we get there?)
    ↓
7. 勢いを維持する (How do we keep the momentum going?)
    ↓ (ループ)
1. に戻る
```

**各ステップの詳細**:
1. **ビジョンは何か**: 組織の戦略・目標と改善活動を整合させる
2. **現在地はどこか**: ベースラインの確立。現状のアセスメント
3. **どこを目指すか**: 測定可能な目標の設定。CSF（重要成功要因）とKPIの定義
4. **どうやって到達するか**: 改善計画の策定。SIP（サービス改善計画）の作成
5. **アクションの実行**: 改善施策の実施
6. **到達したか**: 進捗と成果の測定。目標との比較
7. **勢いを維持する**: 改善の定着化。次の改善サイクルへの接続

### CSI（Continual Service Improvement）— ITIL v3の用語

ITIL v3ではサービスライフサイクルの5段階目として「継続的サービス改善（CSI）」が位置づけられていた。ITIL4では「継続的改善」として全体に統合された。

### KPI（Key Performance Indicator）とメトリクス

**KPIの階層構造**:
```
ビジョン・戦略
  ↓
CSF（Critical Success Factor: 重要成功要因）
  ↓
KPI（Key Performance Indicator: 重要業績評価指標）
  ↓
メトリクス（Metrics: 測定指標）
```

**ITサービスマネジメントの代表的なKPI**:

| プラクティス | KPI例 |
|-------------|------|
| インシデント管理 | MTTR、初回解決率、SLA達成率、再オープン率 |
| 問題管理 | 既知エラー登録数、再発インシデント数、根本原因特定率 |
| 変更管理 | 変更成功率、緊急変更比率、変更起因のインシデント数 |
| サービスデスク | 応答率、放棄呼率、顧客満足度 |
| 可用性管理 | 稼働率、計画外ダウンタイム、MTBF |
| キャパシティ管理 | SLA違反数（パフォーマンス）、リソース使用率 |

### メトリクスの種類

| 種類 | 説明 | 例 |
|------|------|-----|
| **技術メトリクス** | コンポーネントレベルの測定値 | CPU使用率、ネットワーク遅延 |
| **プロセスメトリクス** | プロセスの効率性・有効性 | 平均解決時間、変更成功率 |
| **サービスメトリクス** | エンドユーザー視点のサービス品質 | 可用性、顧客満足度 |

### ベンチマーキング

**定義**: 自組織のパフォーマンスを他組織やベストプラクティスと比較し、改善の機会を特定する手法。

**種類**:
- **内部ベンチマーキング**: 組織内の他部門・他サービスとの比較
- **競合ベンチマーキング**: 同業他社との比較
- **機能ベンチマーキング**: 業界をまたいだ特定機能の比較
- **一般ベンチマーキング**: ベストプラクティス標準との比較

### デミングサイクル（PDCA）との関係

ITIL4の継続的改善モデルはPDCAサイクルと整合する。

| PDCA | 継続的改善モデル |
|------|----------------|
| Plan | ステップ1〜4（ビジョン→計画策定） |
| Do | ステップ5（アクション実行） |
| Check | ステップ6（到達確認） |
| Act | ステップ7（勢いの維持・次サイクルへ） |

### 継続的改善登録簿（CIR: Continual Improvement Register）

全ての改善アイデア・提案を一元管理するリスト。優先順位付けと進捗管理を行う。

### よくある誤解
- **誤**: 継続的改善は問題があるときだけ行う → **正**: 問題がなくても改善の余地はある。成熟した組織はプロアクティブに改善を推進する
- **誤**: KPIは多いほど良い → **正**: 測定する価値のある少数の重要指標に絞る。測定疲れを避ける
- **誤**: ベンチマーキングは他社と同じにすること → **正**: 比較を通じて自組織の改善機会を特定すること。コンテキストの違いを考慮する

### 実務Tips
- 継続的改善登録簿を共有ツールで管理し、改善提案を組織的に収集・評価する仕組みを構築
- KPIダッシュボードを作成し、月次のサービスレビューで報告する
- 改善施策のROIを事前・事後で測定し、効果を定量的に証明する
- **改善の小さな一歩**: 大規模な改善プロジェクトだけでなく、日々の「カイゼン」を奨励する文化づくりが持続的な改善には不可欠

## 13. 実務活用パターン

### 運用設計での活用

#### サービスカタログの設計
- ビジネスサービスカタログとテクニカルサービスカタログの2層構造で設計
- 各サービスにSLA目標値・対応時間・エスカレーションパスを定義
- 利用者が自分で確認できるセルフサービスポータルとの連携

#### 運用プロセスの設計
- ITIL4のプラクティスをベースに、組織の成熟度に合わせたプロセスを設計
- 最低限「インシデント管理」「変更管理」「サービスレベル管理」から開始し、段階的に拡張
- ツール選定はプロセス設計の後に行う（ツール先行は失敗の原因）

#### 監視設計
- イベント管理（モニタリング）のアラート設計は、インシデント管理の分類・優先度と連動させる
- アラート疲れ（Alert Fatigue）を防ぐため、閾値の適切な設定とフィルタリング
- 情報（Information）/警告（Warning）/例外（Exception）の3段階で分類

### 障害対応での活用

#### インシデント対応の標準化
1. **検知→トリアージ→対応→報告→クローズ**のフロー確立
2. 優先度マトリクスの組織合意と自動判定ツールの導入
3. 重大インシデント用のブリッジコール（War Room）手順の整備
4. PIR（Post Incident Review）の定例化

#### 障害情報の活用サイクル
```
障害発生 → インシデント管理（復旧）
    → 問題管理（RCA → KEDB登録）
    → 変更管理（恒久対策の実施）
    → 構成管理（CMDB更新）
    → 継続的改善（プロセス改善）
```

#### エスカレーション設計
- 機能的エスカレーション: L1→L2→L3→ベンダーの技術レベル定義
- 階層的エスカレーション: 時間ベースの自動エスカレーション設計
- 重大インシデントの自動宣言基準: 「影響ユーザー○○人以上」「基幹業務の停止」等

### SLA策定での活用

#### SLA策定プロセス
1. **ビジネス要件の把握**: BIAの結果を活用し、VBFを特定
2. **現状の測定**: 現行の可用性・パフォーマンスのベースライン取得
3. **目標値の設定**: 達成可能かつビジネスニーズを満たすレベル
4. **OLA/UCの整備**: SLAを支える内部合意・外部契約の締結
5. **測定・報告体制の構築**: 自動測定ツールの導入、レポートテンプレートの作成
6. **レビューサイクルの確立**: 月次レビュー＋四半期サービスレビュー

#### SLA設計の実務的注意点
- **測定可能性**: 測定できない目標は設定しない
- **段階的引き上げ**: 初年度は達成可能な目標から始め、年次で引き上げる
- **例外条件の明確化**: 計画停止、不可抗力、利用者起因の障害の扱い
- **ペナルティ条項**: サービスクレジット等の具体的な措置を合意
- **レビュー条項**: 定期的な見直しの仕組みを契約に含める

### 稼働率計算の実践パターン

#### 基本の稼働率計算
サービス提供時間が月曜〜金曜 8:00-20:00（月間240時間）で、月間のダウンタイムが2.4時間の場合:
```
稼働率 = (240 - 2.4) / 240 × 100 = 99.0%
```
計画停止をSLAで除外する合意がある場合、計画停止時間は分母・分子ともに除く。

#### 直列・並列構成の稼働率
サーバーA（稼働率0.99）とサーバーB（稼働率0.99）が冗長構成、その後にネットワーク機器C（稼働率0.999）が直列接続:
```
サーバー部分 = 1 - (1-0.99) × (1-0.99) = 1 - 0.0001 = 0.9999
全体 = 0.9999 × 0.999 = 0.9989 (99.89%)
```

#### ダウンタイム計算
SLA目標稼働率99.9%、月間サービス提供時間720時間の場合:
```
許容ダウンタイム = 720 × (1 - 0.999) = 0.72時間 = 43.2分
```

### ITサービスマネジメントの成熟度モデル

組織の成熟度に応じた段階的導入が現実的。

| レベル | 状態 | 推奨アクション |
|--------|------|---------------|
| 1. 初期 | 場当たり的。個人の能力に依存 | インシデント管理の基本プロセス確立 |
| 2. 反復可能 | 基本プロセスあり。一部標準化 | 変更管理・SLM の導入。CMDB構築開始 |
| 3. 定義済み | プロセスが標準化・文書化 | 問題管理・キャパシティ管理の導入。KPI測定 |
| 4. 管理された | プロセスが測定・管理されている | CSI の本格運用。ベンチマーキング実施 |
| 5. 最適化 | 継続的改善が組織文化として定着 | 自動化推進。イノベーション志向 |

### DevOps/アジャイルとの共存

ITIL4はDevOps/アジャイルとの共存を前提に設計されている。

**実務での統合ポイント**:
- CI/CDパイプラインに変更管理の承認ゲートを組み込む
- Infrastructure as Code（IaC）で構成管理を自動化
- ChatOpsでインシデント対応のコラボレーションを促進
- SREのSLI/SLO概念をSLAと整合させる
- ポストモーテム文化と問題管理プロセスの融合
