# システム監査・内部統制 — 開発リファレンス

---

## 1. システム監査概論

### 1.1 システム監査の目的
システム監査とは、監査人が**一定の基準に基づいて**情報システムの利活用に係る検証・評価を行い、**ガバナンス・マネジメント・コントロール**について保証または改善のための助言を行う活動である。

目的は3つに大別される：
1. **保証型監査**：統制の有効性について合理的保証を提供する
2. **助言型監査**：改善のための助言・勧告を行う
3. **コンサルティング型**：組織の価値向上に資する提案を行う

### 1.2 監査人の独立性
独立性には2つの側面がある：

| 側面 | 定義 | 具体例 |
|------|------|--------|
| **精神上の独立性（Independence of Mind）** | 職業的専門家としての判断を危うくする影響を受けない精神状態 | 被監査部門の業務に直接関与しない |
| **外観上の独立性（Independence in Appearance）** | 合理的な第三者が独立性を疑わない外観 | 被監査部門の長と兼務しない |

独立性を脅かす5つの要因（ISACAフレームワーク準拠）：
- 自己利益の脅威
- 自己レビューの脅威
- 馴れ合いの脅威
- 脅迫の脅威
- 擁護の脅威

### 1.3 客観性
- 監査結論は証拠に基づくこと（エビデンスベース）
- 個人的な利害や偏見を排除すること
- 予断や先入観を持たずに監査を実施すること

### 1.4 正当な注意（Due Professional Care）
- 監査人は職業的専門家としての正当な注意を払って監査業務を遂行する
- 合理的な期待水準を満たす注意義務（完全性の保証ではない）
- 不正の発見を主目的としないが、不正の兆候に対する感度は求められる

### 1.5 システム監査基準（令和5年改訂）

#### 改訂の背景
- IIA基準・倫理要項の改訂への対応
- 技術進展（クラウド・AI・IoT等）への対応
- 2023年4月26日経済産業省公表

#### 構成（3区分12基準）

**監査の属性に係る基準（基準1〜5）**
- 基準1：監査の目的と権限
- 基準2：監査人の独立性・客観性・誠実性
- 基準3：監査人の能力（専門的能力の維持・向上）
- 基準4：監査人の倫理
- 基準5：品質管理

**監査の実施に係る基準（基準6〜10）**
- 基準6：監査計画の策定
- 基準7：監査の実施
- 基準8：監査証拠の入手と評価
- 基準9：監査調書の作成と保管
- 基準10：他の専門家の利用

**監査の報告に係る基準（基準11〜12）**
- 基準11：監査報告の作成
- 基準12：フォローアップ

#### 旧基準からの主な変更点
- 各基準は＜主旨＞＜解釈指針＞で構成
- 「監査人の倫理」が基準本文から独立化
- 実践部分を別冊ガイドライン化（日本システム監査人協会が管理）
- 経済産業省＝原則（What）、ガイドライン＝実施方法（How）の二層構造

### 1.6 IT統制の全体像

```
ITガバナンス
  └─ ITマネジメント
       ├─ IT全般統制（ITGC）
       │    ├─ プログラム変更管理
       │    ├─ アクセス管理
       │    ├─ コンピュータ運用管理
       │    └─ プログラム開発管理
       └─ IT業務処理統制（ITAC）
            ├─ 入力統制
            ├─ 処理統制
            └─ 出力統制
```

### 1.7 実務活用
- システム監査の基本概念は、開発プロジェクトのレビュー、セキュリティ評価、コンプライアンス確認のすべてに応用可能
- 「独立性」の概念はコードレビューやQAプロセスにも通じる（レビュアーが実装者と同一人物では客観性を担保できない）
- 「正当な注意」は技術者の品質保証活動の基本姿勢と同義
- 「エビデンスベース」はオブザーバビリティ設計の根幹（ログ・メトリクス・トレースが監査証跡に相当）

## 2. 監査計画

### 2.1 リスクアプローチ
監査資源は有限であるため、**リスクの高い領域に重点的に資源を配分**する考え方。

リスクアプローチの3段階：
1. **固有リスク（Inherent Risk）**：統制がない場合に存在するリスク
2. **統制リスク（Control Risk）**：内部統制が有効に機能しないリスク
3. **発見リスク（Detection Risk）**：監査手続で重要な虚偽表示を発見できないリスク

```
監査リスク = 固有リスク × 統制リスク × 発見リスク
```

監査人がコントロールできるのは**発見リスクのみ**。固有リスク・統制リスクが高い場合、発見リスクを低く抑える（＝より詳細な監査手続を実施する）必要がある。

### 2.2 監査テーマ選定
監査テーマ選定時の考慮要素：
- 経営戦略・事業目標との関連性
- 過去の監査結果とフォローアップ状況
- 経営環境の変化（法改正・技術変化・組織変更）
- ステークホルダーの関心事項
- リスクアセスメントの結果
- 監査資源（人員・時間・予算）の制約

### 2.3 年次計画（中長期監査計画）
- 3〜5年の中長期計画と年度計画を策定
- リスクアセスメントに基づく優先順位付け
- 監査対象のローテーション（重要度に応じた頻度設定）
- 経営者・監査役等との協議・承認

### 2.4 個別監査計画
個別の監査ごとに策定する詳細計画：
- 監査目的・範囲・対象
- 監査期間・スケジュール
- 監査チーム編成（専門スキルの確保）
- 監査手続の概要
- 必要な資源・ツール
- リスク評価と重点領域

### 2.5 監査手続書
- 個別監査計画をさらに具体化した実施手順書
- 各監査項目に対する：確認すべき事項、入手すべき証拠、実施手法
- 標準化されたチェックリストとカスタマイズ部分の両立

### 2.6 予備調査
本調査前に実施する情報収集活動：
- 被監査部門の業務概要・組織体制の把握
- 関連する規程・マニュアルの閲覧
- 前回監査の結果・フォローアップ状況の確認
- 関連するリスク情報の収集
- ウォークスルー（業務の流れの確認）

### 2.7 実務活用
- リスクアプローチの考え方はテスト戦略（リスクベーステスト）と同じ — 全網羅は非現実的なので、リスクの高い機能・変更箇所に重点配分する
- 監査計画策定の手法はプロジェクト計画にも応用可能
- 予備調査の手法はシステム移行時の現状分析に通じる
- スプリントレトロスペクティブで「監査テーマ選定」の考慮要素を適用すると、改善の優先順位付けが合理的になる

## 3. 監査実施

### 3.1 監査証拠
監査結論を裏付けるために入手する情報。

監査証拠の3要件：
1. **十分性（Sufficiency）**：量的に十分であること
2. **適切性（Appropriateness）**：質的に信頼でき関連性があること
3. **証明力（Persuasiveness）**：監査結論を支持する説得力があること

証拠の信頼性の序列（高→低）：
1. 監査人が直接入手した証拠（観察・再実施）
2. 外部の独立した第三者から入手した証拠
3. 被監査部門から入手した証拠
4. 口頭による証拠

### 3.2 監査技法

#### 質問（Inquiry）
- 被監査部門の担当者・管理者にインタビューする
- 最も一般的だが証拠力は最も低い
- 他の技法と組み合わせて裏付けを取る（裏付け確認: Corroboration）
- 質問技法：オープン質問→クローズド質問の順で深堀り

#### 観察（Observation）
- 業務の実施状況を直接観察する
- ある時点の状況を確認できるが、継続的な状況は保証しない
- 「ホーソン効果」（観察されることで行動が変わる）に注意

#### 閲覧・査閲（Inspection / Review）
- 文書・記録・ログを確認する
- 最も頻繁に使用される技法
- 原本の確認が望ましい（コピーは改ざんの可能性）

#### 再実施（Re-performance）
- 被監査部門が実施した手続を監査人が独立して再実行する
- 証拠力が高い
- 計算の再計算、統制の再テストなど

#### 分析的手続（Analytical Procedures）
- データ間の関係性を分析して異常を検出する
- 趨勢分析（トレンド分析）、比率分析、合理性テスト
- 重要性の判断の基礎となる

#### 突合（Reconciliation / Matching）
- 異なる情報源のデータを照合し整合性を確認する
- 帳簿突合：同一組織内の異なる記録の照合
- 証憑突合：記録と外部証憑の照合

#### 確認（Confirmation）
- 第三者に対して直接確認を依頼する
- 銀行確認、取引先確認など
- 積極的確認（回答必須）と消極的確認（不一致時のみ回答）

### 3.3 CAAT（コンピュータ利用監査技法）
Computer Assisted Audit Techniques の略。

主な手法：
- **汎用監査ソフトウェア（GAS）**：ACL、IDEAなどのデータ分析ツール
- **テストデータ法**：既知のテストデータを本番システムに投入し結果を検証
- **ITF法（統合テスト施設法）**：本番データ処理と並行してテストデータを処理
- **組込み監査モジュール法**：監査用のルーチンをアプリケーションに組み込む
- **スナップショット法**：特定のトランザクション処理時点のデータを記録
- **並行シミュレーション法**：監査人が作成したプログラムで同一データを処理し結果を比較

### 3.4 データ分析
- 全件テスト（母集団全体のデータ分析）が技術的に可能になりつつある
- ベンフォードの法則：数字の先頭桁の分布から不正を検出
- 例外報告分析：閾値を超えるトランザクションの抽出
- 重複検出：同一金額・同一取引先の重複支払チェック

### 3.5 サンプリング

#### 統計的サンプリング
- **属性サンプリング**：統制の逸脱率を推定（主に準拠性テスト）
- **変数サンプリング**：金額の誤謬を推定（主に実証性テスト）
- **MUS（Monetary Unit Sampling）**：金額を抽出単位とするサンプリング

#### 非統計的サンプリング
- **判断サンプリング**：監査人の判断で抽出
- **ブロックサンプリング**：特定期間のデータを一括抽出
- サンプルサイズの決定に統計的根拠がないため、結果の一般化に制約

### 3.6 実務活用
- データ分析技法はログ監視・異常検知に直接応用可能（ベンフォードの法則はトランザクション不正検知に有効）
- サンプリングの考え方はテストケース選定に活用できる
- CAATの手法はETLパイプラインの検証に応用可能
- 「並行シミュレーション法」はShadow Mode / A-Bテストと本質的に同じアプローチ

## 4. 監査報告

### 4.1 監査報告書の構成
標準的な監査報告書の構成要素：

1. **宛先**：経営者・監査役等
2. **監査の目的**
3. **監査の範囲と対象**
4. **監査の実施期間**
5. **監査の方法**
6. **制約事項**（あれば）
7. **監査意見（総合評価）**
8. **指摘事項**
9. **改善勧告**
10. **被監査部門の意見**（回答・反論の機会）
11. **監査責任者の署名・日付**

### 4.2 指摘事項の記述
指摘事項は以下の5要素で構成する（5W構造）：

| 要素 | 内容 | 例 |
|------|------|-----|
| 基準（Criteria） | あるべき姿 | 「変更管理規程では承認を必須としている」 |
| 事実（Condition） | 実態 | 「承認なしに本番移行された変更が3件あった」 |
| 原因（Cause） | 乖離の理由 | 「緊急変更の事後承認プロセスが未整備」 |
| 影響（Effect） | リスクの具体化 | 「未テストのコードが本番環境に反映されるリスク」 |
| 勧告（Recommendation） | 改善策 | 「緊急変更の事後承認手続を整備すること」 |

### 4.3 改善勧告
- 実現可能かつ具体的な勧告を提示する
- 優先度（高・中・低）を付す
- 実装期限の目安を提示する
- 被監査部門の実行責任者を明確にする

### 4.4 フォローアップ
- 改善勧告に対する対応状況を定期的に確認する
- 改善が実施されたことを証拠で確認する（実施したと言うだけでは不十分）
- 未対応の場合はリスクを再評価し、経営者にエスカレーションする
- フォローアップ結果を次回の監査計画に反映する

### 4.5 重要性の判断（Materiality）
- 量的重要性：金額的に重要かどうか
- 質的重要性：法令違反、セキュリティ侵害など金額に関わらず重要な事項
- 集計的重要性：個別には軽微でも集計すると重要になる事項
- 重要性の判断は監査計画段階で設定し、報告段階で再評価する

### 4.6 実務活用
- 指摘事項の5要素構造はバグレポートの書き方（期待結果・実結果・再現手順・影響・修正案）と類似
- 改善勧告の手法はプロジェクト振り返り（レトロスペクティブ）に応用可能
- フォローアップの考え方はポストモーテムのアクションアイテム追跡と同義

## 5. 内部統制

### 5.1 COSOフレームワーク（2013年改訂版）

#### 3つの目的
1. **業務の有効性と効率性**（Operations）
2. **報告の信頼性**（Reporting）— 2013年版で財務報告以外にも拡張
3. **法令等の遵守**（Compliance）

#### 5つの構成要素と17の原則

**統制環境（Control Environment）** — 5原則
1. 誠実性と倫理的価値観に対するコミットメントを表明する
2. 取締役会は経営者から独立し、内部統制の整備・運用を監視する
3. 適切な組織構造、報告経路、権限と責任を確立する
4. 有能な人材の確保・育成・維持に対するコミットメントを表明する
5. 内部統制の目的達成に関する個人の説明責任を強化する

**リスク評価（Risk Assessment）** — 4原則
6. 目的を十分に明確に特定し、リスクの識別・評価を可能にする
7. リスクを識別・分析し、リスク対応方法を決定する
8. 内部統制の目的達成を評価する際に不正リスクを考慮する
9. 内部統制に重要な影響を及ぼす変化を識別し、評価する

**統制活動（Control Activities）** — 3原則
10. リスクを許容水準まで低減する統制活動を選択・整備する
11. テクノロジーに関する全般的統制活動を選択・整備する（**IT全般統制の根拠**）
12. 方針と手続を通じて統制活動を展開する

**情報と伝達（Information and Communication）** — 3原則
13. 内部統制を支える関連性のある質の高い情報を入手・生成・利用する
14. 内部統制の目的達成に必要な情報を組織内部に伝達する
15. 内部統制に影響する事項について外部の関係者と情報を伝達する

**モニタリング活動（Monitoring Activities）** — 2原則
16. 日常的モニタリング及び/または独立的評価を実施する
17. 内部統制の不備を適時に評価し、是正措置を講じる

#### 87の着眼点
17原則の下に87の着眼点（Points of Focus）が設定されており、具体的な評価の手がかりを提供する。

### 5.2 IT全般統制（ITGC: IT General Controls）
情報システム全体に共通する統制：

| 統制領域 | 内容 | 監査のポイント |
|----------|------|---------------|
| プログラム変更管理 | 本番環境への変更の承認・テスト・移行 | 変更申請→承認→テスト→移行の証跡 |
| アクセス管理 | ユーザーID管理・権限設定・特権ID管理 | 定期的なアクセス権レビュー |
| コンピュータ運用管理 | ジョブスケジュール管理・バックアップ | 運用手順書の整備・障害対応記録 |
| プログラム開発管理 | 開発標準・テスト・品質管理 | 開発工程の承認プロセス |

### 5.3 IT業務処理統制（ITAC: IT Application Controls）
個別のアプリケーションに組み込まれた統制：

- **入力統制**：入力データの正確性・完全性を確保（バリデーション、チェックデジット、照合）
- **処理統制**：処理の正確性・完全性を確保（コントロールトータル、論理チェック、リミットチェック）
- **出力統制**：出力の正確性・配布先の適切性を確保（出力リスト照合、配布先管理）

### 5.4 J-SOX（内部統制報告制度）
- 金融商品取引法に基づく内部統制報告制度
- COSOフレームワークをベースに日本独自の要素を追加
- 日本版の追加要素：**ITへの対応**（COSOの5要素＋1）
- 経営者による内部統制の評価→監査法人による監査→内部統制報告書の提出
- 2024年改訂：サステナビリティ報告、不正リスク対応を強化

J-SOXの6つの基本的要素：
1. 統制環境
2. リスクの評価と対応
3. 統制活動
4. 情報と伝達
5. モニタリング
6. **ITへの対応**（日本独自）

### 5.5 財務報告の信頼性
- 財務報告に関する内部統制が有効に整備・運用されているかを評価
- 重要な業務プロセスにおけるIT依存度の評価
- ITGCがITACの有効性を支える関係にある
- ITGCに不備がある場合、ITACの有効性に疑義が生じる

### 5.6 3線モデル（IIA 2020年改訂）

旧「3つのディフェンスライン」モデルを進化させたガバナンスモデル：

| ライン | 役割 | 責任 |
|--------|------|------|
| 第1線 | 事業部門・現場 | リスクのオーナーとして統制を実施 |
| 第2線 | リスク管理・コンプライアンス部門 | 専門的助言・監視・支援 |
| 第3線 | 内部監査部門 | 独立した保証を提供 |

3線モデルの特徴（旧モデルとの違い）：
- 「ディフェンス」の表現を廃止し、価値創造の側面を強調
- 各ラインの連携と協力を重視
- 経営者と取締役会の役割を明確化
- 第1線と第2線の統合的アプローチを容認

### 5.7 実務活用
- COSOの5要素はシステム設計時の統制設計に直接活用可能
- 3線モデルは組織のQA体制設計（開発→QA→内部監査）に応用できる
- ITGCの観点はDevOps/CI/CDパイプラインの統制設計に不可欠
- ITGC不備がITAC全体に波及する関係は、インフラ障害がアプリ全体に影響するのと同構造

## 6. 情報セキュリティ監査

### 6.1 ISMSとの関係
- ISMS（ISO/IEC 27001）：情報セキュリティマネジメントシステムの国際規格
- 情報セキュリティ監査はISMSの有効性を検証する手段の一つ
- 経済産業省「情報セキュリティ管理基準」はISO/IEC 27001/27002をベースに策定

ISMSの主要プロセス：
1. 情報セキュリティ方針の策定
2. リスクアセスメント（資産の特定→脅威と脆弱性の評価→リスク評価）
3. リスク対応（リスク低減・回避・移転・受容）
4. 管理策の選択と実施
5. 有効性の測定と評価
6. マネジメントレビュー
7. 継続的改善

### 6.2 技術的監査
- ネットワーク構成の確認（ファイアウォール設定、セグメント分離）
- 暗号化の適切性（暗号アルゴリズム、鍵管理）
- 認証方式の評価（多要素認証、シングルサインオン）
- パッチ管理の状況確認
- セキュリティ監視体制（SOC/CSIRT）の有効性

### 6.3 ペネトレーションテスト
- 実際の攻撃手法を用いてシステムの脆弱性を検証する
- ブラックボックス型：攻撃者と同じ情報のみで実施
- ホワイトボックス型：内部情報を開示した上で実施
- グレーボックス型：限定的な内部情報を提供して実施
- 監査人は結果の妥当性を評価し、対応策の適切性を検証する

### 6.4 脆弱性診断
- 自動化ツールによるスキャン（ネットワーク脆弱性スキャン、Webアプリ診断）
- CVSSスコアに基づくリスク評価
- 定期的な診断の実施状況と結果への対応を監査
- 診断範囲の網羅性（内部/外部、OS/ミドルウェア/アプリケーション）

### 6.5 ログ監査
- ログの取得範囲・保存期間の適切性
- ログの改ざん防止策（書込み専用メディア、ハッシュ値保管）
- ログの定期的なレビュー体制
- 異常検知の仕組み（SIEM）の有効性
- 特権IDのログ監視

### 6.6 実務活用
- 情報セキュリティ監査の観点はセキュリティレビュー・脅威モデリングに直接活用可能
- ログ監査の知識は運用監視設計に不可欠（SIEM/オブザーバビリティ基盤の設計根拠）
- ペネトレーションテストの手法理解はセキュリティテスト計画に必要
- 脆弱性診断（DAST/SAST/SCA）をCI/CDに組み込むことで継続的な技術的監査を自動化できる

## 7. 個人情報保護監査

### 7.1 個人情報保護法（令和2年改正・令和4年施行）
- 個人情報取扱事業者の義務
- 利用目的の特定と制限
- 安全管理措置（組織的・人的・物理的・技術的）
- 第三者提供の制限
- 本人の関与（開示・訂正・利用停止請求）
- 漏えい等報告の義務化（個人情報保護委員会への報告）
- 令和6年改正：こどもの個人情報に関する規律の整備、AI活用時の対応

監査のポイント：
- 個人データの取得・利用・保管・廃棄の各段階における安全管理措置
- 委託先の監督状況
- 漏えい時の対応手順の整備と訓練
- 個人情報の棚卸し（台帳管理）の実施状況

### 7.2 GDPR（EU一般データ保護規則）
- EU域内の個人データを取り扱う場合に適用
- データ主体の権利（アクセス権、削除権「忘れられる権利」、ポータビリティ権）
- DPO（データ保護オフィサー）の設置義務
- DPIA（データ保護影響評価）の実施義務
- 違反時の制裁金：最大で全世界年間売上高の4%または2,000万ユーロ

監査のポイント：
- 域外移転に関する適切な保護措置（十分性認定、標準契約条項SCC）
- 同意取得プロセスの適法性
- データ主体の権利行使への対応手順

### 7.3 PIA（プライバシー影響評価 / Privacy Impact Assessment）
- 新規システム・サービスの導入前にプライバシーリスクを評価する手法
- GDPRではDPIA（Data Protection Impact Assessment）として義務化
- 日本でも個人情報保護委員会がPIAの実施を推奨

PIAの実施手順：
1. 対象の特定（個人情報を取り扱うシステム・サービス）
2. データフローの把握
3. プライバシーリスクの識別
4. リスク対応策の検討
5. 残存リスクの評価
6. 結果の記録と公表

### 7.4 プライバシーバイデザイン（Privacy by Design）
7つの基本原則：
1. 事後的ではなく事前的、救済的ではなく予防的
2. デフォルトとしてのプライバシー保護
3. 設計に組み込まれたプライバシー
4. 全機能性（ゼロサムではなくポジティブサム）
5. エンドツーエンドのセキュリティ（ライフサイクル全体の保護）
6. 可視性と透明性
7. 利用者のプライバシーの尊重

### 7.5 匿名加工情報
- 特定の個人を識別できないように加工した個人情報
- 本人同意なしで第三者提供が可能
- 加工方法や加工に関する情報の安全管理措置が必要
- 再識別行為の禁止

### 7.6 仮名加工情報（令和2年改正で新設）
- 他の情報と照合しなければ特定の個人を識別できないように加工した情報
- 匿名加工情報より加工度が低く、組織内部での分析に活用
- 第三者提供は原則禁止
- 利用目的の変更が可能（匿名加工情報との大きな違い）

### 7.7 実務活用
- PIA/DPIAの手法は新規サービス開発時のプライバシーリスク評価に直接活用
- プライバシーバイデザインの原則はシステム設計の初期段階から適用（データ最小化・目的限定・暗号化をアーキテクチャに組み込む）
- 匿名加工情報/仮名加工情報の知識はデータ利活用の設計に不可欠
- ML/AIパイプラインにおける学習データのプライバシー保護（差分プライバシー、連合学習等）もこの延長線上にある

## 8. システム開発監査

### 8.1 開発プロセス監査
開発プロセス全体を通じた監査の観点：

| フェーズ | 監査のポイント |
|----------|---------------|
| 要件定義 | 利用者要件の網羅性、実現可能性の評価、承認プロセス |
| 設計 | 設計基準の遵守、セキュリティ要件の反映、レビュー実施 |
| 実装 | コーディング規約の遵守、セキュアコーディング、コードレビュー |
| テスト | テスト計画の妥当性、テスト網羅性、テスト結果の評価 |
| 移行 | 移行計画の策定、リハーサル実施、切り戻し手順の整備 |

### 8.2 品質管理監査
- 品質基準の設定と遵守状況
- 品質メトリクス（バグ密度、テストカバレッジ等）の管理
- レビュープロセスの実施状況
- 品質に関する意思決定の記録

### 8.3 テスト監査
- テスト計画書の妥当性（テスト範囲、テスト条件、合格基準）
- テストケースの網羅性（同値分割、境界値分析、デシジョンテーブル）
- テスト環境の適切性（本番環境との差異の把握）
- テスト結果の記録と欠陥管理
- 回帰テスト（リグレッションテスト）の実施状況
- 受入テストにおけるユーザーの関与

### 8.4 移行監査
- 移行計画（データ移行・システム切替え・並行運用）の策定状況
- データ移行の正確性・完全性の検証方法
- 切り戻し計画（ロールバック手順）の策定と訓練
- 移行リハーサルの実施と結果の評価
- 移行判定基準の設定と判定プロセス

### 8.5 外部委託監査（開発委託）
- 委託先選定プロセスの適切性（評価基準の明確化）
- 契約内容の妥当性（SLA、知的財産権、秘密保持、損害賠償）
- 進捗管理・品質管理の方法
- 成果物の検収プロセス
- 委託先の情報セキュリティ対策
- 再委託の管理

### 8.6 実務活用
- 開発プロセス監査の観点はプロジェクトマネジメントのチェックポイントに活用
- テスト監査の知識はQA戦略の策定に直接応用可能
- 外部委託監査の観点はベンダー管理に不可欠
- CI/CDパイプラインにおけるゲート（lint→test→SAST→承認→deploy）は開発プロセス監査の自動化に相当する

## 9. 運用・保守監査

### 9.1 変更管理
- 変更要求の受付→影響分析→承認→実装→テスト→本番移行→事後レビュー
- 緊急変更の管理（事後承認プロセスの整備）
- 変更履歴の記録と追跡可能性
- 変更管理委員会（CAB: Change Advisory Board）の機能

監査のポイント：
- 承認なしに本番環境へ変更が反映されていないか
- テスト完了前の本番移行がないか
- 緊急変更の事後承認が適切に行われているか

### 9.2 障害管理
- インシデント管理：発生→検知→対応→復旧→記録
- 問題管理：根本原因分析（RCA）→恒久対策→再発防止
- エスカレーション手順の整備
- SLA遵守状況の管理

監査のポイント：
- 障害の検知から復旧までの時間が適切か
- 根本原因分析が実施されているか
- 同種の障害が繰り返し発生していないか
- 障害記録の正確性・完全性

### 9.3 バックアップ
- バックアップ方針（対象、頻度、世代管理、保管場所）
- バックアップの取得状況の監視
- リストアテスト（復元テスト）の定期的な実施
- 遠隔地保管（オフサイト保管）の実施

監査のポイント：
- バックアップが計画通りに実行されているか
- リストアテストが定期的に実施されているか
- 目標復旧時点（RPO）を満たすバックアップ頻度か

### 9.4 BCP/DR監査

#### BCP（事業継続計画）
- BIA（事業影響度分析）に基づくRTO/RPOの設定
- 復旧戦略の策定（代替サイト、クラウド活用等）
- 計画の定期的な見直しと訓練
- サプライチェーンを含めた事業継続性の確保

#### DR（災害復旧）
- DR計画の策定（復旧手順、体制、連絡網）
- DRサイトの整備と同期状況
- DR訓練の実施と結果の評価
- RPO/RTOの達成可能性の検証

監査のポイント：
- BIA（事業影響度分析）が定期的に更新されているか
- DR訓練が計画通り実施され、課題が改善されているか
- RTO/RPOの設定が事業要件と整合しているか

### 9.5 SLA監査
- SLA項目の妥当性（可用性、応答時間、障害復旧時間等）
- SLAの測定方法と測定結果の報告
- SLA未達時のペナルティと改善措置
- SLAの定期的な見直し

### 9.6 実務活用
- 変更管理の監査観点はCI/CDパイプラインの統制設計に直接適用（PRレビュー必須・マージ承認・デプロイ承認の自動ゲート化）
- BCP/DRの知識はクラウドアーキテクチャ設計（マルチリージョン・マルチAZ等）に不可欠
- SLA監査の考え方はSRE（Site Reliability Engineering）のSLI/SLO/SLA設計に通じる
- 障害管理のRCAはポストモーテム文化と一致 — 「blame-free」の姿勢で根本原因を追求する
- Chaos Engineeringは「DR訓練の自動化・継続化」と位置づけられる

## 10. クラウド・外部委託監査

### 10.1 CSA STAR（Cloud Security Alliance — Security, Trust, Assurance and Risk）
クラウドサービスプロバイダのセキュリティ評価プログラム：

| レベル | 名称 | 内容 |
|--------|------|------|
| Level 1 | Self-Assessment | 自己評価（CCMに基づく、無料） |
| Level 2 | Third-Party Assessment | 第三者による評価・認証 |
| Level 3 | Continuous Monitoring | 継続的な監視に基づく認定 |

CSA CCM（Cloud Controls Matrix）：ISO 27001、PCI DSS、HIPAA、SOC 2等の業界基準にマッピングされた統制フレームワーク。

### 10.2 SOC報告書（System and Organization Controls）
AICPAが定義した、受託業務に関する内部統制の保証報告書：

| 種類 | 対象 | 用途 |
|------|------|------|
| SOC 1 | 財務報告に関する内部統制 | J-SOX対応、委託先管理 |
| SOC 2 | セキュリティ・可用性・処理のインテグリティ・機密保持・プライバシー | ITサービス全般の評価 |
| SOC 3 | SOC 2と同じ（簡易版） | 一般公開用 |

Type I と Type II：
- **Type I**：特定時点における統制の設計の適切性を評価
- **Type II**：一定期間における統制の運用の有効性を評価（**Type IIの方が証拠力が高い**）

SOC報告書の利用上の注意点：
- 報告対象期間と利用時点のギャップ（ブリッジレター）
- 「相補的なユーザー組織の統制（CUC）」の確認
- サブサービス組織の統制範囲

### 10.3 委託先管理
- 委託先選定基準の策定と評価プロセス
- 契約事項：SLA、セキュリティ要件、監査権、データの取扱い、解約条件
- 委託先の定期的な評価（質問票、実地監査、SOC報告書）
- 再委託先の管理
- 委託終了時のデータ返却・消去

### 10.4 サプライチェーンリスク
- 委託先・再委託先を経由したデータ漏えいリスク
- 委託先のセキュリティインシデントの自組織への波及
- 経済産業省「SCS評価制度」（2026年度末制度開始予定）：企業のセキュリティ対策を「星」の数で可視化
- サプライチェーン攻撃（ソフトウェアサプライチェーン攻撃を含む）への対策
- SBOM（Software Bill of Materials）の活用

### 10.5 SaaS監査
SaaS固有の監査観点：
- データの所在地（データレジデンシー）
- マルチテナント環境におけるデータ分離
- サービス提供者のアクセス権限
- データポータビリティ（ベンダーロックインの回避）
- サービス停止・終了時のデータ回収
- 共有責任モデル（Shared Responsibility Model）の理解

### 10.6 実務活用
- SOC報告書の読み方はクラウドサービス選定時に直接活用（AWS/Azure/GCPはいずれもSOC 2 Type IIレポートを公開）
- 共有責任モデルの理解はクラウドアーキテクチャ設計の前提 — IaaS/PaaS/SaaSで責任分界点が異なる
- SBOMの知識はOSSライセンス管理・脆弱性管理に活用（`npm audit`/`pip audit`等のツールと連動）
- SaaS監査の観点はサードパーティAPI依存のリスク評価に応用できる

## 11. 新技術監査

### 11.1 AI監査の観点

AIシステム固有の監査ポイント：

| 観点 | 内容 |
|------|------|
| データ品質 | 学習データの正確性・完全性・偏り（バイアス）の評価 |
| モデルの透明性 | 判断過程の説明可能性（XAI: Explainable AI） |
| 公平性 | アルゴリズムによる差別・偏りがないか |
| セキュリティ | 敵対的攻撃（Adversarial Attack）への耐性 |
| プライバシー | 個人情報の取扱い、推論による個人特定リスク |
| 継続的監視 | モデルドリフト（精度劣化）の検知と対応 |
| ガバナンス | AI利用方針、責任体制、倫理委員会の設置 |

IIA「2024年版グローバル内部監査基準」：デジタルテクノロジーの継続的検討と効果的活用を明文化。

### 11.2 IoT監査の観点
- デバイスの認証・認可（大量デバイスの一元管理）
- ファームウェア更新の管理（OTAアップデートのセキュリティ）
- データの暗号化（通信経路・保存データ）
- 物理的セキュリティ（デバイスの改ざん防止）
- エッジコンピューティングにおけるデータ処理の統制
- デバイスのライフサイクル管理（廃棄時のデータ消去）

### 11.3 ブロックチェーン監査の観点
- コンセンサスアルゴリズムの適切性（PoW/PoS等）
- スマートコントラクトの脆弱性
- 秘密鍵の管理
- ガバナンス体制（パブリック/プライベート/コンソーシアム）
- 規制対応（暗号資産関連法令）
- データの不可変性と「忘れられる権利」との両立

### 11.4 RPA監査の観点
- RPAロボットのアクセス権限管理（特権的な権限の付与状況）
- 処理結果の正確性検証（例外処理の適切性）
- 変更管理（RPAシナリオの変更承認プロセス）
- 稼働状況の監視（実行ログの取得・レビュー）
- 業務プロセス変更時のRPA連動更新
- 職務分離（RPAが複数業務を統合して実行する場合の統制）

### 11.5 デジタルフォレンジック
- 電子的証拠の収集・保全・分析・報告の手法
- 証拠の完全性維持（ハッシュ値による原本性の証明）
- Chain of Custody（証拠の管理連鎖）の確保
- 法的に有効な証拠として認められるための手続
- ライブフォレンジック（稼働中のシステムからの証拠収集）
- ネットワークフォレンジック（通信データの分析）

フォレンジック調査のプロセス：
1. 保全（Preservation）：証拠の改変防止
2. 収集（Collection）：証拠データの取得
3. 検査（Examination）：データの特定・フィルタリング
4. 分析（Analysis）：証拠の評価・解釈
5. 報告（Reporting）：調査結果の文書化

### 11.6 実務活用
- AI監査の観点はMLOpsパイプラインの品質管理に直接活用（データバリデーション→モデル評価→デプロイ承認→ドリフト監視の各ゲート）
- RPA監査の知識は自動化ツール（GitHub Actions/cron/Workers等）導入時のリスク評価に必要
- デジタルフォレンジックの手法はインシデント対応に不可欠（ログの保全・証拠の Chain of Custody）
- AI監査における「モデルドリフト検知」は本番監視のアラート設計と同じ発想

## 12. 法令・基準

### 12.1 システム監査基準（経済産業省）
- 令和5年（2023年）4月改訂
- システム監査人の行為規範・監査手続の規則
- 3区分12基準（属性5、実施5、報告2）＋解釈指針
- 詳細は「1. システム監査概論」参照

### 12.2 システム管理基準（経済産業省）
- システム監査基準と対をなす管理基準
- 情報システムの企画・開発・運用・保守に関する管理上の基準
- 2023年改訂で構成を見直し
- ガバナンス・マネジメント・コントロールの3階層で整理

### 12.3 情報セキュリティ管理基準
- ISO/IEC 27001/27002をベースに経済産業省が策定
- 情報セキュリティマネジメントの実践規範
- 情報セキュリティ監査の判断基準として活用
- 管理策の選択と実施に関する指針

### 12.4 ITガバナンス
ITガバナンスの定義（ISO/IEC 38500）：
- 取締役会がITの利用を指示・評価・監視するシステム
- IT投資の意思決定、リスク管理、価値の実現

ITガバナンスの6原則（ISO/IEC 38500）：
1. 責任（Responsibility）
2. 戦略（Strategy）
3. 取得（Acquisition）
4. パフォーマンス（Performance）
5. 適合（Conformance）
6. 人間行動（Human Behaviour）

### 12.5 COBIT 2019（ISACA）

#### 概要
- ISACAが2018年11月に公表
- COBIT 5（2012年）以来6年ぶりの全面刷新
- ITガバナンス → I&Tの全社ガバナンス（EGIT）に拡大
- 監査基準から出発し、事業体I&Tガバナンスフレームワークへ進化

#### ガバナンスとマネジメントの2領域

**ガバナンス領域**（5プロセス）：評価（Evaluate）、指示（Direct）、監視（Monitor）
- EDM01：ガバナンスフレームワークの設定と維持の確保
- EDM02：便益の提供の確保
- EDM03：リスク最適化の確保
- EDM04：資源の最適化の確保
- EDM05：ステークホルダーの透明性の確保

**マネジメント領域**（4ドメイン・35プロセス）：
- APO（Align, Plan and Organize）：整合・計画・組織化
- BAI（Build, Acquire and Implement）：構築・調達・導入
- DSS（Deliver, Service and Support）：提供・サービス・サポート
- MEA（Monitor, Evaluate and Assess）：監視・評価・査定

#### プロセス能力レベル（CMMIベース）
- レベル0：不完全
- レベル1：実施
- レベル2：管理
- レベル3：確立
- レベル4：予測可能
- レベル5：最適化

### 12.6 ISACA基準（国際情報システム監査基準）
ISACAが定めるIS監査・保証の基準体系：
- **基準（Standards）**：監査人が遵守すべき必須要件
- **ガイドライン（Guidelines）**：基準の適用に関する指針
- **ツールと技法（Tools and Techniques）**：実務的な手法

主要な基準：
- ITAF（IT Assurance Framework）：IT保証の統合フレームワーク
- IS監査基準：一般基準、実施基準、報告基準の3区分

ISACA認定資格：
- CISA（Certified Information Systems Auditor）：情報システム監査の国際資格
- CRISC（Certified in Risk and Information Systems Control）：リスク管理
- CISM（Certified Information Security Manager）：セキュリティ管理
- CGEIT（Certified in the Governance of Enterprise IT）：ITガバナンス

### 12.7 実務活用
- COBITのプロセス能力レベルは組織の成熟度評価に活用可能（CMMIレベルをチーム・プロジェクトの自己評価に使う）
- ISACAの基準体系はIT監査部門の品質管理に参照
- ISO/IEC 38500の6原則はIT投資の意思決定フレームワークに活用
- 監査基準と管理基準の区別（監査人の行為規範 vs 被監査側の管理基準）を理解することで、レビュアーと実装者の責任分界が明確になる

## 13. 実務活用パターン

### 13.1 システム監査観点でのレビュー

開発レビューにシステム監査の視点を取り入れることで品質を向上させる。

#### 設計レビューへの活用
- **リスク評価の観点**：この設計にはどのようなリスクがあるか
- **統制設計の観点**：リスクに対する統制は十分か
- **証跡確保の観点**：問題発生時に追跡可能な記録が設計されているか
- **権限分離の観点**：職務分離の原則が守られているか

#### コードレビューへの活用
- **入力バリデーション**：外部入力に対する検証は十分か
- **エラーハンドリング**：例外処理が適切に設計されているか
- **ログ出力**：監査証跡となるログが適切に出力されているか
- **認証・認可**：アクセス制御が正しく実装されているか

### 13.2 内部統制設計への活用

#### ITGCの設計
- **変更管理プロセス**：承認→テスト→移行の自動化（CI/CDパイプラインの統制）
- **アクセス管理**：最小権限の原則、定期的なアクセス権レビュー、特権ID管理
- **運用管理**：監視・アラート・障害対応プロセスの自動化と記録

#### ITACの設計
- **入力統制**：APIバリデーション、フォームバリデーション、データ型チェック
- **処理統制**：トランザクション管理、整合性チェック、ビジネスルールエンジン
- **出力統制**：レポートの承認ワークフロー、データマスキング、配布先管理

### 13.3 コンプライアンス確認への活用

#### チェックリストの作成
監査基準に基づく自己点検チェックリストの活用：

| 確認項目 | 根拠基準 | 確認方法 |
|----------|---------|---------|
| 個人情報の安全管理措置 | 個人情報保護法 | プライバシーポリシー査閲、技術的対策確認 |
| ログの保存・管理 | 情報セキュリティ管理基準 | ログ管理ポリシー査閲、保管状況確認 |
| 変更管理の承認プロセス | システム管理基準 | 変更管理記録査閲、承認フロー確認 |
| バックアップの実施 | システム管理基準 | バックアップポリシー査閲、復元テスト記録確認 |
| 委託先管理 | 個人情報保護法/セキュリティ基準 | 委託契約書査閲、定期評価記録確認 |

#### 監査対応の準備
- 監査に備えた証跡の日常的な蓄積
- ポリシー・規程類の定期的な見直し
- 自己点検（CSA: Control Self-Assessment）の実施
- 監査指摘事項への計画的な対応と改善記録の管理

### 13.4 DevOps/SREへの監査観点適用

| 監査概念 | DevOps/SRE対応 | 実装例 |
|----------|---------------|--------|
| 変更管理 | Git フロー + PR承認 + CI/CDパイプライン | GitHub branch protection + required reviews + status checks |
| 障害管理 | インシデント管理ツール + ポストモーテム | PagerDuty/OpsGenie → Jira/Linear → blame-free RCA文書 |
| アクセス管理 | IAM + RBAC + 最小権限原則 | Cloudflare Zero Trust / AWS IAM / GCP Workload Identity |
| 監査証跡 | 構造化ログ + 分散トレーシング | OpenTelemetry + Datadog/Grafana + 改ざん防止ストレージ |
| BCP/DR | マルチリージョン + フェイルオーバー + Chaos Engineering | Multi-AZ RDS + Route53 failover + Chaos Monkey |
| SLA管理 | SLI/SLO定義 + エラーバジェット | Prometheus/Grafana SLO dashboard + burn-rate alerts |
| 継続的監視 | オブザーバビリティ（メトリクス・ログ・トレース） | 3 pillars + anomaly detection + runbook automation |
| テスト統制 | テスト自動化 + カバレッジゲート | vitest/Jest + Codecov + PR merge block on < threshold |
| 秘密管理 | シークレットマネージャー + ローテーション | Vault/AWS Secrets Manager + 自動ローテーション + 監査ログ |

#### DevOps固有の監査課題と対応
- **Infrastructure as Code（IaC）の変更管理**: Terraform/Pulumi の plan → review → apply フローをPR承認に統合
- **GitOps環境の職務分離**: デプロイ権限とコード承認権限を分離（CODEOWNERS + deployment approvers）
- **Feature Flagの統制**: フラグの有効化/無効化に承認プロセスと監査ログを設ける
- **カナリアデプロイの監査**: 段階的ロールアウトの判定基準（エラーレート・レイテンシ閾値）を文書化

### 13.5 SRE実践への監査フレームワーク適用

| SRE概念 | 対応する監査概念 | 活用方法 |
|---------|-----------------|---------|
| SLI/SLO/SLA | SLA監査 | SLO違反をリスク事象として監査計画に反映 |
| エラーバジェット | リスク許容度 | エラーバジェット消費率をリスク指標として監視 |
| Toil削減 | 統制の効率性 | 手動統制の自動化率を統制活動の有効性指標に |
| Blameless Postmortem | 根本原因分析 | ポストモーテムを監査証跡として活用 |
| Chaos Engineering | DR訓練 | 実験結果をBCP/DR監査のエビデンスに |
| On-call | インシデント対応体制 | ローテーション・エスカレーションパスの監査 |

### 13.6 プロジェクト管理への応用
- リスクベーステストの考え方はリスクアプローチと同義
- プロジェクト計画書のレビューに監査計画の視点を活用
- 成果物の検収に監査証拠の評価基準を適用
- ベンダー管理に外部委託監査の観点を活用

### 13.7 リスク・コントロール・監査証跡の3点セット

監査の基本構造「リスク→コントロール→監査証跡」は、開発・運用のあらゆる場面に適用できる。

| リスク | コントロール | 監査証跡（エビデンス） |
|--------|-------------|----------------------|
| 不正アクセス | アクセス権限管理 | アクセス権限一覧、認証ログ |
| データ改ざん | 変更ログの記録 | Git commit履歴、変更承認記録 |
| 業務中断 | BCP策定 | BCP文書、訓練記録 |
| 不正な変更 | 承認ワークフロー | PRレビュー・マージ記録 |
| テスト不足 | テスト管理手順 | テスト計画書、CI実行結果、カバレッジレポート |
| 設定ドリフト | IaC + GitOps | Terraform state diff、デプロイログ |
| 依存脆弱性 | SCA/SBOM管理 | `npm audit`結果、SBOM出力、対応チケット |
