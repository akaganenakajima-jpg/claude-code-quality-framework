# IT戦略・経営判断 — 開発リファレンス

---

## 1. 経営戦略

### SWOT分析
- **定義**：組織の内部環境（Strengths/Weaknesses）と外部環境（Opportunities/Threats）を整理するフレームワーク
- **詳細**：
  - S（強み）：自社の競争優位性、技術力、ブランド力、人材
  - W（弱み）：自社の課題、リソース不足、技術的負債
  - O（機会）：市場成長、規制緩和、技術革新、顧客ニーズ変化
  - T（脅威）：競合参入、法規制強化、代替品、景気後退
- **クロスSWOT**：
  - SO戦略（強み×機会）：積極攻勢。強みを活かして機会を獲得
  - ST戦略（強み×脅威）：差別化。強みで脅威を回避
  - WO戦略（弱み×機会）：弱点強化。弱みを克服して機会を活用
  - WT戦略（弱み×脅威）：防衛・撤退。弱みと脅威の最小化
- **よくある誤解**：SWOTは「分析」ではなく「整理」のツール。分析はクロスSWOTで行う
- **実務活用**：IT戦略策定の初期フェーズで経営層と現状認識を共有する際に使用。プロダクト方針転換やピボット判断にも有効

### PEST分析
- **定義**：マクロ環境を Political（政治）・Economic（経済）・Social（社会）・Technological（技術）の4要因で分析
- **詳細**：
  - P：法規制、税制、政策方針、国際関係
  - E：景気動向、為替、金利、物価、雇用情勢
  - S：人口動態、ライフスタイル変化、価値観変容、高齢化
  - T：技術革新、特許動向、インフラ整備、デジタル化進展
- **拡張**：PESTLE（+Legal+Environmental）もある
- **よくある誤解**：PESTは外部環境のみの分析。内部環境はSWOTのS/Wで分析する
- **実務活用**：中期経営計画策定時の外部環境スキャン。DX推進の背景説明にも有効

### ポーターの5つの競争要因（Five Forces）
- **定義**：業界の競争構造と収益性を5つの力で分析するフレームワーク（マイケル・ポーター提唱）
- **5つの力**：
  1. **既存競合の競争**：同業他社との価格・品質競争
  2. **新規参入の脅威**：参入障壁の高低（規模の経済、ブランド、規制等）
  3. **代替品の脅威**：異業種からの代替ソリューション
  4. **買い手の交渉力**：顧客の価格交渉力（スイッチングコスト低→交渉力高）
  5. **売り手の交渉力**：サプライヤーの価格交渉力（供給者集中→交渉力高）
- **3つの基本戦略**（5 Forcesに基づく）：
  - コストリーダーシップ：低コストで競争優位
  - 差別化：独自価値で競争優位
  - 集中：特定セグメントに資源集中（コスト集中/差別化集中）
- **よくある誤解**：5 Forcesは「業界」の分析であり、個社の分析ではない
- **実務活用**：IT投資判断時の業界構造理解。新規事業参入検討時の市場魅力度評価

### バリューチェーン分析
- **定義**：企業活動を価値創造プロセスとして分解し、各活動の付加価値と競争優位の源泉を特定するフレームワーク
- **主活動**：購買物流→製造→出荷物流→マーケティング・販売→サービス
- **支援活動**：全般管理（経営管理）、人事・労務管理、技術開発、調達活動
- **マージン**：顧客が支払う総価値から全活動のコストを差し引いた利益
- **よくある誤解**：バリューチェーンは製造業だけのツールではない。サービス業にも適用可能（活動の定義を変える）
- **実務活用**：どの活動にIT投資すべきかの優先順位付け。BPR対象プロセスの特定

### コアコンピタンス
- **定義**：顧客に特定の利益をもたらす、競合他社に模倣されにくい、複数の市場にアクセスできる自社固有の中核的能力（C.K.プラハラード＆G.ハメル）
- **3条件**：
  1. 顧客価値の提供
  2. 競合による模倣困難性
  3. 複数市場への展開可能性
- **ケイパビリティとの違い**：
  - コアコンピタンス：技術・スキルの組み合わせ（例：ホンダのエンジン技術）
  - ケイパビリティ：組織全体のビジネスプロセス能力（例：ウォルマートのSCM）
- **VRIO分析**（バーニー）：Value（価値）・Rareness（希少性）・Imitability（模倣困難性）・Organization（組織）の4基準で内部資源を評価
- **よくある誤解**：コアコンピタンスは技術だけではない。組織文化やプロセスも含む
- **実務活用**：IT投資の「選択と集中」判断。自社で内製すべき領域とアウトソースすべき領域の線引き

### ブルーオーシャン戦略
- **定義**：競争のない未開拓市場（ブルーオーシャン）を創造し、競争を無意味にする戦略（W.チャン・キム＆R.モボルニュ）
- **レッドオーシャン**：既存市場での血みどろの競争。価格競争→利益率低下
- **4アクションフレームワーク**：
  1. **取り除く**（Eliminate）：業界で当たり前とされる要素を除去
  2. **減らす**（Reduce）：業界標準以下に引き下げ
  3. **増やす**（Raise）：業界標準以上に引き上げ
  4. **付け加える**（Create）：業界にない新要素を創造
- **戦略キャンバス**：横軸に競争要因、縦軸にレベルをとり、価値曲線で差別化を可視化
- **よくある誤解**：「新市場＝ニッチ市場」ではない。ブルーオーシャンは需要を創造する大きな市場
- **実務活用**：DXによる新規事業創出時の戦略立案。既存市場での差別化が困難な場合に特に有効

### 両利きの経営（Ambidexterity）
- **定義**：既存事業の「知の深化」（Exploitation）と新規事業の「知の探索」（Exploration）を同時に追求する経営手法（チャールズ・オライリー＆マイケル・タッシュマン）
- **知の深化**：既存事業の効率化・改善・最適化。確実な収益源
- **知の探索**：新規領域への探索・実験・イノベーション。不確実だが成長源
- **サクセストラップ**：知の深化に偏り、探索を怠ると長期的に競争力喪失
- **組織設計**：深化と探索で別組織を設置し、上位で統合（構造的分離＋経営統合）
- **よくある誤解**：「両方やる」だけでは不十分。深化と探索の組織的分離と経営層による戦略的統合がポイント
- **実務活用**：DX推進組織の設計（既存IT部門とDX推進室の関係性）。既存プロダクトの改善と新規プロダクト探索を両立させる組織構造の設計指針

### PPM（プロダクト・ポートフォリオ・マネジメント）
- **定義**：BCGマトリクスを用いて事業を「市場成長率×相対市場シェア」の2軸で4象限に分類
- **4象限**：
  - **花形（Star）**：高成長・高シェア。投資を続けて将来の金のなる木に
  - **金のなる木（Cash Cow）**：低成長・高シェア。安定収益源。投資は最小限
  - **問題児（Question Mark）**：高成長・低シェア。投資判断が必要
  - **負け犬（Dog）**：低成長・低シェア。撤退・売却を検討
- **資金フロー**：金のなる木→問題児（or花形）に資金を配分
- **よくある誤解**：PPMは事業の「今」のスナップショット。動的な変化（問題児→花形→金のなる木→負け犬）を時系列で見る必要がある
- **実務活用**：IT投資ポートフォリオの優先順位付け。プロダクトラインの継続・撤退判断

### アンゾフの成長マトリクス
- **定義**：「製品×市場」の2軸で企業の成長戦略を4つに分類（イゴール・アンゾフ）
- **4象限**：
  1. **市場浸透**（既存製品×既存市場）：シェア拡大、顧客単価向上
  2. **市場開拓**（既存製品×新市場）：新地域、新セグメントへの展開
  3. **製品開発**（新製品×既存市場）：既存顧客への新製品投入
  4. **多角化**（新製品×新市場）：最もリスクが高い。関連多角化/非関連多角化
- **多角化の種類**：
  - 水平型多角化：同業種の新製品
  - 垂直型多角化：サプライチェーンの川上/川下進出
  - 集中型多角化：既存技術を活かした新市場
  - コングロマリット型多角化：全く異なる分野への進出
- **よくある誤解**：「多角化＝成長」ではない。多角化はリスクが最も高く、コアコンピタンスとのシナジーが重要
- **実務活用**：DXによる新規事業展開の方向性を経営層に提案する際のフレームワーク

## 2. IT戦略

### EA（エンタープライズアーキテクチャ）
- **定義**：組織全体のビジネスとITの構造を俯瞰的に設計・管理するためのフレームワーク
- **4つのアーキテクチャ体系**：
  1. **ビジネスアーキテクチャ（BA）**：業務プロセス、組織構造、ビジネスルール
  2. **データアーキテクチャ（DA）**：データモデル、データフロー、マスターデータ管理
  3. **アプリケーションアーキテクチャ（AA）**：アプリケーション構成、連携、ライフサイクル
  4. **テクノロジーアーキテクチャ（TA）**：インフラ、ネットワーク、セキュリティ基盤
- **ザックマンフレームワーク**：Who/What/When/Where/Why/Howの6列×6行で企業情報を体系化
- **TOGAF**：The Open Groupが策定したEAのフレームワーク。ADM（Architecture Development Method）が中核
- **As-Is / To-Be**：現状（As-Is）と目標（To-Be）のギャップ分析がEAの基本手法
- **よくある誤解**：EAは「図を描く活動」ではない。経営戦略とIT実装を繋ぐ「翻訳」の仕組み
- **実務活用**：レガシーシステム刷新計画の策定。マイクロサービス移行やクラウドリフトの全体設計にEAの4層を活用

### IT投資管理
- **定義**：IT投資の企画・評価・実行・事後評価のPDCAサイクルを通じて投資効果を最大化する活動
- **投資分類**（MIT CISR分類）：
  - **戦略的投資**：競争優位のための新規投資（新規事業、DX推進）
  - **情報提供投資**：意思決定支援（BI、データ分析基盤）
  - **トランザクション投資**：業務効率化（ERP、自動化）
  - **インフラ投資**：基盤整備（ネットワーク、クラウド移行）
- **IT投資評価手法**：NPV、IRR、ROI、回収期間法、TCO（→セクション7で詳述）
- **よくある誤解**：IT投資の効果は「コスト削減」だけではない。戦略的価値（売上拡大、顧客体験向上）を含めた総合評価が必要
- **実務活用**：経営会議でのIT投資提案。クラウド移行やSaaS導入の投資判断根拠の整理

### ITポートフォリオ
- **定義**：IT投資案件をリスク・リターンの観点で分類し、最適な投資配分を決定する手法
- **4象限モデル**：
  - 高リスク・高リターン：戦略的IT投資（DX、AI活用）
  - 低リスク・高リターン：業務改善投資（BPR、RPA）
  - 低リスク・低リターン：インフラ維持投資（保守、更新）
  - 高リスク・低リターン：回避すべき投資
- **PPMとの関係**：事業のPPMとITポートフォリオを連動させることで経営戦略とIT戦略の整合性を確保
- **よくある誤解**：「戦略的投資に集中すべき」ではない。インフラ維持を怠ると技術的負債が蓄積
- **実務活用**：年度IT予算策定時の投資配分根拠

### ITガバナンス
- **定義**：取締役会等がステークホルダーのニーズに基づき、ITシステムの利活用のあるべき姿を示すIT戦略と方針の策定・実現のための活動（経済産業省「システム管理基準」）
- **ITマネジメントとの違い**：
  - ITガバナンス：方向付け・評価・監視（取締役会・経営層の責任）
  - ITマネジメント：計画・構築・実行・監視（IT部門の責任）
- **5つの重点領域**（ITGI）：
  1. 戦略的整合性（Strategic Alignment）
  2. 価値の提供（Value Delivery）
  3. リスク管理（Risk Management）
  4. 資源管理（Resource Management）
  5. パフォーマンス測定（Performance Measurement）
- **よくある誤解**：ITガバナンス＝IT統制ではない。ガバナンスは「方向付け」、統制は「管理」
- **実務活用**：ITガバナンス体制の構築提案。組織設計を語る際の枠組み

### COBIT（Control Objectives for Information and Related Technologies）
- **定義**：ISACAが策定したI&T（情報と技術）ガバナンスのためのフレームワーク
- **COBIT 2019の特徴**：
  - 「ITガバナンス」から「I&Tガバナンス」へ表現変更（事業体全体に関わるものとして位置づけ）
  - **ガバナンスの目的**：EDM（Evaluate, Direct, Monitor）
  - **マネジメントの目的**：APO（Align, Plan, Organize）、BAI（Build, Acquire, Implement）、DSS（Deliver, Service, Support）、MEA（Monitor, Evaluate, Assess）
  - デザインファクター（設計要因）と設計ツールキットの導入
- **4つのドメイン（COBIT 4.1）**：
  1. 計画と組織（Plan and Organise）
  2. 調達と導入（Acquire and Implement）
  3. サービス提供とサポート（Deliver and Support）
  4. モニタリングと評価（Monitor and Evaluate）
- **成熟度モデル**：0（不在）→1（初期/場当たり的）→2（反復可能）→3（定義済み）→4（管理）→5（最適化）
- **よくある誤解**：COBITは「監査ツール」だけではない。ガバナンス設計・改善のフレームワークとしても使用
- **実務活用**：ITガバナンス成熟度の自己評価と改善計画策定

### デジタルガバナンス・コード
- **定義**：経済産業省が策定した、企業のDX推進に関する経営者の行動指針
- **4つの柱**：
  1. **ビジョン・ビジネスモデル**：経営ビジョンとビジネスモデルの変革方向性
  2. **戦略**：ビジョン実現のためのDX戦略の策定・公表
  3. **組織づくり・人材・企業文化**：DX推進体制の整備
  4. **ITシステム・デジタル技術活用環境の整備**：レガシー刷新と新技術活用
- **DX認定制度**：デジタルガバナンス・コードの基本事項に対応する企業をIPAが認定
- **DX銘柄**：東証上場企業の中からDX推進優良企業を経産省・東証が選定
- **デジタルガバナンス・コード3.0**（2024年改訂）：成果指標の設定・開示を経営の責務と位置づけ
- **よくある誤解**：「認定＝DX完了」ではない。認定は最低基準の充足であり、継続的な推進が必要
- **実務活用**：自社のDX推進状況の客観的評価。外部ステークホルダーへの説明材料

## 3. DX戦略

### デジタルトランスフォーメーション（DX）
- **定義**：企業がビジネス環境の激しい変化に対応し、データとデジタル技術を活用して、顧客や社会のニーズを基に、製品やサービス、ビジネスモデルを変革するとともに、業務そのものや組織、プロセス、企業文化・風土を変革し、競争上の優位性を確立すること（経産省DX推進ガイドライン）
- **DXの3段階**：
  1. **デジタイゼーション**：アナログ→デジタルへの変換（紙→PDF等）
  2. **デジタライゼーション**：業務プロセスのデジタル化（ワークフロー電子化等）
  3. **デジタルトランスフォーメーション**：ビジネスモデル自体の変革
- **DXレポート**（経産省2018年）：「2025年の崖」を提唱。レガシーシステム刷新の緊急性を警告
- **DXレポート2**（2020年）：コロナ禍を契機にDXの加速を提言。「素早く」「変化に対応」がキーワード
- **DXレポート2.1**（2021年）：デジタル産業への変革の方向性を提示
- **よくある誤解**：「IT化＝DX」ではない。DXはビジネスモデル変革を伴う。単なるIT導入はデジタイゼーション
- **実務活用**：経営層へのDX推進提案。自社サービスの変革方向性を定義する際の基本概念

### DX推進指標
- **定義**：経営者や社内関係者がDX推進に向けた現状や課題の認識を共有し、アクションにつなげるための自己診断ツール（経産省策定、IPA運用）
- **2つの指標群**：
  1. **DX推進のための経営のあり方、仕組みに関する指標**（経営指標）
  2. **DX を実現する上で基盤となる IT システムの構築に関する指標**（IT指標）
- **成熟度レベル**（0〜5）：
  - レベル0：未着手
  - レベル1：一部での散発的実施
  - レベル2：一部での戦略的実施
  - レベル3：全社戦略に基づく部門横断的推進
  - レベル4：全社戦略に基づく持続的実施
  - レベル5：グローバル市場におけるデジタル企業
- **現状**（2024年分析）：1,349件提出、レベル0〜2未満に偏在、レベル4以上は全体の1%
- **よくある誤解**：指標が高ければDX成功ではない。指標は「気づきの機会」であり、アクションに繋げることが目的
- **実務活用**：自社DX推進状況の定量評価と改善ロードマップ作成

### 2025年の崖
- **定義**：経産省DXレポート（2018年）で提唱された概念。既存レガシーシステムが複雑化・ブラックボックス化し、DX推進の阻害要因となり、2025年以降に最大年間12兆円の経済損失が生じるリスク
- **背景**：
  - 基幹系システムの21年以上稼働率が6割超
  - IT人材の引退・不足（2025年に約43万人不足）
  - SAP ERP（S/4 HANA移行期限）
  - メインフレーム縮退
- **DX動向2025**（IPA調査）：
  - DX取組企業は約8割（米国並み）
  - しかし「成果が出ている」のは日本6割弱（米独8割超）
  - 7割の企業が「外向きのDX成果」を出せないまま2025年を迎えた
  - 日本は「コスト削減」中心、米独は「売上・利益増加」中心
- **よくある誤解**：「2025年の崖＝2025年に破綻」ではない。構造的リスクの蓄積を警告する概念
- **実務活用**：レガシー刷新の投資提案。技術的負債の可視化と経営層への説明

### データドリブン経営
- **定義**：勘・経験・度胸（KKD）ではなく、データ分析に基づいて経営判断を行う手法
- **構成要素**：
  - **データ基盤**：データレイク、データウェアハウス、ETL/ELTパイプライン
  - **分析基盤**：BI、統計分析、機械学習、AI
  - **データガバナンス**：データ品質管理、メタデータ管理、データカタログ
  - **組織・文化**：データリテラシー向上、データ活用文化の醸成
- **CDO（Chief Data Officer）**：データ戦略を統括する経営幹部
- **DAMA-DMBOK**：データマネジメントの知識体系（Data Management Body of Knowledge）
- **よくある誤解**：「データを集めればデータドリブン」ではない。データの品質・整合性・活用基盤が不可欠
- **実務活用**：BI/AI基盤構築の企画提案。KPIダッシュボード設計やA/Bテスト基盤の構築判断

### プラットフォーム戦略
- **定義**：複数のユーザーグループ（生産者・消費者等）を結びつけ、相互作用を促進する基盤を提供するビジネス戦略
- **ネットワーク効果**：
  - 直接的ネットワーク効果：同じ側のユーザーが増えるほど価値向上（SNS等）
  - 間接的ネットワーク効果：一方のユーザーが増えると他方の価値向上（アプリストア等）
- **プラットフォーム設計の原則**：
  1. **キラー機能**：ユーザーを惹きつける中核価値
  2. **マッチング機能**：需要と供給の効率的なマッチング
  3. **信頼・安全の仕組み**：レビュー、評価、本人確認
  4. **開発者エコシステム**：API公開、SDK提供
- **WTA（Winner Takes All）**：プラットフォーム市場では独占的勝者が生まれやすい
- **よくある誤解**：「プラットフォーム＝IT基盤」ではない。ビジネスモデルとしてのプラットフォームは仲介型
- **実務活用**：デジタルエコシステム構築の戦略立案。API経済やマーケットプレイス型サービスの設計

## 4. ビジネスモデル

### ビジネスモデルキャンバス（BMC）
- **定義**：アレクサンダー・オスターワルダーが提唱した、ビジネスモデルを9つの構成要素で可視化するフレームワーク
- **9つの構成要素**：
  1. **顧客セグメント（CS）**：誰に価値を届けるか
  2. **価値提案（VP）**：顧客にどんな価値を提供するか
  3. **チャネル（CH）**：どうやって届けるか
  4. **顧客との関係（CR）**：どんな関係を構築するか
  5. **収益の流れ（RS）**：どうやって収益を得るか
  6. **リソース（KR）**：価値提供に必要な経営資源
  7. **主要活動（KA）**：価値提供に必要な活動
  8. **パートナー（KP）**：外部との連携
  9. **コスト構造（CS）**：かかるコスト
- **バリュープロポジションキャンバス（VPC）**：BMCの「顧客セグメント」と「価値提案」を深掘りするツール
- **よくある誤解**：BMCは「一度描いて終わり」ではない。仮説検証を繰り返して進化させる
- **実務活用**：新規事業企画の整理。DX推進時のビジネスモデル変革の設計。スタートアップのピッチ準備にも有効

### サブスクリプションモデル
- **定義**：製品やサービスを一定期間の利用権として定額課金で提供するビジネスモデル
- **特徴**：
  - 継続的な収益（MRR/ARR）
  - 顧客との長期的関係構築
  - データ蓄積による改善サイクル
  - 初期コストのハードル低下
- **重要指標**：
  - **MRR**（Monthly Recurring Revenue）：月次経常収益
  - **ARR**（Annual Recurring Revenue）：年次経常収益
  - **チャーンレート**（Churn Rate）：解約率
  - **LTV**（Lifetime Value）：顧客生涯価値
  - **CAC**（Customer Acquisition Cost）：顧客獲得コスト
  - **LTV/CAC比**：3以上が健全な目安
- **よくある誤解**：「月額課金＝サブスクリプション」ではない。継続的な価値提供と顧客体験の改善が本質
- **実務活用**：従来の売切りモデルからの転換提案。SaaS導入の投資評価

### シェアリングエコノミー
- **定義**：個人や企業が保有する遊休資産（モノ・場所・スキル・時間等）をインターネット上のプラットフォームを介して他者と共有・交換するビジネスモデル
- **5つの領域**：
  1. モノ（フリマ、レンタル）
  2. 空間（民泊、コワーキング）
  3. 移動（ライドシェア、カーシェア）
  4. スキル（クラウドソーシング、家事代行）
  5. お金（クラウドファンディング、ソーシャルレンディング）
- **信頼の仕組み**：相互評価、本人確認、保険・保証、エスクロー決済
- **よくある誤解**：「CtoC＝シェアリング」ではない。BtoCやBtoBのシェアリングも存在する
- **実務活用**：自社遊休資産の活用による新規収益源の創出提案

### フリーミアムモデル
- **定義**：基本機能を無料（Free）で提供し、高度な機能・サービスを有料（Premium）で提供するビジネスモデル
- **収益構造**：
  - 無料ユーザー→有料への転換率（コンバージョン率）：一般的に2〜5%
  - 無料ユーザーのコスト＜有料ユーザーの収益で成立
  - ネットワーク効果で無料ユーザーが有料ユーザーの価値を高める場合もある
- **設計原則**：
  - 無料版で十分な価値を提供（ユーザー獲得）
  - 有料版で明確な追加価値（アップセル動機）
  - 無料→有料の壁（ペイウォール）の位置が重要
- **よくある誤解**：「無料で撒けば有料に転換する」わけではない。無料・有料の境界設計が成否を分ける
- **実務活用**：SaaS事業の収益モデル設計。デジタルサービスの価格戦略

### マルチサイドプラットフォーム
- **定義**：2つ以上の異なるユーザーグループを結びつけ、相互作用を促進するプラットフォーム
- **典型例**：
  - クレジットカード（加盟店×カード会員）
  - OS（アプリ開発者×エンドユーザー）
  - EC モール（出店者×購入者）
  - 広告メディア（広告主×視聴者）
- **鶏と卵問題**：どちらのグループを先に獲得するか。一般的に価格感応度が高い側を無料/低価格にする
- **価格設計**：一方のグループに補助金を出し（マネーサイド→サブサイドサイド）、もう一方から収益を得る非対称価格設定
- **よくある誤解**：「仲介業＝マルチサイドプラットフォーム」ではない。間接的ネットワーク効果の存在が条件
- **実務活用**：プラットフォーム型新規事業の設計。マーケットプレイス立ち上げ時の初期戦略

## 5. システム戦略

### 情報システム戦略
- **定義**：経営戦略を実現するために、情報システムの方向性・投資方針・推進体制を定める戦略
- **策定プロセス**：
  1. 経営戦略の理解と整合性確認
  2. 現状（As-Is）分析：業務プロセス、既存システム、データ
  3. 目標（To-Be）策定：あるべき姿の設計
  4. ギャップ分析：As-IsとTo-Beの差異を特定
  5. 移行計画策定：優先順位付け、ロードマップ作成
  6. 推進体制・ガバナンスの整備
- **よくある誤解**：情報システム戦略＝IT部門の計画ではない。経営戦略と一体の「事業戦略の一部」
- **実務活用**：経営課題→IT戦略→施策→評価の一貫した計画策定

### 全体最適化計画
- **定義**：組織全体のIT投資・システム構築を部分最適ではなく全体最適の観点で計画する活動
- **アプローチ**：
  - **EA（エンタープライズアーキテクチャ）**による全体俯瞰
  - **BPR（ビジネスプロセスリエンジニアリング）**との連動
  - **MDM（マスターデータマネジメント）**によるデータ統合
  - **ERP**による業務プロセスの標準化・統合
- **阻害要因**：
  - サイロ化された組織構造
  - 部門ごとの個別最適投資の慣行
  - レガシーシステムのスパゲッティ連携
- **よくある誤解**：「全体最適化＝大規模一括刷新」ではない。段階的アプローチ（マイクロサービス化等）も有効
- **実務活用**：基幹システム刷新計画の策定。「2025年の崖」対策の提案

### SOA（サービス指向アーキテクチャ）
- **定義**：ビジネス機能をサービスとして定義し、それらを組み合わせてシステムを構築するアーキテクチャスタイル
- **特徴**：
  - サービスの疎結合（Loose Coupling）
  - サービスの再利用性（Reusability）
  - サービスの自律性（Autonomy）
  - サービスのインターフェース標準化
- **ESB（Enterprise Service Bus）**：サービス間の通信を仲介するミドルウェア
- **BPEL**：ビジネスプロセスの実行を記述するXMLベースの言語
- **よくある誤解**：「SOA＝Webサービス」ではない。SOAは設計思想であり、実装技術は限定しない
- **実務活用**：レガシーシステムの段階的モダナイゼーション戦略

### マイクロサービスアーキテクチャ
- **定義**：アプリケーションを独立したサービスの集合として構築するアーキテクチャスタイル。SOAの発展形
- **特徴**：
  - 各サービスが独立してデプロイ・スケール可能
  - サービスごとに最適な技術スタックを選択可能（ポリグロット）
  - API（REST/gRPC等）による通信
  - サービスごとの独立したデータストア
- **モノリスとの比較**：
  - モノリス：開発・デプロイが単純だが変更影響が大きい
  - マイクロサービス：変更が局所化されるが運用の複雑性が増す
- **関連技術**：コンテナ（Docker）、オーケストレーション（Kubernetes）、サービスメッシュ（Istio）、API Gateway
- **よくある誤解**：「マイクロサービス＝正義」ではない。組織規模・チーム体制・運用能力に応じた選択が必要（コンウェイの法則）
- **実務活用**：レガシー刷新の段階的アプローチ（ストラングラーパターン）の提案

### クラウド戦略
- **定義**：クラウドサービスの活用方針を定め、オンプレミスとの最適な組み合わせを設計する戦略
- **サービスモデル**：
  - **IaaS**：インフラ提供（仮想マシン、ストレージ、ネットワーク）
  - **PaaS**：プラットフォーム提供（開発環境、ミドルウェア）
  - **SaaS**：アプリケーション提供（完成品サービス）
- **デプロイメントモデル**：パブリック、プライベート、ハイブリッド、マルチクラウド
- **クラウドバイデフォルト原則**：政府情報システムの整備にあたり、クラウドを第一候補として検討する原則（政府CIO補佐官等連絡会議）
- **移行パターン（6R）**：
  1. Rehost：リフト＆シフト
  2. Replatform：一部最適化して移行
  3. Refactor：クラウドネイティブに再構築
  4. Repurchase：SaaSに置き換え
  5. Retain：オンプレミス維持
  6. Retire：廃止
- **よくある誤解**：「クラウド＝コスト削減」とは限らない。従量課金の管理やベンダーロックインのリスクも考慮が必要
- **実務活用**：クラウド移行計画の策定。6Rモデルによる既存システムの移行判断

### API経済
- **定義**：APIを公開・連携することで、組織間のデータ・機能の相互利用を促進し、新たなビジネス価値を創出するエコシステム
- **APIの公開レベル**：
  - プライベートAPI：社内利用
  - パートナーAPI：特定のパートナー企業に公開
  - パブリックAPI：一般公開
- **API管理**：API Gateway、認証（OAuth2.0/OpenID Connect）、レート制限、バージョニング
- **API収益化モデル**：従量課金、定額課金、フリーミアム、収益シェア
- **オープンAPI**：銀行法改正（2017年）により金融機関にAPI公開の努力義務。フィンテック連携
- **よくある誤解**：「API公開＝セキュリティリスク」ではない。適切なAPI管理とセキュリティ設計で安全に公開可能
- **実務活用**：デジタルエコシステム構築戦略。異業種連携・オープンイノベーションの基盤

## 6. 要件定義・調達

### RFI（Request for Information）
- **定義**：情報提供依頼書。ベンダーに対して技術動向、製品情報、実績等の情報提供を依頼する文書
- **目的**：RFP作成前の予備調査。市場調査、技術動向把握、概算費用感の確認
- **記載項目**：
  - 背景と目的
  - 情報提供を求める範囲（技術、製品、事例、費用感）
  - 回答期限と提出方法
  - 秘密保持に関する条件
- **よくある誤解**：RFIは「見積依頼」ではない。情報収集が目的であり、ベンダー選定を伴わない
- **実務活用**：大規模IT投資の初期段階で市場を把握。技術選定の事前調査

### RFP（Request for Proposal）
- **定義**：提案依頼書。システム要件を記載し、ベンダーに対して具体的な提案と見積を依頼する文書
- **記載項目**：
  - プロジェクトの背景・目的・スコープ
  - 機能要件・非機能要件
  - 前提条件・制約条件
  - 評価基準と選定プロセス
  - スケジュール・予算目安
  - 契約条件（SLA、瑕疵担保・契約不適合責任等）
- **RFPの品質がプロジェクト成否を左右する**：要件の曖昧さ→提案のバラつき→選定ミス→プロジェクト失敗
- **よくある誤解**：「RFP＝仕様書」ではない。RFPは「何をしたいか」を伝え、「どうやるか」はベンダーに委ねる
- **実務活用**：ベンダー選定プロセスの設計。外部パートナーとの協業開始時の基本文書

### ベンダー選定
- **評価手法**：
  - **総合評価方式**：技術点と価格点の加重平均で評価
  - **プロポーザル方式**：提案内容の質で評価（価格は参考）
  - **入札方式**：価格のみで評価（公共調達で多用）
- **評価基準**：
  - 技術力（提案の妥当性、技術的実現性）
  - 実績・信頼性（類似プロジェクト経験、財務健全性）
  - 体制（PM能力、要員の質と量）
  - 価格（初期費用＋運用費＝TCO）
  - リスク管理（移行計画、BCP対応）
- **よくある誤解**：「最安値＝最適」ではない。TCO（総保有コスト）と戦略的適合性で総合判断
- **実務活用**：調達プロセスの設計と実行。評価基準の妥当性検証

### IT投資対効果
- **定量的効果**：コスト削減額、売上増加額、生産性向上率、処理時間短縮
- **定性的効果**：顧客満足度向上、従業員満足度向上、ブランド価値向上、リスク低減
- **KGI/KPI設定**：
  - KGI（Key Goal Indicator）：最終的な達成目標（売上○億円等）
  - KPI（Key Performance Indicator）：目標達成のための中間指標（顧客獲得数等）
  - CSF（Critical Success Factor）：目標達成に不可欠な成功要因
- **投資効果の測定タイミング**：事前評価→中間評価→事後評価のPDCA
- **よくある誤解**：「IT投資効果＝コスト削減のみ」ではない。戦略的価値を含めた多面的評価が必要
- **実務活用**：BSCとの組み合わせによる多面的な投資効果測定

### Make-or-Buy意思決定
- **定義**：システムを自社開発（Make）するか、外部調達（Buy）するかの意思決定
- **判断基準**：
  - **Make（内製）が適する場合**：コアコンピタンスに関わる、差別化要因、外部に適切な製品がない
  - **Buy（外部調達）が適する場合**：汎用的な機能、コスト優位、短期間で導入が必要
- **Buy の選択肢**：パッケージ導入、SaaS利用、アウトソーシング、オフショア開発
- **ハイブリッド**：コア部分をMake、周辺をBuyとする組み合わせが現実的
- **TCO比較**：初期費用だけでなく、運用・保守・移行・廃棄のコストを含めて比較
- **よくある誤解**：「Make＝高い、Buy＝安い」とは限らない。カスタマイズコストやベンダーロックインリスクを考慮
- **実務活用**：クラウド移行、ERP導入判断。自社プロダクトの内製/外注判断

## 7. 財務分析

### NPV（正味現在価値）
- **定義**：投資によって得られる将来のキャッシュフローの現在価値合計から初期投資額を差し引いた値
- **計算式**：NPV = Σ[CFt / (1+r)^t] - I₀
  - CFt：t年目のキャッシュフロー
  - r：割引率（資本コスト/WACC）
  - I₀：初期投資額
- **判断基準**：NPV > 0 なら投資価値あり。複数案件ではNPVが最大のものを選択
- **メリット**：貨幣の時間価値を考慮。絶対的な価値創造額がわかる
- **デメリット**：割引率の設定が恣意的になりやすい
- **よくある誤解**：「NPVがプラスなら即投資」ではない。資本制約がある場合はPI（収益性指数＝NPV/投資額）で比較
- **実務活用**：大規模IT投資の可否判断。複数プロジェクト間の優先順位付け

### IRR（内部収益率）
- **定義**：NPVがゼロになる割引率。投資の期待収益率を示す
- **判断基準**：IRR > 資本コスト（ハードルレート）なら投資価値あり
- **NPVとの関係**：
  - 単一案件：NPVとIRRは同じ結論になる
  - 複数案件の比較：NPVを優先すべき（規模の違いを反映するため）
  - 非通常型キャッシュフロー（正負が交互）では複数のIRRが存在しうる
- **MIRR（修正IRR）**：再投資利率をIRRではなく資本コストで仮定する修正版
- **よくある誤解**：「IRRが高い＝良い投資」とは限らない。小規模高IRR vs 大規模中IRRでは後者がNPV大の場合がある
- **実務活用**：限られた資金を複数事業に配分する場合の比較指標

### ROI（投資利益率）
- **定義**：投資に対する利益の割合。ROI = 利益 / 投資額 × 100%
- **IT-ROIの計算**：
  - 利益 = IT投資による効果（コスト削減額＋売上増加額）- IT関連費用
  - 投資額 = 初期導入費用＋運用費用
- **ROIの限界**：
  - 時間の概念がない（いつの利益か不明）
  - 定性的効果を反映しにくい
  - 計算期間によって結果が大きく変わる
- **よくある誤解**：「ROI＝投資効果の全て」ではない。NPVやIRRと組み合わせて多面的に評価すべき
- **実務活用**：経営層への簡易的な投資効果説明。施策のクイック評価

### 回収期間法
- **定義**：投資額を回収するまでに要する期間を算出し、許容期間内かどうかで投資判断する手法
- **計算方法**：
  - 単純回収期間法：累積キャッシュフロー＝投資額となる時点
  - 割引回収期間法：割引キャッシュフローの累積＝投資額となる時点
- **メリット**：計算が簡単、直感的に理解しやすい、流動性リスクの評価に有効
- **デメリット**：回収後のキャッシュフローを無視、貨幣の時間価値を考慮しない（単純法の場合）
- **よくある誤解**：「回収期間が短い＝良い投資」とは限らない。回収後に大きなキャッシュフローが見込める案件を却下する恐れ
- **実務活用**：リスク評価の補助指標。短期的な投資判断に有効

### TCO（総保有コスト）
- **定義**：システムのライフサイクル全体にわたるコスト総額（Total Cost of Ownership）
- **コスト構成**：
  - **初期コスト**：ハードウェア、ソフトウェアライセンス、導入コンサル、カスタマイズ、データ移行
  - **運用コスト**：保守、運用人件費、ライセンス更新、ヘルプデスク、教育
  - **間接コスト**：ダウンタイム損失、生産性低下、エンドユーザーの自助時間
  - **廃棄コスト**：データ移行、旧システム撤去、契約解除
- **隠れたコスト（Hidden Cost）**：間接コストは全体の60〜80%を占めるとも言われる
- **よくある誤解**：「初期費用が安い＝TCOが安い」ではない。クラウドのTCOは長期的にオンプレを超える場合がある
- **実務活用**：Make-or-Buy判断、クラウド移行判断の根拠

### EVA（経済的付加価値）
- **定義**：税引後営業利益から投下資本にかかる資本コストを差し引いた値。企業が真に価値を創造しているかを測定
- **計算式**：EVA = NOPAT - 投下資本 × WACC
  - NOPAT（Net Operating Profit After Tax）：税引後営業利益
  - WACC（Weighted Average Cost of Capital）：加重平均資本コスト
- **意味**：EVA > 0 は資本コストを上回る価値を創造。EVA < 0 は価値を毀損
- **NPVとの関係**：将来予想されるEVAの現在価値の総和 = 全プロジェクトのNPV総和
- **よくある誤解**：「営業利益がプラス＝価値創造」ではない。資本コスト（株主・債権者が期待するリターン）を超えて初めて価値創造
- **実務活用**：IT投資の「真の価値」を経営層に説明する際の指標

### BSC（バランススコアカード）
- **定義**：財務指標だけでなく、非財務指標も含めた4つの視点から組織のパフォーマンスを包括的に評価するフレームワーク（ロバート・キャプラン＆デイビッド・ノートン）
- **4つの視点**：
  1. **財務の視点**：売上、利益、ROI、EVA
  2. **顧客の視点**：顧客満足度、市場シェア、顧客維持率
  3. **業務プロセスの視点**：品質、サイクルタイム、生産性
  4. **学習と成長の視点**：従業員スキル、情報システム活用度、組織文化
- **戦略マップ**：4つの視点間の因果関係を可視化（学習と成長→業務プロセス→顧客→財務）
- **IT-BSC**：IT部門のパフォーマンスをBSCで評価（ITユーザー視点、運用品質視点、将来志向視点、経営貢献視点）
- **よくある誤解**：「BSC＝業績評価ツール」だけではない。戦略の実行管理ツールとしての活用が本質
- **実務活用**：IT投資の多面的効果測定。KPI設計時の視点漏れ防止チェックリストとしても有効

### ABC（活動基準原価計算）
- **定義**：間接費を「活動（Activity）」単位で把握し、コストドライバーに基づいて製品・サービスに配賦する原価計算手法
- **従来の配賦方式との違い**：
  - 従来：直接作業時間等の単一基準で配賦→間接費の歪みが大きい
  - ABC：活動ごとのコストドライバーで配賦→より正確な原価把握
- **ABM（活動基準管理）**：ABCの結果を用いて活動の付加価値/非付加価値を分析し、改善につなげる管理手法
- **よくある誤解**：「ABC＝製造業だけ」ではない。IT部門のコスト配賦にも有効（サービス別、プロジェクト別原価計算）
- **実務活用**：IT部門のコスト可視化、チャージバック制度の設計

### 損益分岐点分析（CVP分析）
- **定義**：売上高と費用が等しくなる点（損益分岐点）を算出し、利益計画の基礎とする分析手法
- **計算式**：
  - 損益分岐点売上高 = 固定費 / (1 - 変動費率)
  - 変動費率 = 変動費 / 売上高
  - 損益分岐点販売量 = 固定費 / (販売単価 - 単位変動費)
- **安全余裕率**：(実際売上 - BEP売上) / 実際売上 × 100%。高いほど余裕がある
- **IT投資への応用**：SaaS等の従量課金モデルでの損益分岐点計算。ユーザー数×単価モデル
- **よくある誤解**：「固定費が高い＝悪い」ではない。固定費が高いほど損益分岐点は高いが、それを超えた後の利益率も高い（レバレッジ効果）
- **実務活用**：新規IT事業の採算シミュレーション。SaaS価格設定の根拠計算

## 8. 法務・コンプライアンス

### 個人情報保護法
- **定義**：個人情報の適正な取扱いに関するルールを定めた法律（2003年制定、2022年全面施行改正）
- **2022年改正の主要ポイント**：
  - 個人の権利利益の保護強化（利用停止・消去請求権の拡大）
  - **仮名加工情報**の新設：他の情報と照合しない限り個人を識別できないように加工した情報
  - **個人関連情報**の新設：Cookie等の提供規制
  - 漏洩等報告義務の法定化（個人情報保護委員会への報告＋本人への通知）
  - 法定刑の引き上げ（法人：1億円以下の罰金）
  - 越境移転規制の強化
- **個人情報の定義**：生存する個人に関する情報で、特定の個人を識別できるもの（氏名、生年月日、顔写真等）
- **要配慮個人情報**：人種、信条、社会的身分、病歴、犯罪歴等。取得には本人同意が必要
- **よくある誤解**：「匿名化すれば自由に使える」ではない。匿名加工情報にも取扱いルールがある
- **実務活用**：データ利活用戦略の法的制約の理解。個人データの越境移転設計

### GDPR（EU一般データ保護規則）
- **定義**：EU域内の個人データ保護を規定する法律（2018年施行）。域外適用あり
- **主要原則**：
  - **適法性・公正性・透明性**：データ処理の法的根拠を明確に
  - **目的制限**：収集時の目的以外に使用禁止
  - **データ最小化**：必要最小限のデータのみ収集
  - **正確性**：データの正確性を維持
  - **保存期間の制限**：目的達成後は削除
  - **完全性・機密性**：適切なセキュリティ措置
- **データ主体の権利**：アクセス権、訂正権、消去権（忘れられる権利）、データポータビリティ権、異議申立権
- **DPO（Data Protection Officer）**：一定規模以上の組織に設置義務
- **制裁金**：最大2,000万ユーロまたは全世界年間売上高の4%のいずれか高い方
- **十分性認定**：日本はEUから十分性認定を受けている（2019年）→日EU間のデータ移転が円滑化
- **よくある誤解**：「EU域外の企業には無関係」ではない。EU居住者のデータを扱う企業は域外適用の対象
- **実務活用**：グローバルシステム設計時のデータ保護要件。クラウドのデータロケーション設計

### 不正競争防止法
- **定義**：事業者間の公正な競争を確保するため、不正競争行為を規制する法律
- **主な不正競争行為**：
  - **営業秘密の不正取得・使用・開示**：3要件（秘密管理性・有用性・非公知性）
  - **限定提供データの不正取得**：2018年改正で新設（ビッグデータ保護）
  - **ドメイン名の不正取得**：サイバースクワッティング
  - **技術的制限手段の回避**：コピーガード解除等
  - **形態模倣**：他人の商品の形態を模倣した商品の提供
- **営業秘密の3要件**：
  1. 秘密管理性：秘密として管理されていること
  2. 有用性：事業活動に有用な情報であること
  3. 非公知性：公然と知られていないこと
- **よくある誤解**：「パスワードをかければ営業秘密」ではない。3要件すべてを満たす必要がある
- **実務活用**：IT戦略におけるデータ保護方針の策定。ベンダーとの秘密保持契約

### 知的財産権
- **産業財産権**（特許庁管轄）：
  - **特許権**：発明の保護（出願から20年）。新規性・進歩性・産業上利用可能性
  - **実用新案権**：物品の形状等の考案（出願から10年）。無審査登録
  - **意匠権**：デザインの保護（出願から25年）。画像デザインも対象（2020年改正）
  - **商標権**：ブランドの保護（登録から10年、更新可能）
- **著作権**（文化庁管轄）：
  - 創作と同時に発生（無方式主義）
  - プログラムの著作物：ソースコードは著作権で保護される
  - 著作権の保護期間：著作者の死後70年
  - 職務著作：法人等の発意に基づく場合、法人が著作者となる
- **ソフトウェアの保護**：
  - アルゴリズム→特許権（ビジネスモデル特許含む）
  - ソースコード→著作権
  - UI/UXデザイン→意匠権
- **よくある誤解**：「特許出願中＝保護されている」ではない。特許権は登録されて初めて効力を持つ
- **実務活用**：IT戦略におけるIPポートフォリオ管理。オープンソース活用時のライセンス管理

### 電子契約
- **定義**：書面の代わりに電子データで契約を締結する方式
- **法的根拠**：
  - 電子署名法（2001年）：電子署名に手書き署名と同等の法的効力を付与
  - 電子帳簿保存法（2022年改正）：電子取引データの保存義務化
  - e-文書法：法令で保存が義務づけられた書類の電子化を認める
- **電子署名の種類**：
  - 当事者型（本人の秘密鍵で署名）：法的効力が強い
  - 立会人型（事業者が本人確認を代行）：導入が容易
- **タイムスタンプ**：文書の存在時刻と非改ざんを証明
- **よくある誤解**：「電子メールでの合意＝電子契約」ではない。電子署名法の要件を満たす必要がある
- **実務活用**：ベンダー契約のデジタル化推進。電子帳簿保存法への対応戦略

### クラウドバイデフォルト原則
- **定義**：政府情報システムの整備にあたり、クラウドサービスの利用を第一候補（デフォルト）として検討する原則
- **背景**：2018年「政府情報システムにおけるクラウドサービスの利用に係る基本方針」で明文化
- **ISMAP（Information system Security Management and Assessment Program）**：政府のクラウドサービス安全性評価制度。ISMAPリストに登録されたクラウドサービスから調達
- **判断フロー**：
  1. SaaS/PaaSで実現可能か検討
  2. IaaSで実現可能か検討
  3. オンプレミスの検討（クラウドが不適な場合のみ）
- **よくある誤解**：「クラウドバイデフォルト＝クラウド強制」ではない。機密性、可用性等の要件によりオンプレミスが適切な場合は除外
- **実務活用**：公共系システム案件の調達戦略。政府系クラウド要件への対応

## 9. マーケティング

### STP分析
- **定義**：市場をセグメンテーションし、ターゲットを選定し、ポジショニングを決定する戦略立案プロセス
- **Segmentation（市場細分化）**：
  - 地理的変数：地域、気候、人口密度
  - 人口統計的変数：年齢、性別、所得、職業
  - 心理的変数：ライフスタイル、価値観、性格
  - 行動変数：使用頻度、ロイヤルティ、購買動機
- **Targeting（ターゲット選定）**：
  - 無差別型：全市場に同一製品
  - 差別型：複数セグメントに異なる製品
  - 集中型：特定セグメントに特化
  - マイクロマーケティング：個々の顧客にカスタマイズ
- **Positioning（位置取り）**：
  - 競合との差別化ポイントを明確化
  - ポジショニングマップで可視化（2軸：価格×品質等）
- **よくある誤解**：「STPは一方通行」ではない。P→S/Tへのフィードバックで反復的に精緻化
- **実務活用**：DXによる新サービス設計時のマーケティング戦略立案

### 4P / 4C
- **4P（売り手視点）**：
  1. Product（製品）：品質、機能、デザイン、ブランド
  2. Price（価格）：定価、割引、支払条件
  3. Place（流通）：チャネル、立地、物流
  4. Promotion（販促）：広告、PR、セールス、SP
- **4C（買い手視点）**（ロバート・ラウターボーン）：
  1. Customer Value（顧客価値）← Product
  2. Cost（顧客コスト）← Price
  3. Convenience（利便性）← Place
  4. Communication（コミュニケーション）← Promotion
- **7P**（サービスマーケティング）：4P + People（人）+ Process（プロセス）+ Physical Evidence（物的証拠）
- **よくある誤解**：「4Pは古い」ではない。4Cは4Pを顧客視点で再定義したもの。両方を使い分ける
- **実務活用**：デジタルサービスのマーケティングミックス設計

### CRM（顧客関係管理）
- **定義**：顧客との長期的な関係を構築・維持し、顧客満足度とLTVを最大化するための経営戦略
- **CRMの3要素**：
  1. **オペレーショナルCRM**：営業・マーケティング・サポートの業務効率化（SFA、MA、カスタマーサポート）
  2. **アナリティカルCRM**：顧客データ分析による意思決定支援（データマイニング、予測分析）
  3. **コラボレーティブCRM**：部門間・チャネル間の顧客情報共有
- **RFM分析**：Recency（最終購買日）、Frequency（購買頻度）、Monetary（購買金額）で顧客をセグメント化
- **よくある誤解**：「CRM＝CRMツール導入」ではない。CRMは経営戦略であり、ツールは手段に過ぎない
- **実務活用**：顧客データ基盤構築の戦略立案。データドリブンマーケティングの推進

### デジタルマーケティング
- **定義**：デジタル技術・チャネルを活用したマーケティング活動の総称
- **主な手法**：
  - **SEO/SEM**：検索エンジン最適化/マーケティング
  - **コンテンツマーケティング**：価値あるコンテンツで顧客を惹きつける
  - **SNSマーケティング**：ソーシャルメディアを活用した認知・関係構築
  - **メールマーケティング**：セグメント別のパーソナライズメール
  - **リスティング広告/ディスプレイ広告**：運用型広告
  - **アフィリエイト**：成果報酬型広告
- **データ活用**：
  - **DMP（Data Management Platform）**：顧客データの統合管理
  - **CDP（Customer Data Platform）**：ファーストパーティデータの統合
  - **A/Bテスト**：施策の効果検証
- **よくある誤解**：「デジタルマーケティング＝Web広告」ではない。データに基づく顧客体験全体の最適化
- **実務活用**：DXにおけるマーケティング変革の提案。EC事業のグロース設計

### MA（マーケティングオートメーション）
- **定義**：マーケティング活動を自動化し、見込み顧客の獲得・育成・選別を効率化するツール・プロセス
- **主要機能**：
  - **リードジェネレーション**：見込み顧客の獲得（Webフォーム、LP等）
  - **リードナーチャリング**：見込み顧客の育成（メール配信、コンテンツ提供）
  - **リードスコアリング**：見込み顧客の購買意欲をスコア化
  - **リードクオリフィケーション**：営業に引き渡すリードの選別
- **SFAとの関係**：MA→リード獲得・育成、SFA→商談管理・受注。MAからSFAへリードを引き渡す
- **よくある誤解**：「MAを導入すれば自動で売れる」ではない。コンテンツ設計とシナリオ設計が成功の鍵
- **実務活用**：BtoBマーケティングのデジタル化推進

### カスタマージャーニー
- **定義**：顧客が製品・サービスを認知してから購入・利用・推奨に至るまでの一連の体験プロセスを可視化したもの
- **構成要素**：
  - **ペルソナ**：典型的な顧客像の設定
  - **フェーズ**：認知→興味・関心→比較検討→購入→利用→推奨
  - **タッチポイント**：各フェーズでの顧客接点（Web、店舗、SNS、コールセンター等）
  - **行動・思考・感情**：各フェーズでの顧客の心理状態
  - **課題・機会**：各フェーズでの課題と改善機会
- **CX（カスタマーエクスペリエンス）**：全タッチポイントを通じた顧客体験の総合評価
- **よくある誤解**：「カスタマージャーニー＝一方通行」ではない。デジタル時代は非線形で複雑（行きつ戻りつ）
- **実務活用**：DXによるCX改善施策の設計。オムニチャネル戦略の立案

### LTV（顧客生涯価値）
- **定義**：一人の顧客が取引期間を通じてもたらす利益の総額
- **計算式（基本）**：LTV = 平均購買単価 × 購買頻度 × 継続期間
- **計算式（利益ベース）**：LTV = (平均購買単価 × 粗利率 × 購買頻度 × 継続期間) - 顧客獲得コスト
- **サブスクリプションモデル**：LTV = ARPU（月額平均収益）/ チャーンレート（月次解約率）
- **LTV/CAC比**：顧客生涯価値÷顧客獲得コスト。3以上が健全、1未満は損失
- **LTV向上施策**：アップセル、クロスセル、リテンション施策、解約防止
- **よくある誤解**：「LTVが高い＝優良顧客」だが、獲得コストが高すぎると利益貢献しない。LTV/CAC比で判断
- **実務活用**：マーケティング投資の最適配分。CRM戦略の評価指標

## 10. イノベーション管理

### オープンイノベーション
- **定義**：組織の境界を越えて、外部の知識・技術・アイデアを活用し、イノベーションを加速する手法（ヘンリー・チェスブロウ）
- **2つの方向**：
  - **アウトサイドイン**：外部の知識・技術を社内に取り込む（技術ライセンス、M&A、産学連携）
  - **インサイドアウト**：社内の知識・技術を外部に提供する（スピンオフ、ライセンス供与）
- **実践手段**：
  - アクセラレータープログラム
  - CVC（コーポレートベンチャーキャピタル）
  - ハッカソン
  - 産学官連携
  - オープンソース活用
- **クローズドイノベーションとの違い**：クローズドは自社R&Dのみ。研究開発費高騰と製品ライフサイクル短縮により限界
- **よくある誤解**：「オープン＝無料公開」ではない。戦略的な知識の授受と知財管理が不可欠
- **実務活用**：DX推進におけるスタートアップ連携戦略。OSS活用方針の策定

### デザイン思考
- **定義**：ユーザーの本質的なニーズを深く理解し、創造的な解決策を生み出す人間中心のイノベーション手法（IDEO/スタンフォードd.school）
- **5つのプロセス**：
  1. **共感（Empathize）**：ユーザーの行動・感情を観察・理解
  2. **問題定義（Define）**：真の課題を明確化（着眼点=POV）
  3. **創造（Ideate）**：解決策のアイデアを大量に創出（ブレインストーミング）
  4. **プロトタイプ（Prototype）**：素早く形にする（ラピッドプロトタイピング）
  5. **テスト（Test）**：ユーザーからフィードバックを得る
- **反復プロセス**：5ステップは一方通行ではなく、何度も行き来する
- **よくある誤解**：「デザイン思考＝デザイナーのもの」ではない。経営者・技術者を含む全員のための思考法
- **実務活用**：DXにおけるユーザー体験設計。新規サービス開発の初期フェーズ

### リーンスタートアップ
- **定義**：仮説を素早く検証し、最小限のリソースで事業を構築するスタートアップ手法（エリック・リース）
- **BMLループ（Build-Measure-Learn）**：
  1. **Build**：MVPを構築
  2. **Measure**：データを測定
  3. **Learn**：学びを得て仮説を修正
- **ピボット**：仮説が検証できなかった場合に方向転換する意思決定
- **革新会計（Innovation Accounting）**：従来の財務指標ではなく、学習を評価する指標（エンゲージメント等）
- **よくある誤解**：「リーンスタートアップ＝スタートアップ専用」ではない。大企業の新規事業開発にも適用可能
- **実務活用**：DXの新規サービス開発プロセス。段階的な検証プロセスの設計

### PoC（Proof of Concept）
- **定義**：新しい技術や手法が実現可能であることを検証するための試験的な取り組み
- **PoCの位置づけ**：構想→PoC→パイロット→本番導入
- **PoC成功の条件**：
  - 明確な検証項目と成功基準の事前設定
  - 期間と予算の限定（だらだら続けない）
  - 本番環境に近い条件での検証
  - 検証結果に基づく明確な判断（Go/No-Go）
- **PoCの罠（PoC疲れ）**：PoCを繰り返すだけで本番導入に至らない問題
- **よくある誤解**：「PoC成功＝事業成功」ではない。技術的実現可能性と事業的実現可能性は別
- **実務活用**：AI/IoT等の新技術導入判断。技術選定の根拠形成

### MVP（Minimum Viable Product）
- **定義**：顧客に価値を提供できる最小限の機能を持つ製品。仮説検証と学習を目的とする
- **MVPの目的**：最小限のコストと時間で市場の反応を確認し、製品開発の方向性を検証する
- **MVPの種類**：
  - コンシェルジュ型：手動で価値を提供して需要を検証
  - ウィザード・オブ・オズ型：裏で人力対応しながらシステムに見せる
  - ランディングページ型：LPだけ作って反応を見る
  - プロトタイプ型：動作する最小限の製品
- **PoCとの違い**：PoCは技術検証、MVPは市場検証（顧客の反応を見る）
- **よくある誤解**：「MVP＝品質の低い製品」ではない。最小限だが顧客に価値を提供できる品質は必須
- **実務活用**：DXにおける新規サービス開発の初期フェーズ

### アジャイル経営
- **定義**：アジャイル開発の原則を経営全体に拡張し、環境変化に素早く適応する経営スタイル
- **特徴**：
  - 小さな実験を繰り返して学習する（試行→検証→適応）
  - 階層型組織からネットワーク型組織へ
  - 権限委譲と自律的チーム
  - 顧客フィードバックを迅速に反映
- **SAFe（Scaled Agile Framework）**：大規模組織でアジャイルを実践するためのフレームワーク
- **よくある誤解**：「アジャイル経営＝計画不要」ではない。方向性（ビジョン）は明確にし、実行方法をアジャイルにする
- **実務活用**：DX推進組織の運営方針。プロダクト開発チームの組織設計

## 11. 組織・人材戦略

### CISO（Chief Information Security Officer）
- **役割**：組織の情報セキュリティを統括する最高責任者
- **責任範囲**：
  - 情報セキュリティ戦略の策定・推進
  - セキュリティポリシーの制定・運用
  - インシデント対応の指揮
  - コンプライアンス（個人情報保護法、GDPR等）の確保
  - セキュリティ投資の意思決定
- **CIOとの関係**：CISOはCIOの配下に置かれることが多いが、独立性を確保するために同格に置く組織も増加
- **よくある誤解**：「CISO＝技術者」ではない。経営層としてビジネスリスクの観点からセキュリティを判断する能力が必要
- **実務活用**：セキュリティガバナンス体制の設計。DX推進時のセキュリティ組織論

### CDO（Chief Digital Officer / Chief Data Officer）
- **2つのCDO**：
  - **Chief Digital Officer**：デジタルトランスフォーメーションを統括
  - **Chief Data Officer**：データ戦略・データガバナンスを統括
- **CDO（デジタル）の役割**：
  - DX戦略の策定・推進
  - デジタル技術を活用した事業変革の主導
  - 全社横断のDX推進体制の構築
  - デジタル人材の確保・育成
- **CDO（データ）の役割**：
  - データ戦略の策定
  - データガバナンスの確立
  - データ品質管理
  - データ活用基盤の整備
- **日本の現状**：CDO設置率は米独に比べて低い（DX動向2025）。経営層のデジタルリテラシー不足が課題
- **よくある誤解**：「CDO＝IT部門長」ではない。CDOは事業変革のリーダーであり、IT部門を超えた全社的な役割
- **実務活用**：DX推進組織の設計提案。データ戦略の責任者設置の根拠

### CTO（Chief Technology Officer）
- **役割**：技術戦略を統括する最高技術責任者
- **責任範囲**：
  - 技術戦略の策定・推進（R&D方針、技術ロードマップ）
  - 技術的な意思決定（アーキテクチャ選定、技術投資判断）
  - 技術組織の構築・マネジメント
  - イノベーション推進（新技術の評価・導入）
- **CIO・CDOとの違い**：
  - CIO：情報システム全般の統括（IT投資・運用）
  - CTO：技術戦略・R&D（技術的イノベーション）
  - CDO：デジタル変革・データ活用（事業変革）
- **よくある誤解**：「CTO＝最も技術に詳しい人」ではない。経営戦略と技術戦略を橋渡しする能力が最重要

### デジタル人材育成
- **デジタル人材の類型**（経産省分類）：
  - **ビジネスアーキテクト**：DX推進を主導する企画人材
  - **データサイエンティスト**：データ分析・AI活用の専門人材
  - **ソフトウェアエンジニア**：システム開発の技術人材
  - **サイバーセキュリティ人材**：セキュリティの専門人材
  - **デザイナー**：UX/UIの設計人材
- **リスキリング**：既存の従業員にデジタルスキルを習得させる再教育
- **デジタルリテラシー標準（Di-Lite）**：全ビジネスパーソンが身につけるべきデジタルスキルの標準（IPA）
  - IT・ソフトウェア
  - 数理・データサイエンス
  - AI
- **よくある誤解**：「デジタル人材＝プログラマー」ではない。ビジネスとデジタルを橋渡しできる人材が最も不足
- **実務活用**：DX推進の人材戦略立案。採用・育成計画の根拠

### 組織変革マネジメント
- **コッターの8段階変革モデル**（ジョン・コッター）：
  1. 危機意識の醸成
  2. 変革推進チームの結成
  3. ビジョンと戦略の策定
  4. ビジョンの周知
  5. 従業員の自発的行動の促進
  6. 短期的な成果の実現
  7. さらなる変革の推進
  8. 新しい文化の定着
- **レヴィンの変革モデル**：解凍（Unfreezing）→変革（Changing）→再凍結（Refreezing）
- **変革の抵抗要因**：
  - 現状維持バイアス
  - スキル不安
  - 権力構造の変化への恐れ
  - コミュニケーション不足
- **よくある誤解**：「良い戦略があれば変革は成功する」ではない。組織文化と人の行動変容が最大の課題
- **実務活用**：DX推進時の組織変革設計。新システム導入時のユーザー受容性向上策

### チェンジマネジメント
- **定義**：組織変革を計画的に管理し、変革の成功確率を高めるための体系的アプローチ
- **ADKARモデル**（プロサイ）：
  - **A**wareness：変革の必要性の認識
  - **D**esire：変革への参画意欲
  - **K**nowledge：変革に必要な知識
  - **A**bility：変革を実行する能力
  - **R**einforcement：変革の定着・強化
- **チェンジエージェント**：変革を推進するキーパーソン。経営層のスポンサーシップとミドルマネジメントの巻き込みが鍵
- **よくある誤解**：「チェンジマネジメント＝プロジェクト管理」ではない。プロジェクト管理は「技術面」、チェンジマネジメントは「人的側面」
- **実務活用**：DX推進プロジェクトにおける利用者の受容性向上策

## 12. 実務活用パターン

### IT投資判断での活用
- **投資提案書の構成**：
  1. 経営環境分析（PEST/SWOT/5 Forces）
  2. 経営課題とIT戦略の整合性（BSC/戦略マップ）
  3. 投資効果の定量化（NPV/IRR/ROI/回収期間法）
  4. TCO分析（Make-or-Buy/クラウド vs オンプレミス）
  5. リスク評価と対策
  6. 段階的推進計画（PoC→パイロット→本番）
- **フレームワークの使い分け**：
  - 新規投資：NPV/IRRで絶対評価
  - 複数案件比較：PI（収益性指数）/BSCで相対評価
  - 短期回収必要：回収期間法で流動性評価
  - 既存投資見直し：EVAで価値創造評価

### DX推進での活用
- **DX戦略策定プロセス**：
  1. 外部環境分析（PEST + DX動向調査）
  2. 内部環境分析（SWOT + DX推進指標による自己診断）
  3. ビジネスモデル設計（BMC + バリューチェーン分析）
  4. IT戦略策定（EA + クラウド戦略 + API経済）
  5. 組織・人材戦略（CDO設置 + デジタル人材育成 + チェンジマネジメント）
  6. 推進計画策定（ロードマップ + KGI/KPI + BSC）
  7. ガバナンス体制構築（ITガバナンス + COBIT + デジタルガバナンス・コード）
- **DX推進の成功要因**：
  - 経営層のコミットメント（トップダウン）
  - 全社横断の推進体制（サイロ化の打破）
  - データ基盤の整備（データドリブン経営の土台）
  - アジャイルな実行（小さく始めて素早く検証）
  - 組織文化の変革（失敗を許容する文化）

### ビジネスモデル分析での活用
- **分析フレームワークの組み合わせ**：
  1. **マクロ分析**：PEST → 市場機会/脅威の特定
  2. **業界分析**：5 Forces → 業界構造と収益性の把握
  3. **自社分析**：SWOT/VRIO → 競争優位の源泉特定
  4. **戦略策定**：アンゾフ/PPM → 成長方向性の決定
  5. **ビジネスモデル設計**：BMC → 新しい価値提供の設計
  6. **マーケティング**：STP/4P → 顧客への届け方
  7. **実行計画**：BSC → 戦略の実行管理
  8. **投資評価**：NPV/IRR → 投資判断
- **分析レポートの構成**：
  - エグゼクティブサマリー（結論と提言）
  - 環境分析（外部→内部の順）
  - 戦略オプションの評価（複数案の比較）
  - 推奨戦略と実行計画
  - リスクと対策
  - 投資効果と評価指標

### 知識を実務に活かすマインドセット
1. **フレームワークは「考えるための道具」**：フレームワークを埋めることが目的ではなく、思考を構造化するために使う
2. **複数のフレームワークを組み合わせる**：単一のフレームワークで全てを分析できない。目的に応じて組み合わせる
3. **定量と定性の両面で評価する**：NPVだけでなくBSCの非財務指標も含めた多面評価
4. **技術と経営の橋渡し**：ITストラテジストの最大の価値は、技術を経営の言葉で語れること
5. **変革は人の問題**：どんなに良い戦略も、組織と人が変わらなければ実現しない。チェンジマネジメントを忘れない
