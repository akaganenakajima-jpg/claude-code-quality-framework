# IPA システムアーキテクト（SA）試験 — 満点レベル知識ベース

> 最終更新: 2026-03-16
> 対象: 令和8年度（2026年度）以降 CBT方式対応

---

## 1. 試験概要

### 試験の位置づけ
- **レベル4**（高度情報処理技術者試験）
- 対象者像: ITストラテジストの提案を受け、情報システムの**要件定義→アーキテクチャ設計→開発主導**を担う者
- データベーススペシャリスト試験と出題傾向に類似性が高い（春DB→秋SAのダブル受験が多い）

### 試験構成（2026年度〜 CBT方式）

| 科目 | 旧名称 | 形式 | 出題/解答 | 時間 | 合格点 |
|------|--------|------|-----------|------|--------|
| 科目A-1 | 午前I | 多肢選択 | 30問/30問 | 50分 | 60点 |
| 科目A-2 | 午前II | 多肢選択 | 25問/25問 | 40分 | 60点 |
| 科目B-1 | 午後I | 記述式 | 3問中2問選択 | 90分 | 60点 |
| 科目B-2 | 午後II | 論述式 | 2問中1問選択 | 120分 | A判定 |

- 科目A-1未達の場合、以降の採点は行われない（足切り方式）
- 科目A-1は2年間の免除制度あり（応用情報合格者、他の高度試験合格者）

### 合格率
- 例年 **13〜16%** を推移（令和6年度: 15.0%）
- 受験者の大半が応用情報以上の実力を持つため、実質的な難易度は数値以上に高い
- 論述試験（小論文）があるため、スペシャリスト系試験より一般に難しいとされる

### 重点出題分野（科目A-2）
1. **システム開発技術**（最重点）
2. **情報セキュリティ**（2020年シラバス改訂で強化）
3. **システム企画**

### 午後試験の特徴

**科目B-1（旧午後I）記述式**:
- 45分×2問のペース配分が合否を分ける
- 「国語力」が重視される — 問題文中の根拠に基づく解答が求められる
- 出題パターン: **設計系**（最頻出）、**DB系**（1〜2問）、**ST（戦略）系**（1問程度）
- 令和6年度から組込み分野はエンベデッドシステムスペシャリスト（ES）に集約

**科目B-2（旧午後II）論述式**:
- 2,200〜3,600字の小論文を手書き120分で記述
- 「ア」（800字以内）「イ」（800〜1,600字）「ウ」（600〜1,200字）の3設問構成
- **攻めの問題**（DX・先進技術・アジャイル）と**守りの問題**（既存改修・バッチ設計）が交互に出題

### 論述試験のポイント
- 素晴らしいアイデアや豊富な経験は不要 — **設問の要求に正確に答える**ことが最重要
- **不合格答案の特徴を回避する**ことが合格への最短路
  - 問われていないことを書く
  - 具体性がない（「適切に対応した」等の抽象表現）
  - 規模感・工数の整合性が取れていない
  - 字数不足（設問ごとの下限を割る）
- システム概要欄の規模・開発期間・チーム人数は**事前にテンプレート化**しておく

### よくある誤解
- 「高度な技術論を展開すべき」→ 実際は**実務に基づく具体的な判断と根拠**が評価される
- 「実務経験がないと受からない」→ 参考書の事例をベースに**自分の言葉で再構成**すれば対応可能
- 「午前対策は不要」→ 科目A-2の専門知識は午後の解答にも直結する

## 2. システム要件定義

### 機能要件と非機能要件

**機能要件（Functional Requirements）**:
- システムが「何をするか」を定義する
- ユースケース、ビジネスルール、データ処理ロジック、画面・帳票仕様
- 要件定義書の中核を構成し、ユーザーが直接認識できる振る舞い

**非機能要件（Non-Functional Requirements）**:
- システムが「どのように動くか」の品質特性を定義する
- ユーザーが明示的に要求しないが、システム品質に直結する

### IPA 非機能要求グレード（6大カテゴリ・239メトリクス）

| カテゴリ | 内容 | 代表的メトリクス |
|----------|------|------------------|
| **可用性** | 継続利用の保証 | 稼働率、RPO、RTO、冗長構成 |
| **性能・拡張性** | 処理能力と将来対応 | レスポンスタイム、スループット、同時接続数 |
| **運用・保守性** | 運用のしやすさ | 監視方式、バックアップ、パッチ適用 |
| **移行性** | 移行時の制約 | 移行方式、データ変換、並行運用期間 |
| **セキュリティ** | 情報保護 | 認証方式、暗号化、監査ログ |
| **システム環境・エコロジー** | 物理環境・省エネ | 設置環境、CO2排出、電力効率 |

適用の流れ: (1)モデルシステム選定 → (2)重要項目のレベル決定 → (3)それ以外のレベル決定
社会的影響度により3段階（社会的影響が極めて大きい/限定的/ほとんどない）でベースラインを設定する。

### FURPS+

Hewlett-Packard発祥の品質モデル。非機能要件を体系的に整理するフレームワーク。

| 要素 | 意味 | 具体例 |
|------|------|--------|
| **F**unctionality | 機能性 | 機能適合性、相互運用性、セキュリティ |
| **U**sability | 使用性 | 学習容易性、操作性、アクセシビリティ |
| **R**eliability | 信頼性 | 成熟性、障害許容性、回復性 |
| **P**erformance | 性能 | 応答時間、スループット、資源利用効率 |
| **S**upportability | 保守性 | テスト容易性、変更容易性、移植性 |
| **+** | 追加制約 | 設計制約、実装制約、インターフェース要件、物理制約 |

### 要件定義プロセス

```
ビジネス要件 → システム要件 → ソフトウェア要件
（なぜ作るか）  （何を作るか）  （どう作るか）
```

1. **利害関係者の識別**: ステークホルダー分析（関心・影響力マトリクス）
2. **要件の抽出**: インタビュー、ワークショップ、プロトタイピング、ユースケース分析
3. **要件の分析**: 優先順位付け（MoSCoW法）、競合解消、実現可能性評価
4. **要件の仕様化**: SRS（Software Requirements Specification）、IEEE 830準拠
5. **要件の検証**: レビュー、ウォークスルー、形式検査
6. **要件の管理**: 変更管理、ベースライン管理

### 要件トレーサビリティ

要件の「前方追跡性（Forward Traceability）」と「後方追跡性（Backward Traceability）」を確保する。

**トレーサビリティマトリクス（RTM）**:
- ビジネス要件 ↔ システム要件 ↔ 設計 ↔ テストケース の対応関係を管理
- 変更影響分析の基盤となる
- 要件の漏れ・過剰実装を検出する

### よくある誤解
- 「非機能要件は後から決めればよい」→ アーキテクチャに直結するため**要件定義段階で確定**が必要
- 「機能要件と非機能要件は独立」→ 非機能要件は機能要件の**実現方式を制約**する関係
- 「ユーザーは非機能要件を理解しない」→ ビジネスインパクトに変換して説明する（例: 「システム停止1時間あたり売上損失○万円」）

### 実務活用
- 要件定義書のレビュー時に非機能要求グレードの6カテゴリをチェックリストとして使う
- 要件の「完全性」はトレーサビリティマトリクスで機械的に検証する
- MoSCoW法で優先順位を明確にし、MVP（Minimum Viable Product）の範囲を決定する

## 3. システム方式設計

### 集中型 vs 分散型

| 観点 | 集中型 | 分散型 |
|------|--------|--------|
| データ一貫性 | 容易（単一DB） | 困難（CAP定理の制約） |
| スケーラビリティ | 垂直拡張のみ | 水平拡張可能 |
| 障害影響 | 全体停止リスク | 部分障害に限定可能 |
| 運用複雑性 | 低い | 高い |
| コスト特性 | 初期高・運用低 | 初期低・運用高（場合による） |

### クライアントサーバ（C/S）アーキテクチャ
- **2層C/S**: クライアント（プレゼンテーション+ビジネスロジック）+ サーバ（データ）
  - ファットクライアント問題（配布・更新コスト）
- **3層C/S**: プレゼンテーション層 + アプリケーション層 + データ層
  - 各層を独立にスケール可能
  - Webアプリケーションの基本形

### 3層アーキテクチャ

```
[プレゼンテーション層] ← UI/UX、入力バリデーション
         ↓↑
[ビジネスロジック層]   ← ドメインルール、トランザクション制御
         ↓↑
[データアクセス層]     ← CRUD、ORM、クエリ最適化
```

- 層間の依存は**上から下への一方向**が原則（依存性逆転を適用する場合は後述）
- 各層をプロセス分離すると「ティア」、論理分離のみは「レイヤー」

### マイクロサービスアーキテクチャ

**定義**: 小さく独立してデプロイ可能なサービスの集合としてシステムを構成する方式。

**特徴**:
- サービスごとに独立したデータストア（Database per Service）
- サービス間はAPI（REST/gRPC）またはメッセージングで通信
- 各サービスを独立にデプロイ・スケール可能
- チームの自律性（Conway's Law との整合）

**トレードオフ**:
| メリット | デメリット |
|----------|-----------|
| 独立デプロイ | 分散トランザクションの複雑性 |
| 技術選択の自由度 | サービス間通信のオーバーヘッド |
| 障害の局所化 | 運用・監視の複雑性増大 |
| スケールの柔軟性 | データ一貫性の確保が困難 |

**モノリスからの移行判断基準**:
- チーム規模が2-pizza rule（8〜10名）を超える
- デプロイ頻度を上げたいがモノリスが障壁
- 特定機能のみスケールアウトが必要

### サーバレスアーキテクチャ

**定義**: サーバ管理を抽象化し、関数単位でコードを実行する方式（FaaS: Function as a Service）。

**特徴**:
- イベント駆動（HTTPリクエスト、キューメッセージ、スケジュール等がトリガー）
- 使用量課金（実行時間×メモリ）
- 自動スケール（ゼロからの起動を含む）
- コールドスタート問題（初回起動の遅延）

**適用場面**: 不規則なトラフィック、バッチ処理、Webhook処理、IoTバックエンド
**不向き**: 長時間実行処理、低レイテンシ要求、ステートフルな処理

### イベント駆動アーキテクチャ（EDA）

**定義**: イベントの生成・検出・消費・反応によってシステムを構成する方式。

**パターン**:
- **イベント通知**: 発生を通知するだけ（購読者が詳細を取得）
- **イベント伝搬（Event-Carried State Transfer）**: イベントにデータを含めて伝搬
- **イベントソーシング**: 状態変更をイベントの連鎖として記録

**構成要素**: イベントプロデューサー → イベントチャネル（ブローカー/バス） → イベントコンシューマー

**利点**: 疎結合、非同期処理による高スループット、時間的デカップリング
**課題**: イベント順序の保証、冪等性の確保、デバッグの困難さ

### CQRS（Command Query Responsibility Segregation）

**定義**: データの書き込み（Command）と読み取り（Query）を別モデルで処理する方式。

```
[Command側]          [Query側]
Write Model   →(同期/非同期)→  Read Model
  ↓                              ↑
Command DB                    Query DB（非正規化・最適化）
```

**適用場面**:
- 読み取りと書き込みの負荷特性が大きく異なる場合
- 複雑なドメインモデルが読み取りのパフォーマンスを阻害する場合
- イベントソーシングとの組み合わせ

**注意**: 読み書きモデルの**結果整合性（Eventual Consistency）**を受け入れる必要がある

### ヘキサゴナルアーキテクチャ（Ports & Adapters）

**定義**: ドメインロジックを中心に置き、外部との接続をポート（インターフェース）とアダプター（実装）で抽象化する方式。

```
         [外部システム]
              ↓
    [Driving Adapter] → [Port(入力IF)] → [ドメインロジック]
                                                ↓
                         [Port(出力IF)] ← [Driven Adapter] → [DB/外部API]
```

- **Driving Adapter**: 外部からドメインを呼び出す（REST Controller、CLI等）
- **Driven Adapter**: ドメインが外部リソースを利用する（Repository実装、メール送信等）
- ドメインは外部に**一切依存しない**（依存性逆転原則）

**利点**: テスト容易性、技術変更への耐性、ドメインの純粋性維持
**試験での出題**: 午後Iで「なぜインターフェースを介在させるのか」の設計意図を問われる

### よくある誤解
- 「マイクロサービスは常にモノリスより優れている」→ **複雑性のトレードオフ**であり、小規模チームではモノリスが適切
- 「サーバレスは万能」→ コールドスタート・実行時間制限・ベンダーロックインを考慮
- 「CQRSはどこでも使える」→ 読み書きの負荷差が小さいシステムでは過剰設計

### 実務活用
- アーキテクチャ選定は**品質特性（非機能要件）駆動**で行う
- ADR（Architecture Decision Record）でアーキテクチャ判断の根拠を記録する
- アーキテクチャレビューではトレードオフの**明示的な文書化**を求める

## 4. 信頼性設計

### 基本概念の体系

```
信頼性設計
├── フォールトアボイダンス（障害を起こさない）
│   ├── 高品質部品の使用
│   ├── 設計レビュー・検証の徹底
│   └── バーンイン試験（初期故障の排除）
├── フォールトトレランス（障害が起きても動く）
│   ├── フェールセーフ（安全側に停止）
│   ├── フェールソフト（縮退運転で継続）
│   ├── フェールオーバー（待機系に切替）
│   └── フールプルーフ（人的ミス防止）
└── フォールトリカバリ（障害から復旧する）
    ├── ロールバック（障害前の状態に戻す）
    ├── ロールフォワード（ログから最新に進める）
    └── チェックポイントリスタート
```

### フェールセーフ（Fail-Safe）
- **定義**: 障害発生時に**安全側に倒して停止**する設計思想
- **例**: 信号機の故障時は赤点滅、踏切の故障時は遮断機を下ろす
- **IT例**: データベース不整合検出時にトランザクションをロールバック、入力値異常時に処理を拒否

### フェールソフト（Fail-Soft）
- **定義**: 障害発生時に**機能を縮退させて運用を継続**する設計思想
- **例**: CPUの1コア故障時に残コアで処理継続、ネットワーク帯域低下時に画質を落として配信継続
- **IT例**: 検索サービスの一部インデックスが破損しても部分的な検索結果を返す

### フォールトトレラント（Fault-Tolerant）
- **定義**: 障害が発生しても**正常な機能を維持**する設計思想
- 冗長化（Redundancy）が主要手段
- **例**: RAID、クラスタリング、デュアル系システム

### フォールトアボイダンス（Fault-Avoidance）
- **定義**: そもそも**障害を発生させない**ための予防的設計
- 高信頼性部品の採用、徹底したテスト、形式検証
- **例**: 航空宇宙分野の高信頼性設計、医療機器のバリデーション

### 冗長化の方式

| 方式 | 説明 | 切替時間 | コスト |
|------|------|----------|--------|
| **デュプレックス（ホットスタンバイ）** | 待機系がリアルタイム同期 | 秒〜分 | 高 |
| **デュプレックス（コールドスタンバイ）** | 待機系は起動のみ | 分〜時間 | 中 |
| **デュプレックス（ウォームスタンバイ）** | 待機系は部分同期 | 分 | 中〜高 |
| **デュアル** | 両系が同時稼働、結果照合 | 即時 | 最高 |
| **N+1冗長** | N台の本番+1台の予備 | 分 | 比較的低 |
| **Active-Active** | 全系がアクティブ、負荷分散 | 即時（再配分） | 高 |

### 稼働率の計算

**直列システム**: A = A₁ × A₂ × ... × Aₙ
**並列システム**: A = 1 - (1-A₁) × (1-A₂) × ... × (1-Aₙ)

**MTBF（Mean Time Between Failures）**: 平均故障間隔
**MTTR（Mean Time To Repair）**: 平均修復時間
**稼働率 A = MTBF / (MTBF + MTTR)**

**計算例**:
- 単体稼働率 0.99 のサーバ2台の並列構成:
  - A = 1 - (1-0.99)² = 1 - 0.0001 = 0.9999（99.99%）
- 稼働率 0.99 のサーバと稼働率 0.95 のDBの直列構成:
  - A = 0.99 × 0.95 = 0.9405（94.05%）

**SLAの「ナイン」**:

| 稼働率 | 年間停止時間 |
|--------|-------------|
| 99%（ツーナイン） | 3.65日 |
| 99.9%（スリーナイン） | 8.76時間 |
| 99.99%（フォーナイン） | 52.6分 |
| 99.999%（ファイブナイン） | 5.26分 |

### RTO / RPO

- **RTO（Recovery Time Objective）**: 復旧目標時間 — 障害発生から復旧までの許容時間
- **RPO（Recovery Point Objective）**: 復旧目標地点 — データ損失の許容範囲（どこまで巻き戻せるか）
- RTOとRPOの値によってバックアップ・冗長化の方式が決まる

### よくある誤解
- 「フェールセーフとフェールソフトの混同」→ **停止が安全か、継続が安全か**で使い分ける
- 「冗長化すれば稼働率は必ず上がる」→ 共通原因故障（Common Cause Failure）は冗長化で防げない
- 「MTBFが大きければ良い」→ MTTR（復旧速度）も同等に重要。稼働率は両方の関数

### 実務活用
- SLA策定時にRTO/RPOを**ビジネスインパクト分析（BIA）**から逆算する
- 冗長化方式の選定は**コストと稼働率のトレードオフ**で判断
- 計算問題では「直列と並列の混在」に注意（ブロック図を描いて段階的に計算）

## 5. 性能設計

### スケーリング戦略

**スケールアップ（垂直拡張）**:
- CPU/メモリ/ストレージを増強する
- アプリケーション変更不要
- 物理的上限がある、SPOF（単一障害点）が残る
- 適用場面: RDBMS、レガシーアプリ、短期的な対処

**スケールアウト（水平拡張）**:
- 同等のノードを追加する
- 理論上は無限にスケール可能
- ステートレス設計・データ分散が前提
- 適用場面: Webサーバ、マイクロサービス、NoSQL

### キャッシュ戦略

**キャッシュ配置パターン**:

| パターン | 説明 | 用途 |
|----------|------|------|
| **Cache-Aside（Lazy Loading）** | アプリがキャッシュを明示的に管理（ミス時にDBから取得→キャッシュ保存） | 汎用 |
| **Read-Through** | キャッシュ自体がDB読み取りを代行 | 読み取り中心 |
| **Write-Through** | 書き込み時にキャッシュとDBを同時更新 | データ一貫性重視 |
| **Write-Behind（Write-Back）** | キャッシュに書き込み、非同期でDBに反映 | 書き込み性能重視 |
| **Write-Around** | 書き込みはDBのみ、読み取り時にキャッシュ | 書き込み後の即時読み取りが少ない |

**キャッシュの失効戦略**:
- **TTL（Time To Live）**: 時間ベースで失効
- **LRU（Least Recently Used）**: 最も長く使われていないエントリを削除
- **イベント駆動**: データ変更イベントでキャッシュを無効化
- **SWR（Stale While Revalidate）**: 期限切れデータを返しつつバックグラウンドで更新

### CDN（Content Delivery Network）
- エッジサーバにコンテンツをキャッシュし、ユーザーに近い地点から配信
- 静的コンテンツ（画像・CSS・JS）だけでなく、動的コンテンツのエッジキャッシュも活用
- キャッシュヒット率の最大化がコストと性能の両面で重要
- **オリジンシールド**: エッジ→オリジン間にシールド層を設けてオリジン負荷を軽減

### 非同期処理

**方式**:
- **メッセージキュー**: プロデューサーとコンシューマーを時間的にデカップリング
- **バッチ処理**: 大量データをまとめて処理
- **イベント駆動**: 非同期イベントで後続処理をトリガー
- **Pub/Sub**: 1対多の非同期配信

**設計上の考慮点**:
- 冪等性（Idempotency）の確保 — 同じメッセージの再処理で副作用がない
- 順序保証の要否 — FIFOキュー vs 順序なし
- DLQ（Dead Letter Queue）— 処理失敗メッセージの退避
- バックプレッシャー — コンシューマーの処理能力を超えない流量制御

### 性能見積り

**基本指標**:
- **レスポンスタイム**: リクエスト送信から応答受信までの時間
- **スループット**: 単位時間あたりの処理件数（TPS/RPS）
- **同時接続数**: 同時にシステムを利用するユーザー数
- **レイテンシ**: ネットワーク遅延時間

### リトルの法則（Little's Law）

**L = λ × W**

- **L**: システム内の平均リクエスト数（同時処理数）
- **λ**: 到着率（単位時間あたりのリクエスト数）
- **W**: 平均滞在時間（レスポンスタイム）

**計算例**:
- 1秒に10リクエスト到着（λ=10）、平均処理時間2秒（W=2）
- 同時にシステム内にあるリクエスト数: L = 10 × 2 = 20
- 必要なサーバ数 = L / サーバあたり同時処理数

### 待ち行列理論の応用

**M/M/1モデル（最も基本的な待ち行列）**:
- ポアソン到着、指数分布サービス時間、サーバ1台
- **利用率 ρ = λ/μ**（λ: 到着率、μ: サービス率）
- **平均待ち時間 Wq = ρ / (μ(1-ρ))**
- **ρが1に近づくと待ち時間が指数的に増大**する

**M/M/c モデル（サーバc台）**:
- 複数サーバによる負荷分散効果を定量評価
- Erlang C式で待ち確率を算出

**試験での出題パターン**:
- 「サーバの利用率が80%のとき、1台追加すると平均待ち時間はどう変わるか」
- 「ピーク時のスループット要件を満たすサーバ台数を求めよ」

### よくある誤解
- 「レスポンスタイム = 処理時間」→ ネットワーク遅延+キュー待ち時間+処理時間の合計
- 「スケールアウトすればリニアに性能向上」→ アムダールの法則により並列化できない部分が律速
- 「キャッシュは常に有効」→ キャッシュの無効化は「CSの2大難問」の1つ（Phil Karlton）

### 実務活用
- 性能要件はリトルの法則で**サーバ台数の初期見積り**を行い、負荷テストで検証する
- 待ち行列理論で**ρ≥0.7のシステムは増強を検討**する経験則
- キャッシュ戦略は「読み書き比率」と「データ鮮度要件」で選定する

## 6. セキュリティアーキテクチャ

### 認証・認可設計

**認証（Authentication）**: 「あなたは誰か」を確認する
- **知識要素**: パスワード、PIN
- **所持要素**: トークン、スマートカード、SMS
- **生体要素**: 指紋、顔認証、虹彩
- **多要素認証（MFA）**: 2つ以上の要素を組み合わせる

**認可（Authorization）**: 「何ができるか」を制御する
- **RBAC（Role-Based Access Control）**: ロール単位で権限を付与
- **ABAC（Attribute-Based Access Control）**: 属性の組み合わせで動的に判定
- **PBAC（Policy-Based Access Control）**: ポリシー記述言語で制御
- **最小権限の原則**: 必要最小限のアクセス権のみを付与

**認証プロトコル**:
- **OAuth 2.0**: 認可フレームワーク（アクセストークンの発行）
- **OpenID Connect**: OAuth 2.0上の認証レイヤー（IDトークン）
- **SAML**: XMLベースのフェデレーション認証（エンタープライズ向け）
- **FIDO2/WebAuthn**: パスワードレス認証

### ゼロトラストアーキテクチャ

**原則**: 「何も信頼しない、常に検証する（Never Trust, Always Verify）」

**NIST SP 800-207 の7つのテネット**:
1. すべてのデータソースとコンピューティングサービスはリソースとみなす
2. ネットワークの場所に関係なく、すべての通信は保護される
3. 個々のリソースへのアクセスはセッション単位で許可される
4. リソースへのアクセスは動的ポリシーによって決定される
5. すべての所有・関連デバイスのセキュリティ状態を監視・測定する
6. すべてのリソースの認証・認可はアクセス許可前に動的に厳密に実施される
7. 資産・ネットワークインフラ・通信の現状について可能な限り情報を収集し、セキュリティ改善に活用する

**実装要素**: ID管理（IdP）、デバイス信頼性評価、マイクロセグメンテーション、継続的監視

### マイクロセグメンテーション
- ネットワークを細かいセグメントに分割し、セグメント間の通信を制御
- 従来の境界防御（ペリメータセキュリティ）の限界を補完
- ワークロード単位でポリシーを適用（East-Westトラフィックの制御）
- SDN（Software-Defined Networking）やサービスメッシュで実装

### セキュアバイデザイン

**定義**: セキュリティをシステム開発の**最初の段階から組み込む**設計思想。

**原則**:
- **Attack Surface Minimization**: 攻撃対象面の最小化
- **Defense in Depth**: 多層防御
- **Principle of Least Privilege**: 最小権限
- **Secure Defaults**: デフォルトで安全な設定
- **Separation of Duties**: 職務分離
- **Economy of Mechanism**: メカニズムの簡素化

### 脅威モデリング

**目的**: システムに対する脅威を体系的に識別・評価し、対策を設計する。

**プロセス**:
1. システムのモデル化（データフロー図、信頼境界の識別）
2. 脅威の識別（STRIDEモデル等）
3. 脅威の評価（DREAD等でリスクスコアリング）
4. 対策の設計
5. 対策の検証

### STRIDE脅威モデル

| 脅威 | 意味 | 対策例 |
|------|------|--------|
| **S**poofing | なりすまし | 多要素認証、デジタル証明書 |
| **T**ampering | 改ざん | 完全性チェック、デジタル署名 |
| **R**epudiation | 否認 | 監査ログ、タイムスタンプ |
| **I**nformation Disclosure | 情報漏洩 | 暗号化、アクセス制御 |
| **D**enial of Service | サービス拒否 | レート制限、CDN、WAF |
| **E**levation of Privilege | 権限昇格 | 最小権限、入力バリデーション |

### よくある誤解
- 「ファイアウォールで囲めば安全」→ 内部脅威・横展開（Lateral Movement）に対応できない
- 「暗号化すれば安全」→ 鍵管理が脆弱なら暗号化は無意味
- 「ゼロトラスト = 製品導入」→ ゼロトラストは**設計思想**であり、特定製品ではない

### 実務活用
- 設計レビューでSTRIDEを**チェックリスト**として使い、脅威の見落としを防ぐ
- ゼロトラストの段階的導入: まずID管理の統合→次にマイクロセグメンテーション→継続的監視
- 脅威モデリングはスプリント0（設計フェーズ）で実施し、ADRに記録する

## 7. データベースアーキテクチャ

### 正規化 vs 非正規化

**正規化の目的**: データの冗長性を排除し、更新異常（挿入・更新・削除異常）を防ぐ。

| 正規形 | 条件 | 概要 |
|--------|------|------|
| 第1正規形（1NF） | 繰り返し項目の排除 | すべての属性が原子値 |
| 第2正規形（2NF） | 1NF + 部分関数従属の排除 | 主キーの一部への依存を分離 |
| 第3正規形（3NF） | 2NF + 推移的関数従属の排除 | 非キー→非キーの依存を分離 |
| BC正規形（BCNF） | すべての決定項が候補キー | 3NFより厳密 |

**非正規化の目的**: 読み取り性能の向上。JOINを減らすためにデータを意図的に冗長化する。

**トレードオフ**:
- 正規化: 更新整合性◎、ストレージ効率◎、読み取り性能△（JOIN多発）
- 非正規化: 読み取り性能◎、更新時の一貫性維持コスト高、ストレージ増大

### シャーディング（水平分割）

**定義**: データを複数のデータベースインスタンスに水平分割して格納する。

**シャーディングキーの選定基準**:
- カーディナリティが高い（十分な分散）
- クエリパターンに一致（クロスシャードクエリの最小化）
- ホットスポットが発生しにくい

**方式**:
- **ハッシュベース**: キーのハッシュ値でシャードを決定。均等分散。範囲クエリ不向き
- **レンジベース**: キーの範囲でシャードを決定。範囲クエリ向き。ホットスポットリスク
- **ディレクトリベース**: ルックアップテーブルで管理。柔軟だが管理コスト増

**課題**: クロスシャードJOIN、分散トランザクション、リバランシング

### レプリケーション

| 方式 | 説明 | 一貫性 | 性能 |
|------|------|--------|------|
| **同期レプリケーション** | 全レプリカ確認後にコミット | 強い | 遅い |
| **非同期レプリケーション** | マスター書き込み後に非同期で伝搬 | 結果整合 | 速い |
| **半同期レプリケーション** | 1台以上のレプリカ確認でコミット | 中間 | 中間 |

**トポロジ**: マスター-スレーブ、マスター-マスター、チェーン

### CQRS + イベントソーシング（DB観点）

**イベントソーシング**:
- 現在の状態ではなく**状態変更イベントの列**を永続化する
- 任意の時点の状態を再構築可能（イベントリプレイ）
- 監査証跡が自動的に記録される
- スナップショットで再構築コストを軽減

**CQRSとの組み合わせ**:
- イベントストア（Command側）→ Projection → ReadモデルDB（Query側）
- Readモデルはユースケースに最適化した非正規化ビュー

### ポリグロットパーシステンス

**定義**: 用途に応じて**複数の異なるデータストア技術**を使い分ける戦略。

| データ種別 | 適切なストア | 理由 |
|-----------|-------------|------|
| トランザクションデータ | RDBMS | ACID保証、JOIN |
| セッションデータ | Redis等KVS | 高速読み書き、TTL |
| 全文検索 | Elasticsearch | 転置インデックス |
| グラフ関係 | Neo4j等 | 関係性の高速探索 |
| 時系列データ | InfluxDB等 | 時間軸最適化 |
| 文書データ | MongoDB等 | スキーマ柔軟性 |

### CAP定理

分散データストアにおいて、以下の3つを同時に満たすことは不可能:
- **C**onsistency（一貫性）: 全ノードが同じデータを返す
- **A**vailability（可用性）: 全リクエストに応答を返す
- **P**artition tolerance（分断耐性）: ネットワーク分断時も動作する

ネットワーク分断は避けられないため、実質的には**CP**（一貫性優先）か**AP**（可用性優先）の選択となる。

### よくある誤解
- 「正規化は常に正しい」→ 読み取り中心のOLAPでは非正規化が有利
- 「NoSQL = スキーマレス = 設計不要」→ データモデリングはRDBMS以上に重要
- 「シャーディングは早期に導入すべき」→ **可能な限り遅らせる**（複雑性のコスト）
- 「CAP定理のCはACIDのCと同じ」→ 異なる概念（ACIDのCは整合性制約、CAPのCは線形化可能性）

### 実務活用
- OLTP用途は正規化、OLAP用途は非正規化の**使い分け**を設計書に明記する
- ポリグロットパーシステンスを採用する際は**運用コスト（学習・監視・バックアップ）**を評価
- シャーディングの前に「リードレプリカ」「キャッシュ」「クエリ最適化」を検討する

## 8. ネットワークアーキテクチャ

### トポロジ設計

| トポロジ | 特徴 | 用途 |
|----------|------|------|
| **スター型** | 中央集中、管理容易、SPOF | LAN（スイッチ中心） |
| **リング型** | 双方向冗長可、遅延均一 | トークンリング（歴史的） |
| **メッシュ型** | 高冗長性、コスト高 | WAN、コアネットワーク |
| **ツリー型** | 階層的、スケーラブル | 大規模LAN |
| **スパイン-リーフ** | 低遅延、均等帯域 | データセンター |

**スパイン-リーフアーキテクチャ**:
- スパイン（コア）スイッチとリーフ（アクセス）スイッチの2層構成
- 任意のリーフ間が最大2ホップ — 低遅延・予測可能な性能
- East-Westトラフィック（サーバ間通信）が多いDC環境に最適

### 帯域設計

**帯域計算の基本**:
- **必要帯域 = ピークトラフィック × オーバーヘッド係数 × 将来成長率**
- オーバーヘッド: プロトコルヘッダ（TCP/IP: 約3〜5%）、再送、制御トラフィック
- 帯域利用率の目標: 通常70%以下（バースト吸収のマージン）

**QoS（Quality of Service）**:
- トラフィックの優先制御（音声 > 動画 > データ）
- DiffServ（Differentiated Services）: DSCPフィールドでクラス分け
- トラフィックシェーピング vs ポリシング

### セグメンテーション

**VLAN（Virtual LAN）**: 物理構成に依存せず論理的にネットワークを分割
**サブネット設計**: IPアドレス空間の効率的な割り当て（CIDR）
**DMZ**: 外部公開サーバを内部ネットワークから隔離
**マイクロセグメンテーション**: ワークロード単位の細粒度制御（ゼロトラストの実装手段）

### SDN（Software-Defined Networking）

**定義**: ネットワークの制御プレーン（経路決定）とデータプレーン（パケット転送）を分離し、ソフトウェアで集中制御する。

**構成要素**:
- **コントローラ**: ネットワーク全体の制御ロジック（OpenDaylight, ONOS等）
- **サウスバウンドIF**: コントローラ→スイッチ間（OpenFlow等）
- **ノースバウンドIF**: アプリケーション→コントローラ間（REST API等）

**利点**: ネットワーク構成変更の自動化、一元管理、プログラマブル
**課題**: コントローラのSPOF、レガシー機器との互換性

### サービスメッシュ

**定義**: マイクロサービス間の通信をインフラストラクチャ層で透過的に管理する。

**構成要素**:
- **データプレーン**: 各サービスに付随するサイドカープロキシ（Envoy等）
- **コントロールプレーン**: プロキシの設定を集中管理（Istio等）

**提供機能**: サービス検出、負荷分散、mTLS（相互TLS）、リトライ・サーキットブレーカー、可観測性（メトリクス・トレーシング・ロギング）

### よくある誤解
- 「VLANだけでセキュリティは十分」→ VLAN間のACLだけでは内部脅威に対応できない
- 「SDNを入れればすべて自動化」→ 既存ネットワークとの統合・運用スキルの移行コストが大きい
- 「サービスメッシュはマイクロサービスに必須」→ サービス数が少ない段階ではオーバーヘッドが過剰

### 実務活用
- データセンター新規構築時はスパイン-リーフ構成を第一候補にする
- 帯域設計は**3〜5年の成長予測**を織り込む
- サービスメッシュ導入はサービス数10以上を目安に検討する

## 9. クラウドアーキテクチャ

### サービスモデルの選定

| モデル | 管理範囲 | 適用場面 |
|--------|----------|----------|
| **IaaS** | OS以上をユーザー管理 | レガシー移行、カスタムインフラ |
| **PaaS** | アプリとデータのみ管理 | 新規開発、運用負荷軽減 |
| **SaaS** | ユーザーは利用のみ | 標準業務（メール、CRM等） |
| **FaaS** | 関数単位の実行 | イベント駆動処理、API |
| **CaaS** | コンテナ単位の管理 | マイクロサービス、CI/CD |

**選定の判断軸**:
- カスタマイズ要件の度合い
- 運用チームのスキル・規模
- コスト構造（CapEx vs OpEx）
- コンプライアンス・データ主権要件

### マルチクラウド

**定義**: 複数のクラウドプロバイダーのサービスを組み合わせて使用する。

**メリット**: ベンダーロックイン回避、ベストオブブリード、障害分散
**デメリット**: 運用複雑性の増大、スキル分散、データ転送コスト、統合的なセキュリティ管理の困難

**実用的なアプローチ**:
- プライマリクラウド + セカンダリクラウド（DR用）
- 特定のサービスだけ別クラウド（例: AI/MLはGCP、基幹はAWS）
- 抽象化レイヤー（Terraform、Kubernetes）でポータビリティを確保

### ハイブリッドクラウド

**定義**: オンプレミスとクラウドを組み合わせた環境。

**適用場面**:
- データ主権・コンプライアンスで一部データをオンプレミスに保持する必要
- レガシーシステムの段階的移行（リフト&シフト→リファクタ→リビルド）
- バースト対応（通常はオンプレ、ピーク時はクラウドにスケールアウト）

**接続方式**: VPN、専用線（Direct Connect/ExpressRoute）、SD-WAN

### Well-Architectedフレームワーク

AWS Well-Architected Frameworkを代表例として、主要クラウドプロバイダーが類似のフレームワークを提供。

**6つの柱**（AWS）:

| 柱 | 内容 |
|----|------|
| **運用上の優秀性** | 自動化、IaC、継続的改善 |
| **セキュリティ** | ゼロトラスト、暗号化、最小権限 |
| **信頼性** | 障害復旧、スケーリング、変更管理 |
| **パフォーマンス効率** | リソースの最適な選択と最適化 |
| **コスト最適化** | 無駄の排除、適正なサイジング |
| **サステナビリティ** | 環境負荷の最小化 |

### コスト最適化

**戦略**:
- **ライトサイジング**: 実使用量に基づくインスタンスタイプの最適化
- **リザーブドインスタンス/Savings Plans**: 長期コミットメントによる割引（30〜70%）
- **スポットインスタンス**: 余剰キャパシティの格安利用（中断リスクあり）
- **サーバレス**: アイドル時のコストゼロ
- **ストレージ階層化**: ホット→ウォーム→コールド→アーカイブ
- **データ転送最適化**: リージョン内通信の最大化、CDN活用

### よくある誤解
- 「クラウドは常にオンプレミスより安い」→ 長期運用・安定負荷ではオンプレミスが有利な場合がある
- 「マルチクラウドで可用性が上がる」→ 運用の複雑化が障害リスクを増加させる可能性
- 「PaaSなら運用不要」→ アプリケーションレベルの運用・監視は依然として必要

### 実務活用
- クラウド選定は**TCO（Total Cost of Ownership）**で5年計算して判断する
- Well-Architectedレビューを四半期ごとに実施する
- 月次でコストレポートを確認し、未使用リソースを自動停止するガバナンスを構築する

## 10. アプリケーションアーキテクチャ

### DDD（Domain-Driven Design）

**戦略的設計**:
- **ユビキタス言語**: ドメインエキスパートと開発者が共有する用語体系
- **境界づけられたコンテキスト（Bounded Context）**: モデルの適用範囲を明確にする境界
- **コンテキストマップ**: 境界づけられたコンテキスト間の関係を可視化
  - Shared Kernel / Customer-Supplier / Conformist / Anticorruption Layer / Open Host Service / Published Language

**戦術的設計**:
- **エンティティ**: 同一性（ID）を持つオブジェクト
- **値オブジェクト**: 同一性を持たず、属性値で等価判定
- **集約（Aggregate）**: 整合性の境界。集約ルートを通じてのみアクセス
- **リポジトリ**: 集約の永続化を抽象化
- **ドメインサービス**: エンティティや値オブジェクトに属さないドメインロジック
- **ドメインイベント**: ドメインで起きた出来事を表現

### クリーンアーキテクチャ

```
[外側] フレームワーク・ドライバ
  ↓
[  ] インターフェースアダプター（Controller, Presenter, Gateway）
  ↓
[  ] アプリケーションビジネスルール（ユースケース）
  ↓
[内側] エンタープライズビジネスルール（エンティティ）
```

**依存性ルール**: 依存の方向は常に**外側から内側**へ。内側は外側を知らない。

**同心円の各層**:
1. **エンティティ層**: ビジネスルールのコア（最も変わりにくい）
2. **ユースケース層**: アプリケーション固有のビジネスルール
3. **インターフェースアダプター層**: データ変換（DTO⇔エンティティ）
4. **フレームワーク・ドライバ層**: DB、Web、UI等の外部ツール

### 12-Factor App

クラウドネイティブアプリケーションの設計原則（Heroku発祥）。

| # | 原則 | 内容 |
|---|------|------|
| 1 | **Codebase** | 1アプリ = 1リポジトリ（複数デプロイ） |
| 2 | **Dependencies** | 明示的な依存関係宣言・分離 |
| 3 | **Config** | 環境変数で設定を注入 |
| 4 | **Backing services** | アタッチされたリソースとして扱う |
| 5 | **Build, release, run** | ビルド・リリース・実行を厳密に分離 |
| 6 | **Processes** | ステートレスなプロセスとして実行 |
| 7 | **Port binding** | ポートバインドでサービスを公開 |
| 8 | **Concurrency** | プロセスモデルによるスケールアウト |
| 9 | **Disposability** | 高速な起動とグレースフルシャットダウン |
| 10 | **Dev/prod parity** | 開発・ステージング・本番の差異を最小化 |
| 11 | **Logs** | ログをイベントストリームとして扱う |
| 12 | **Admin processes** | 管理タスクを1回限りのプロセスとして実行 |

### API設計

**REST（Representational State Transfer）**:
- リソース指向（URI = リソースの識別子）
- HTTPメソッドで操作（GET/POST/PUT/PATCH/DELETE）
- ステートレス、キャッシュ可能
- HATEOAS（Hypermedia As The Engine Of Application State）

**gRPC**:
- Protocol Buffersによるスキーマ定義（.proto）
- HTTP/2ベース、バイナリフォーマット（高速）
- 双方向ストリーミング対応
- サービス間通信（内部API）に適する

**GraphQL**:
- クライアントが必要なデータの構造を指定
- オーバーフェッチ・アンダーフェッチの解消
- 単一エンドポイント
- N+1問題に注意（DataLoaderで対策）

**API設計の原則**:
- **バージョニング**: URIパス（/v1/）またはヘッダー（Accept-Version）
- **ページネーション**: カーソルベース（大規模データ向き）、オフセットベース（小規模向き）
- **エラーレスポンス**: RFC 7807（Problem Details for HTTP APIs）
- **冪等性**: PUT/DELETEは冪等、POSTは非冪等（冪等キーで対応）
- **レート制限**: Token Bucket、Sliding Window

### メッセージング

**パターン**:
- **Point-to-Point**: 1つのメッセージを1つのコンシューマーが処理
- **Pub/Sub**: 1つのメッセージを複数のサブスクライバーに配信
- **Request-Reply**: 非同期のリクエスト-レスポンス

**配信保証**:
- **At-Most-Once**: 最大1回（メッセージ喪失あり）
- **At-Least-Once**: 最低1回（重複あり→冪等性が必須）
- **Exactly-Once**: 正確に1回（実装コスト最大、多くの場合At-Least-Once+冪等性で代替）

**代表的プロダクト**: Apache Kafka（ログベース）、RabbitMQ（AMQP）、Amazon SQS（マネージド）

### よくある誤解
- 「DDDは設計パターンの集合」→ DDDの本質は**ドメインの理解**であり、パターンはその手段
- 「クリーンアーキテクチャは4層固定」→ 層の数は自由。依存の方向ルールが本質
- 「RESTは常にgRPCより優れている」→ **用途に応じた使い分け**が重要（外部API→REST、内部→gRPC）
- 「12-Factorを全部守らなければならない」→ 段階的に適用し、特にConfig/Stateless/Disposabilityを優先

### 実務活用
- DDDの境界づけられたコンテキストはマイクロサービスの**分割単位**と整合させる
- クリーンアーキテクチャの依存性ルールをコードレビューの**自動チェック**に組み込む
- API設計は**契約先行（Contract-First）**で進め、OpenAPI仕様書から自動生成する

## 11. 移行・統合

### レガシー移行戦略

| 戦略 | 説明 | リスク | コスト |
|------|------|--------|--------|
| **リホスト（Lift & Shift）** | そのままクラウドに移行 | 低 | 低 |
| **リプラットフォーム（Lift & Reshape）** | 一部最適化して移行 | 低〜中 | 中 |
| **リファクタ** | コード構造を改善して移行 | 中 | 中〜高 |
| **リアーキテクト/リビルド** | 新アーキテクチャで再構築 | 高 | 高 |
| **リプレース** | パッケージ/SaaSに置換 | 中 | 中 |
| **リテイン** | 移行しない（現状維持） | — | 低 |
| **リタイア** | 廃止 | — | — |

### ストラングラーパターン（Strangler Fig Pattern）

**定義**: レガシーシステムを段階的に新システムに置き換える移行パターン。

```
[クライアント]
     ↓
[ファサード/ルーター]
   ↙        ↘
[新システム]  [レガシー]
(機能A)      (機能B,C)
```

**プロセス**:
1. ファサード（APIゲートウェイ/リバースプロキシ）を前段に配置
2. 新機能は新システムに実装し、ファサードでルーティング
3. 既存機能を1つずつ新システムに移行
4. 全機能の移行が完了したらレガシーを廃止

**利点**: ビッグバンリスクの回避、段階的な検証、ロールバック容易
**注意点**: 移行期間中は2つのシステムの並行運用コストが発生

### データ移行

**方式**:
- **ビッグバン移行**: 停止時間を設けて一括移行。シンプルだが停止時間が必要
- **段階移行**: データをカテゴリごとに分割して段階的に移行
- **並行運用**: 新旧システムを並行稼働し、徐々に切り替え
- **トリクル移行**: 少しずつ継続的にデータを同期（CDC: Change Data Capture）

**検証**:
- **件数照合**: 移行元と移行先のレコード数一致確認
- **値照合**: サンプリングまたは全件のデータ内容比較
- **整合性チェック**: 外部キー制約、一意性制約の検証
- **業務照合**: 移行後のシステムで業務シナリオを実行して結果を確認

### ETL / ELT

**ETL（Extract → Transform → Load）**:
- データを抽出し、変換してからDWH/データレイクにロード
- 変換処理はETLサーバで実行
- 従来型DWHに適する

**ELT（Extract → Load → Transform）**:
- データをそのままロードし、DWH/データレイク上で変換
- DWHの計算能力を活用（BigQuery、Snowflake等）
- クラウドDWH時代の主流

### API統合・EAI（Enterprise Application Integration）

**統合パターン**:
- **ポイントツーポイント**: システム間を直接接続（N×(N-1)/2の接続数、スパゲッティ化リスク）
- **ハブ&スポーク**: 中央のハブ（ESB等）を介して接続（ハブがSPOF・ボトルネック）
- **メッセージバス**: 共通バスにパブリッシュ/サブスクライブ（疎結合）
- **APIゲートウェイ**: 外部API公開の一元管理（認証・レート制限・変換）

**Enterprise Integration Patterns（EIP）**:
- メッセージルーター（Content-Based Router、Message Filter等）
- メッセージトランスフォーマー（Content Enricher、Claim Check等）
- メッセージエンドポイント（Competing Consumers、Idempotent Receiver等）

### よくある誤解
- 「ビッグバン移行は悪」→ 小規模で依存関係が少ないシステムでは最も効率的
- 「ストラングラーパターンで失敗しない」→ ファサードの設計ミスで新旧システムの整合性が崩れるリスク
- 「ETLは古い、ELTが正しい」→ データ量・変換の複雑性・DWHの計算能力で使い分ける

### 実務活用
- 移行計画書にはロールバック手順と判断基準を**必ず**含める
- データ移行の検証は自動化し、**リハーサルを3回以上**実施する
- ストラングラーパターンのファサードは**カナリアリリース**と組み合わせて段階的に切り替える

## 12. テスト戦略

### テストピラミッド

```
        /  E2E  \        ← 少数、遅い、高コスト
       / 統合テスト \     ← 中間
      / ユニットテスト \  ← 大量、高速、低コスト
```

**各層の役割**:
- **ユニットテスト**: 関数・メソッド単位の検証。高速・安定。外部依存はモック化
- **統合テスト**: コンポーネント間の連携検証。DB・API等の実際の接続を含む
- **E2Eテスト**: ユーザー操作シナリオの検証。ブラウザ自動化等

**アンチパターン — アイスクリームコーン型**:
- E2Eテストが多すぎ、ユニットテストが少ない
- テスト実行が遅く、不安定で、原因特定が困難

### 非機能テスト

| テスト種別 | 目的 | 代表的ツール |
|-----------|------|-------------|
| **性能テスト** | レスポンスタイム・スループットの測定 | JMeter, Gatling, k6 |
| **負荷テスト** | 想定ピーク負荷での動作確認 | 同上 |
| **ストレステスト** | 限界を超えた負荷での挙動確認 | 同上 |
| **耐久テスト（ソークテスト）** | 長時間運転でのメモリリーク等 | 同上 |
| **セキュリティテスト** | 脆弱性の検出 | OWASP ZAP, Burp Suite |
| **ユーザビリティテスト** | 使いやすさの評価 | ユーザーテスト |

### 負荷テスト

**テストシナリオの設計**:
1. **対象トランザクションの特定**: ビジネス上重要な操作、高頻度の操作
2. **負荷モデルの作成**: ユーザー数のランプアップ、定常負荷、ピーク負荷
3. **目標値の設定**: レスポンスタイム95パーセンタイル値、エラー率
4. **実行環境**: 本番同等環境が理想（少なくとも縮小版で比例計算）

**分析のポイント**:
- レスポンスタイムの**パーセンタイル値**（平均値だけでは外れ値が隠れる）
- **スループットの飽和点**: 負荷増加に対してスループットが頭打ちになるポイント
- **ボトルネック特定**: CPU/メモリ/ディスクI/O/ネットワーク/DB接続プールのどこが律速か

### カオスエンジニアリング

**定義**: 本番環境で意図的に障害を注入し、システムの耐障害性を検証する手法。

**原則（Principles of Chaos Engineering）**:
1. 定常状態の仮説を立てる
2. 実世界のイベントを反映した変数を多様化する
3. 本番環境で実験を行う
4. 継続的に実験を自動化する
5. 影響範囲を最小化する

**障害注入の例**:
- サーバの強制終了、ネットワーク遅延の挿入、DNS障害のシミュレーション
- 依存サービスのレスポンス遅延、ディスク障害、時刻ずれ

**代表的ツール**: Netflix Chaos Monkey, Gremlin, AWS Fault Injection Simulator, Litmus（Kubernetes）

### テスト自動化戦略

**自動化すべきテスト**:
- 繰り返し実行するリグレッションテスト
- CI/CDパイプラインに組み込むテスト
- 環境ごとのスモークテスト

**自動化すべきでないテスト**:
- 探索的テスト
- ユーザビリティテスト
- 1回限りの確認テスト

**CI/CDパイプラインへの組み込み**:
```
Commit → Lint → Unit Test → Build → Integration Test → Deploy(STG) → E2E Test → Deploy(PRD)
```

### よくある誤解
- 「テストカバレッジ100% = バグなし」→ カバレッジは**テストの不足**を示す指標であり、品質の保証ではない
- 「E2Eテストを増やせば品質が上がる」→ テストピラミッドの原則に従い、ユニットテストを厚くする
- 「カオスエンジニアリングは本番を壊す行為」→ **仮説に基づく科学的実験**であり、影響範囲を制御する
- 「性能テストは開発完了後に1回」→ **継続的に**パイプラインに組み込む

### 実務活用
- テスト戦略書に**テストピラミッドの各層の比率と役割**を明記する
- 非機能テストの目標値は**非機能要件（SLA/SLO）から逆算**して設定する
- カオスエンジニアリングは**ゲームデイ（全員参加の障害訓練）**から始める

## 13. 午後I（科目B-1）対策

### アーキテクチャ選定理由の記述

**解答の型**:
「[要件]を満たすために、[アーキテクチャ/技術]を採用した。その理由は、[特性]により[メリット]が得られ、[デメリット]は[緩和策]で対処できるためである。」

**具体例**:
- 「将来の利用者数増加に対応するため、マイクロサービスアーキテクチャを採用した。サービス単位で独立にスケールアウトでき、特定のサービスのみ増強可能なためである。」
- 「データの一貫性が業務上必須であるため、RDBMSの同期レプリケーションを採用した。非同期方式はレイテンシが低いが、データ損失のリスクがあり、金融取引では許容できないためである。」

### トレードオフ分析の記述

**午後Iで頻出するトレードオフ**:

| 対立軸 | 選択肢A | 選択肢B | 判断基準 |
|--------|---------|---------|----------|
| 一貫性 vs 可用性 | 同期処理 | 非同期処理 | データ損失の許容度 |
| 性能 vs 保守性 | 非正規化 | 正規化 | 読み書き比率 |
| 柔軟性 vs 複雑性 | マイクロサービス | モノリス | チーム規模・デプロイ頻度 |
| コスト vs 信頼性 | シングル構成 | 冗長構成 | SLA要件・業務損失 |
| 開発速度 vs 品質 | 既製品/SaaS | スクラッチ開発 | カスタマイズ要件 |

**記述のコツ**: 「Aを選んだ理由」だけでなく、「Bを選ばなかった理由」も1文入れると得点が上がる。

### 非機能要件への回答パターン

**可用性の問い**:
- 「障害時にもサービスを継続するための方式を述べよ」
- → 冗長構成（Active-Standby/Active-Active）+ フェイルオーバー手順 + RTO/RPOとの整合性

**性能の問い**:
- 「レスポンスタイムの要件を満たすための設計を述べよ」
- → キャッシュ導入 + 非同期処理 + DB最適化（インデックス、クエリ改善）+ スケールアウト

**セキュリティの問い**:
- 「機密データを扱うシステムのセキュリティ対策を述べよ」
- → 認証・認可設計 + 暗号化（通信・保存） + 監査ログ + アクセス制御

**移行性の問い**:
- 「既存システムからの移行方式と、リスク軽減策を述べよ」
- → 段階移行 + 並行運用 + データ照合 + ロールバック手順

### 解答テクニック

1. **問題文をマーキング**: 固有名詞、数値、制約条件に下線を引く
2. **設問を先に読む**: 何が問われているかを把握してから本文を読む
3. **問題文中の根拠を探す**: 解答は問題文中にヒントがある（創造的な答えは求められていない）
4. **字数の目安**: 設問の配点に比例した字数で回答する
5. **45分ルール**: 1問45分を厳守。迷ったら次の問題に移る
6. **「なぜ」を意識**: 技術名だけでなく「その技術を選んだ理由」を必ず付記

### よくある失点パターン
- 問題文の条件を無視して一般論で回答する
- 技術用語の羅列で具体的な説明がない
- トレードオフの片面しか述べない
- 字数が少なすぎる（解答欄の半分以下）
- 時間配分ミスで最後の設問が白紙

## 14. 午後II（科目B-2）論述対策

### 論文構成

**設問ア（800字以内）**: システム概要と背景
- 業種、システムの目的、規模（ユーザー数・データ量・チーム人数）
- 自身の役割（システムアーキテクトとしての立場を明示）
- **テンプレート化**しておき、問題に応じて微修正する

**設問イ（800〜1,600字）**: 本論（設問の中核）
- 課題の分析と解決策の設計
- **具体的な技術選定とその根拠**を述べる
- 図表は使えないため、箇条書きや構造的な記述で明確にする

**設問ウ（600〜1,200字）**: 評価と反省
- 設計・実装の結果をどう評価したか
- 改善点や今後の課題
- **客観的な評価基準**（レビュー結果、テスト結果、性能測定値等）を含める

### 技術者視点の書き方

**評価される記述**:
- 「要件Xを満たすために方式Aと方式Bを比較検討し、[具体的理由]からAを採用した」
- 「性能要件のレスポンスタイム3秒以内を達成するため、[具体的な施策]を実施した」
- 「リスクとして[具体的リスク]を想定し、[緩和策]を設計に組み込んだ」

**評価されない記述**:
- 「適切に設計した」「最適な方式を採用した」（何が適切か不明）
- 「チームで協力して対応した」（具体的なアーキテクトとしての貢献が不明）
- 「うまくいった」（評価基準が不明）

### 合格答案パターン

**パターン1: 技術選定型（最頻出）**
1. 複数の候補技術を列挙
2. 評価基準を明示（非機能要件から導出）
3. 各候補を評価基準で比較
4. 選定結果と根拠を述べる
5. 結果の評価と改善点

**パターン2: 課題解決型**
1. 現行システムの課題を分析
2. 課題の原因を特定
3. 解決策の設計（アーキテクチャの変更）
4. 実施結果と効果測定
5. 残課題と今後の方針

**パターン3: 移行・刷新型**
1. 現行システムの制約と移行の動機
2. 移行方式の検討（ビッグバン vs 段階 vs ストラングラー）
3. リスク分析と緩和策
4. 移行の実施と検証
5. 移行後の評価

### 図表の活用
- 午後IIは**手書き論述**のため図表は基本的に使えない
- ただし**箇条書き・番号付きリスト・簡易な表形式**は有効
- 「以下に示す3つの方式を比較検討した。(1)方式A: ... (2)方式B: ... (3)方式C: ...」のように構造化する

### 時間配分

| フェーズ | 時間 | 内容 |
|---------|------|------|
| 問題選択・構想 | 15分 | 2問を読み、書けそうな方を選択。章立てメモ |
| 設問ア | 20分 | システム概要（テンプレートベース） |
| 設問イ | 50分 | 本論（最も配点が高い） |
| 設問ウ | 25分 | 評価と反省 |
| 見直し | 10分 | 字数確認・誤字修正 |

### ネタの準備（事前テンプレート）

**業界×テーマのマトリクスで5〜6パターン用意しておく**:
- 製造業 × 基幹システム刷新
- 金融業 × セキュリティ基盤設計
- 小売業 × EC基盤のスケーラビリティ
- 公共 × レガシー移行
- サービス業 × DX推進

**各パターンに以下を準備**:
- システム概要（規模・チーム体制・開発期間）
- 使用した技術スタック
- 直面した課題とその解決策
- 評価指標と結果

### よくある不合格パターン
- **設問の要求に答えていない**: 「アーキテクチャの選定理由」を問われているのに実装詳細を書く
- **具体性がない**: 抽象的な記述に終始し、数値や技術名が出てこない
- **規模感の矛盾**: 「10名のチームで3ヶ月」と書いた後に大企業レベルの施策を述べる
- **字数不足**: 設問イが800字未満だと採点対象外の可能性
- **時間切れ**: 設問ウが書ききれない（最も多い失敗原因）

## 15. 実務活用パターン

### アーキテクチャ設計での活用

**体系的な設計プロセス**:
1. **品質特性シナリオの作成**: 非機能要件を「刺激→環境→応答→応答測定値」で定量化
2. **アーキテクチャドライバーの特定**: 設計に最も影響を与える要件を絞り込む
3. **アーキテクチャスタイルの選定**: ドライバーに基づいて候補を列挙・評価
4. **ADR（Architecture Decision Record）**で判断を記録

**ADRのテンプレート**:
```
# ADR-001: [タイトル]
## ステータス: 提案中 / 承認 / 却下 / 廃止
## コンテキスト: [決定が必要な背景]
## 決定: [選択した方式]
## 根拠: [選定理由、比較検討結果]
## 結果: [決定の影響、トレードオフ]
```

### 技術選定での活用

**評価フレームワーク**:

| 評価軸 | 重み | 候補A | 候補B | 候補C |
|--------|------|-------|-------|-------|
| 機能適合性 | 0.3 | ◎ | ○ | △ |
| 性能 | 0.2 | ○ | ◎ | ○ |
| 運用コスト | 0.2 | △ | ○ | ◎ |
| 学習コスト | 0.15 | ○ | △ | ◎ |
| コミュニティ | 0.15 | ◎ | ○ | ○ |

- 重み付けは**プロジェクトの非機能要件**から導出する
- PoC（Proof of Concept）で**定量的な比較データ**を取得する
- 選定結果はADRに記録し、将来の見直し時に参照可能にする

### アーキテクチャレビューでの活用

**ATAM（Architecture Tradeoff Analysis Method）**:
1. アーキテクチャの提示
2. ビジネスドライバーの特定
3. アーキテクチャアプローチの特定
4. 品質特性ユーティリティツリーの作成
5. アーキテクチャアプローチの分析
6. ブレインストーミング・シナリオ優先順位付け
7. アーキテクチャアプローチの再分析
8. 結果の提示

**レビュー観点**:
- **感度点（Sensitivity Point）**: 品質特性に大きな影響を与えるアーキテクチャ判断
- **トレードオフ点（Tradeoff Point）**: 複数の品質特性に相反する影響を与えるポイント
- **リスク（Risk）**: まだ解決されていないアーキテクチャ上の問題
- **非リスク（Non-risk）**: 分析の結果、問題ないと判断されたポイント

### 日常業務での活用

**設計書レビュー時のチェックリスト**:
- [ ] 非機能要求グレード6カテゴリの考慮
- [ ] トレードオフの明示的な文書化
- [ ] 障害シナリオと復旧手順
- [ ] スケーラビリティ戦略（現在と3年後）
- [ ] セキュリティ脅威モデリング
- [ ] データ移行方式とロールバック手順
- [ ] テスト戦略（テストピラミッド準拠）
- [ ] 運用・監視の設計

**チーム育成**:
- SA試験の知識体系をチーム内の**共通言語**として活用する
- ADRの文化を根付かせ、アーキテクチャ判断の**暗黙知を形式知**に変換する
- 定期的なアーキテクチャレビュー会で**トレードオフ分析の訓練**を行う
