# 午前試験（80問）— FEからの深化ポイント

## テクノロジ系（問1-50、50問）

### T1. 基礎理論の深化
FEでは公式の適用で済んだ計算問題が、導出過程や応用が問われる。

- **待ち行列理論（M/M/1モデル）**
  - 到着率λ、サービス率μ、利用率ρ=λ/μ（ρ<1が安定条件）
  - **平均待ち時間 Wq = ρ/(μ(1-ρ))**
  - **平均系内時間 W = 1/(μ(1-ρ))**
  - **平均待ち行列長 Lq = ρ²/(1-ρ)**
  - **原理**: ポアソン過程（到着）+ 指数分布（サービス）の確率過程。ρが1に近づくと待ち時間が急激に増大（非線形）
  - **実務活用**: Webサーバの性能見積もり、キュー設計

- **BNF記法・正規表現**
  - BNF: 文法を形式的に定義（構文解析の基礎）
  - 正規表現: パターンマッチング（*,+,?,|,[],^,$）
  - **オートマトン**: 有限オートマトン(DFA/NFA)と正規言語の等価性

- **計算量理論の深化**
  - P問題: 多項式時間で解ける / NP問題: 多項式時間で検証できる
  - NP完全: 全てのNP問題から多項式変換可能（巡回セールスマン/充足可能性）
  - **原理**: P≠NPは未解決問題。NP完全問題は現実的な時間で最適解を求められないため近似アルゴリズムやヒューリスティクスを使う

### T2. コンピュータアーキテクチャの深化

- **パイプラインの性能計算**
  - k段パイプラインでn命令を実行: k+(n-1)クロック
  - **スピードアップ率**: (k×n)/(k+n-1) → nが十分大きいとkに収束
  - **パイプラインハザード対策**: フォワーディング(データ)、分岐予測(制御)、ストール
  - **よくある誤解**: パイプライン化でCPIが1になるとは限らない（ハザードでストールが入る）

- **マルチプロセッサ**
  - 密結合（共有メモリ: UMA/NUMA）/ 疎結合（分散メモリ: MPI）
  - **アムダールの法則**: S = 1/(F + (1-F)/P)
    - F=並列化不可能な割合、P=プロセッサ数
    - 例: F=0.1, P=∞ → S=10（10%が直列なら最大10倍）
  - **実務活用**: マイクロサービスのスケーラビリティ限界を予測

- **仮想化技術**
  - ハイパーバイザ型（Type1: ベアメタル / Type2: ホスト型）
  - コンテナ型（Docker）: OSカーネル共有、軽量・高速起動
  - **原理**: 特権命令のトラップ→VMM（仮想マシンモニタ）がエミュレート

### T3. データベースの深化

- **SQL高度技法**
  - **相関副問合せ**: 外側クエリの行ごとに内側クエリが実行される
    ```sql
    SELECT e.name FROM emp e
    WHERE e.salary > (SELECT AVG(salary) FROM emp WHERE dept = e.dept)
    ```
  - **EXISTS**: 存在確認（INより高速な場合が多い）
  - **CASE式**: SELECT内での条件分岐
  - **集合演算**: UNION/INTERSECT/EXCEPT
  - **ウィンドウ関数**: ROW_NUMBER()/RANK()/SUM() OVER (PARTITION BY...)

- **ER図→関係スキーマ変換**（午後頻出）
  - 1:1関連 → どちらかのテーブルに外部キー追加
  - 1:N関連 → N側のテーブルに外部キー追加
  - M:N関連 → **中間テーブル**（連関エンティティ）を作成
  - **スーパータイプ/サブタイプ**: 共通属性を親テーブル、固有属性を子テーブルに分離

- **性能チューニング**
  - インデックス: B+木（範囲検索）/ ハッシュ（等価検索）
  - 実行計画: フルスキャン vs インデックススキャン vs ネステッドループ結合 vs ハッシュ結合
  - **原理**: オプティマイザがコストベースで最適な実行計画を選択

### T4. ネットワークの深化

- **VPN・トンネリング**
  - IPsec: ネットワーク層のVPN（トランスポートモード/トンネルモード）
  - SSL-VPN: アプリケーション層のVPN（ブラウザベース）
  - **原理**: 暗号化+認証+完全性で仮想的な専用線を構築

- **VLAN**
  - ポートベースVLAN / タグVLAN(IEEE 802.1Q)
  - **トランクポート**: 複数VLANのフレームを1本のリンクで転送（タグ付き）
  - **原理**: ブロードキャストドメインの分割→セキュリティ+トラフィック制御

- **負荷分散・冗長化**
  - L4負荷分散（TCP/IPレベル）/ L7負荷分散（HTTPレベル）
  - アルゴリズム: ラウンドロビン/最小接続/IPハッシュ
  - VRRP: デフォルトゲートウェイの冗長化

- **SDN(Software-Defined Networking)**
  - コントロールプレーンとデータプレーンの分離
  - OpenFlow: 中央コントローラがフロー制御ルールを配布
  - **実務活用**: クラウド環境のネットワーク自動化

### T5. セキュリティの深化（午後必須）

- **インシデントレスポンス手順**
  1. 検知・トリアージ
  2. 封じ込め（ネットワーク隔離）
  3. 根絶（マルウェア除去・パッチ適用）
  4. 復旧（システム復元・監視強化）
  5. 事後分析（再発防止策）

- **ゼロトラスト**
  - 「信頼しない、常に検証」原則
  - マイクロセグメンテーション / 最小権限アクセス / 多要素認証
  - **原理**: 境界防御モデル（城と堀）の限界→全通信を検証

- **午後記述の頻出パターン**
  - 「なぜ〜が必要か」→ 脅威×脆弱性を特定し対策理由を述べる
  - 「どのような対策を講じるか」→ 具体的な技術名+効果を記述
  - FWルール設計: 送信元IP/宛先IP/ポート番号/アクション（許可/拒否）

### T6. システムアーキテクチャ（午後選択）

- **非機能要件の設計**
  - 性能: スループット目標→負荷試験→ボトルネック特定→チューニング
  - 可用性: 目標稼働率→冗長構成設計→フェイルオーバー手順
  - 拡張性: スケールアップ（垂直）vs スケールアウト（水平）
  - **実務活用**: 要件定義時の非機能要件チェックリスト

- **キャパシティプランニング**
  - ピーク時のリクエスト数 × 1リクエストの処理時間 = 必要CPU時間
  - 待ち行列理論（M/M/1）で応答時間を見積もり

### T7. ソフトウェア開発の深化

- **オブジェクト指向の深化**
  - SOLID原則: SRP/OCP/LSP/ISP/DIP
  - デザインパターン: Strategy/Observer/Factory/Singleton/Adapter
  - **原理**: 変更に強い設計→インタフェースに依存し実装に依存しない

- **アジャイル開発の深化（Ver.7.0追加強化）**
  - **スクラム**: スプリント計画→デイリースクラム→スプリントレビュー→レトロスペクティブ
  - **カンバン**: WIP制限、リードタイム短縮、フロー効率
  - **DevOps**: CI(継続的インテグレーション) → CD(継続的デリバリー/デプロイ)
  - **IaC(Infrastructure as Code)**: Terraform/CloudFormation等でインフラをコード管理

## マネジメント系（問51-60、10問）

### M1. プロジェクトマネジメントの深化

- **EVM（アーンドバリューマネジメント）計算の応用**
  - **EAC(完了時見積り)** = BAC/CPI（今のペースなら最終コスト）
  - **ETC(残作業見積り)** = EAC - AC
  - **TCPI(残作業効率指標)** = (BAC-EV)/(BAC-AC)（目標達成に必要なCPI）
  - **VAC(完了時差異)** = BAC - EAC

- **スケジュール管理**
  - **CPM(クリティカルパスメソッド)**: 最長経路=プロジェクト全体期間
  - **PERT**: 3点見積り（楽観+最頻+悲観）。期待値=(O+4M+P)/6
  - **クラッシング**: 資源追加で短縮（コスト増）
  - **ファストトラッキング**: 工程の並行実施（リスク増）

- **リスクマネジメントの深化**
  - **定量的リスク分析**: 期待金額値(EMV)=確率×影響額。デシジョンツリー
  - **モンテカルロシミュレーション**: 確率分布を使った全体工期/コストのシミュレーション

### M2. サービスマネジメントの深化

- **ITIL 4フレームワーク**（Ver.7.0追加強化）
  - **サービスバリューチェーン**: 計画→改善→エンゲージ→設計&トランジション→取得/構築→提供&サポート
  - **34プラクティス**: 一般管理(14) + サービス管理(17) + 技術管理(3)
  - **原理**: 「価値」の共創。サービスプロバイダとコンシューマの関係

- **可用性管理の深化**
  - **SPoF(単一障害点)分析**: 故障すると全体停止するコンポーネントの特定
  - **BIA(ビジネスインパクト分析)**: 業務中断の影響度評価→RTO/RPO設定
  - **DR(ディザスタリカバリ)**: ホットサイト/ウォームサイト/コールドサイト

## ストラテジ系（問61-80、20問）

### S1. 経営戦略の深化

- **IT投資評価**
  - **ROI** = (利益-投資額)/投資額 × 100%
  - **NPV(正味現在価値)** = Σ(CFₜ/(1+r)ᵗ) - 初期投資
    - CFₜ=t年目のキャッシュフロー、r=割引率
    - NPV>0 → 投資価値あり
  - **IRR(内部収益率)**: NPV=0となる割引率。IRR>資本コスト → 投資価値あり
  - **TCO(総所有コスト)**: 初期費用+運用費+教育費+廃棄費の総額
  - **実務活用**: システム投資の経済的妥当性評価

- **EA(エンタープライズアーキテクチャ)**
  - 4層: ビジネス→データ→アプリケーション→テクノロジ
  - As-Is(現状) → To-Be(あるべき姿) → ギャップ分析 → 移行計画

### S2. システム戦略の深化

- **要件定義プロセス**
  - 業務要件 → システム要件 → ソフトウェア要件
  - 機能要件 + 非機能要件（性能/信頼性/セキュリティ/移行/運用保守）
  - **RFI(情報提供依頼書)** → **RFP(提案依頼書)** → ベンダー選定

- **SoE vs SoR**
  - SoR(Systems of Record): 基幹系（正確性・安定性重視）
  - SoE(Systems of Engagement): 顧客接点系（スピード・UX重視）
  - **バイモーダルIT**: 両方を使い分ける経営戦略

### S3. 法務の深化

- **契約形態の比較**
  | 契約 | 指揮命令 | 成果物責任 | 適用場面 |
  |---|---|---|---|
  | 請負 | なし | あり（瑕疵担保） | 開発・構築 |
  | 準委任 | なし | なし（善管注意義務） | 運用・コンサル |
  | 派遣 | あり | なし | 人材補充 |
  - **偽装請負**: 請負契約なのに発注者が直接指揮命令→違法

- **知的財産の深化**
  - プログラムの著作権: 著作物として保護（特許とは別。登録不要）
  - 職務著作: 法人の業務で作成→法人が著作者
  - OSS利用時: GPLのコピーレフト条件→派生物も同ライセンスで公開義務

# 午後試験対策（記述式）

## 回答技法

### 基本原則
1. **問題文から抽出**: 答えの根拠は必ず問題文中にある。下線・注記に印をつけながら読む
2. **問題文の表現をそのまま使う**: 言い換えによる減点を回避
3. **字数制限厳守**: 20-40字が多い。キーワード先行で組み立て
4. **空欄禁止**: 部分点の可能性があるため必ず記入

### 設問タイプ別パターン
| 設問タイプ | 回答パターン |
|---|---|
| 「〜の理由を述べよ」 | 「〜であるため」で終わる |
| 「〜を具体的に述べよ」 | 技術名+効果を明記 |
| 「〜の問題点を指摘せよ」 | 現状の脆弱性/不備を特定 |
| 「〜の対策を述べよ」 | 具体的手段+期待効果 |
| 「〜を図で示せ」 | ER図/ネットワーク構成図等 |

### 時間配分（150分）
- 問1（必須セキュリティ）: 25分
- 選択4問: 各25分 = 100分
- 見直し: 25分
- **鉄則**: 1問に固執しない。25分経過で次へ

## 分野別午後対策

### 問1. 情報セキュリティ（必須）
- **頻出**: FWルール設計、インシデント対応、認証方式設計、暗号化通信
- **ポイント**: 「どこが攻撃可能か」→「何を守るべきか」→「どう守るか」の3段論法

### 問5. ネットワーク
- **頻出**: DNS設計、VLAN構成、VPN構築、トラフィック解析
- **ポイント**: ネットワーク図を正確に読み取り、パケットの流れを追跡

### 問6. データベース
- **頻出**: ER図→スキーマ変換、SQL記述、正規化判定
- **ポイント**: E-R図の多重度(1:N等)を正確に読み取る。SQL は構文ミスに注意

### 問9. プロジェクトマネジメント
- **頻出**: EVM計算、遅延対策、リスク対応
- **ポイント**: 計算問題+状況判断。問題文中の数値を使って計算→判断を記述

### 問11. システム監査
- **頻出**: 統制の不備指摘、監査手続き、改善勧告
- **ポイント**: 「統制が有効でない理由」を問題文から特定→「あるべき姿」を記述
