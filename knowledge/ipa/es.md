# 組込み・IoTシステム設計 — 開発リファレンス

# 1. マイコンアーキテクチャ

## 1.1 プロセッサ
- **ARM Cortex-Mシリーズ**: 組込み分野で圧倒的シェア。低消費電力+高性能
  - Cortex-M0/M0+: 超低消費電力（IoTエンド）
  - Cortex-M3/M4: 汎用MCU（DSP命令/FPU付き）
  - Cortex-M7: 高性能MCU（パイプライン6段）
  - Cortex-M33/M55: TrustZone対応、ML推論向けMVE拡張
- **RISC-V**: オープンソースISA。ライセンス料不要。モジュール化された命令セット（RV32I+拡張）
  - カスタム命令追加が容易（特定アプリケーション向け最適化）
  - ESP32-C3/C6など商用製品で採用拡大
- **ノイマン型 vs ハーバード型**
  - ノイマン: 命令とデータが同じバス（単純だがボトルネック）
  - ハーバード: 命令/データバス分離（並列フェッチ可能。DSP/MCUで主流）

## 1.2 割込み
- **外部割込み**: I/O完了、タイマー、通信受信
- **内部割込み(例外)**: ゼロ除算、未定義命令、メモリ保護違反
- **割込み優先度**: NVIC(Nested Vectored Interrupt Controller)で優先度管理
- **ネスト割込み**: 高優先度割込みが低優先度割込み処理中に割り込み可能
- **割込み応答時間**: 割込み発生→ISR実行開始までの時間（レイテンシ）
  - コンテキスト保存+ベクタフェッチ+パイプラインフラッシュ
- **設計上の注意**: 割込み禁止区間が長すぎるとリアルタイム性を損なう。ISRは最小限の処理に留め、重い処理はタスクに委譲する
- **実務パターン**: 割込みでフラグ/キューにポスト → メインループまたはRTOSタスクでデキューして処理（Top Half / Bottom Half パターン）

## 1.3 DMA (Direct Memory Access)
- CPU介在なしでメモリ↔周辺機器間のデータ転送
- **サイクルスチール**: DMAがバスを1サイクルずつ借用（CPU実行とインターリーブ）
- **バーストモード**: DMAがバスを占有して一括転送（高速だがCPU停止）
- **DMA転送時間の計算**: データ量(byte) / バス幅(byte) × バスサイクル時間
- **実務活用**: ADC連続サンプリング→メモリ直接格納でCPU負荷ゼロ
- **ダブルバッファリング**: DMAが片方のバッファに書き込み中にCPUがもう一方を処理（データ欠損防止）
- **リンクドリスト(Scatter-Gather)DMA**: 不連続メモリ領域への転送をチェーン実行

## 1.4 バス
- **AHB(Advanced High-performance Bus)**: 高速バス（CPU/メモリ/DMA）
- **APB(Advanced Peripheral Bus)**: 低速周辺バス（GPIO/UART/SPI）
- **バスアービトレーション**: 複数マスタの競合解決（固定優先度/ラウンドロビン）
- **AXI(Advanced eXtensible Interface)**: 高性能SoC向け。リード/ライト独立チャネル、アウトオブオーダー完了対応

# 2. リアルタイムOS (RTOS)

## 2.1 タスク管理
- **タスク状態遷移**:
  - 休止(Dormant) → 実行可能(Ready) → 実行(Running) → 待ち(Blocked) → 実行可能
  - ディスパッチ: Ready→Running / プリエンプション: Running→Ready
- **スケジューリング**: 優先度ベースプリエンプティブが基本
  - 同一優先度: ラウンドロビン or FIFO
  - **レートモノトニック(RM)**: 周期短い→高優先度（最適な固定優先度割当て）
  - **EDF(Earliest Deadline First)**: デッドラインが近い→高優先度（動的、CPU利用率100%まで理論保証）
- **主要RTOS**: FreeRTOS（AWS IoT統合）/ Zephyr（Linux Foundation、マルチアーキテクチャ）/ TOPPERS/ASP（日本発、μITRON準拠）/ Azure RTOS(ThreadX)

## 2.2 タスク間通信・同期
| 機構 | 用途 | 特徴 |
|---|---|---|
| セマフォ | 排他制御/同期 | P(wait)/V(signal)操作 |
| ミューテックス | 排他制御専用 | 所有権あり（取得者のみ解放可能） |
| イベントフラグ | 複数条件の待ち合わせ | AND/OR待ち |
| メールボックス | メッセージ1件の送受信 | 固定長 |
| メッセージキュー | メッセージ複数件の送受信 | FIFO、可変長 |

## 2.3 デッドロック
- **発生条件4つ**: 相互排除/保持と待ち/横取り不可/循環待ち
- **典型例**: タスクAがリソース1を保持しリソース2を待つ、タスクBがリソース2を保持しリソース1を待つ
- **防止策**: 資源獲得順序の統一（全タスクで同じ順番でロック取得）
- **検出**: 資源割当てグラフの循環検出
- **実務対策**: タイムアウト付きロック取得で永久待ちを回避、ウォッチドッグタイマーで異常検出

## 2.4 優先度逆転 (Priority Inversion)
- **現象**: 低優先度タスクLが資源Rを保持→高優先度タスクHがR待ち→中優先度タスクMがLをプリエンプト→Hが待ち続ける
- **対策**:
  - **優先度継承**: 低タスクの優先度を一時的に高タスクの優先度に引き上げ
  - **優先度上限(シーリング)**: 資源に紐づく最高優先度まで引き上げ
- **実例**: 1997年Mars Pathfinder探査機で発生→優先度継承で修正

## 2.5 リアルタイム性の分析
- **WCET(Worst Case Execution Time)**: 最悪実行時間
- **応答時間**: 割込み発生→処理完了までの時間
- **スケジューラビリティ分析**: RM条件 Σ(Ci/Ti) ≤ n(2^(1/n)-1)
  - Ci=実行時間、Ti=周期、n=タスク数
  - n→∞でln2≈0.693が上限（CPU利用率69.3%まで保証）
- **実務での検証手法**: ロジックアナライザでISR/タスク実行時間を計測、ジッタ（時間変動）を統計分析

# 3. HW/SW協調設計

## 3.1 パーティショニング
| 判断基準 | HW実装 | SW実装 |
|---|---|---|
| 速度要件 | リアルタイム制約が厳しい | 余裕がある |
| 並列性 | 並列処理が必要 | 逐次処理で可 |
| 変更頻度 | 変更なし/少 | 変更多い |
| コスト | 大量生産(ASIC) | 少量多品種 |
| 消費電力 | 最適化可能 | 汎用プロセッサ依存 |

## 3.2 実装技術
- **ASIC**: 専用IC。大量生産で低コスト。変更不可
- **FPGA**: プログラマブルロジック。少量生産/プロトタイプ向き。再構成可能
  - **部分再構成**: 動作中にFPGAの一部領域だけ書き換え（停止不要）
  - **HLS(High Level Synthesis)**: C/C++からRTLを自動生成（開発効率向上）
- **SoC**: CPU+周辺機能を1チップに統合
- **協調シミュレーション**: HW(VHDL/Verilog)とSW(C)を同時検証
- **MPSoC**: 複数プロセッサコア+FPGAファブリックを1チップに統合（Xilinx Zynq等）

# 4. センサ・アクチュエータ

## 4.1 AD変換
- **分解能**: nビット → 2^n段階（12bit=4096段階）
- **サンプリング定理(ナイキスト)**: サンプリング周波数 ≥ 信号周波数の2倍
  - **エイリアシング**: ナイキスト条件を満たさない→偽信号発生。対策: アンチエイリアシングフィルタ
- **変換方式**:
  | 方式 | 速度 | 精度 | コスト |
  |---|---|---|---|
  | フラッシュ型 | 最速 | 低bit | 高い |
  | 逐次比較型(SAR) | 中速 | 中精度 | 中 |
  | ΔΣ型 | 低速 | 高精度 | 低 |
- **量子化誤差**: ±1/2 LSB（最小分解単位の半分）
- **実務Tips**: ADC入力にRCフィルタを置いてノイズ除去、リファレンス電圧の安定化が精度に直結

## 4.2 DA変換
- R-2Rラダー型: 抵抗ネットワークで電圧分割
- PWM+LPF: パルス幅変調→ローパスフィルタで平滑化

## 4.3 PWM（パルス幅変調）
- **デューティ比** = ON時間/周期 × 100%
- 平均電圧 = Vmax × デューティ比
- **用途**: DCモータ速度制御/LED調光/サーボ角度制御/ヒーター温度制御
- **計算例**: 5V電源、デューティ比60% → 平均3V出力

## 4.4 センサ種類
| センサ | 検出対象 | 原理 |
|---|---|---|
| サーミスタ | 温度 | 抵抗値の温度変化 |
| 加速度センサ | 加速度 | MEMS静電容量変化 |
| ジャイロ | 角速度 | コリオリ力 |
| 赤外線 | 人体/距離 | 赤外線放射/反射 |
| 超音波 | 距離 | 伝播時間 |
| ロードセル | 荷重 | ひずみゲージ |
| ToF(Time of Flight) | 距離 | レーザー光の往復時間 |
| 気圧センサ | 気圧/高度 | ピエゾ抵抗/静電容量 |

## 4.5 モータ制御
- **DCモータ**: PWMで速度制御、Hブリッジで正逆転
- **ステッピングモータ**: パルス数で角度制御（オープンループ）、マイクロステッピングで微細制御
- **BLDCモータ**: ホールセンサ/逆起電力で転流タイミング検出、FOC(磁界方向制御)で高効率運転
- **PID制御**: 目標値との偏差に比例(P)+積分(I)+微分(D)でフィードバック制御

# 5. 通信インタフェース

## 5.1 有線シリアル通信
| プロトコル | 線数 | 同期/非同期 | 全二重/半二重 | 速度 | マスタ/スレーブ |
|---|---|---|---|---|---|
| UART | 2(TX/RX) | 非同期 | 全二重 | ~数Mbps | P2P |
| SPI | 4(MOSI/MISO/SCK/SS) | 同期 | 全二重 | ~数十Mbps | 1マスタ |
| I2C | 2(SDA/SCL) | 同期 | 半二重 | 100k~3.4M | マルチマスタ |
| CAN | 2(差動) | 非同期 | 半二重 | ~1Mbps | マルチマスタ |

- **UART**: スタートビット(1)+データ(8)+パリティ(0/1)+ストップ(1/2)。ボーレート合意必要
- **SPI**: クロック極性(CPOL)とクロック位相(CPHA)の4モード。SS線でスレーブ選択
- **I2C**: 7bit/10bitアドレスでデバイス識別。ACK/NACKでハンドシェイク。プルアップ抵抗必要
- **CAN**: CSMA/CA+優先度アービトレーション（ID値が小さいほど高優先度）。差動信号でノイズ耐性
- **選定指針**: 高速&近距離→SPI、少ピン&複数デバイス→I2C、長距離&ノイズ環境→CAN、デバッグ/ログ出力→UART

## 5.2 無線通信
| 規格 | 距離 | 速度 | 消費電力 | 用途 |
|---|---|---|---|---|
| BLE | ~100m | ~2Mbps | 極低 | ウェアラブル/IoT |
| Zigbee | ~100m | 250kbps | 低 | センサNW/メッシュ |
| Wi-Fi | ~100m | ~数Gbps | 高 | 高速通信 |
| LoRa | ~15km | ~50kbps | 極低 | 広域IoT |
| Wi-SUN | ~1km | ~300kbps | 低 | スマートメータ |
| Thread | ~100m | 250kbps | 低 | スマートホーム(IPv6メッシュ) |
| Matter | - | - | - | スマートホーム統一アプリ層(Thread/Wi-Fi上) |

## 5.3 車載ネットワーク
- **CAN**: 標準（11bit ID, 1Mbps）/ 拡張（29bit ID）。エラーフレーム自動再送
- **CAN FD**: データフレーム最大64byte（従来8byte）、最大8Mbps
- **LIN**: シングルマスタ、19.2kbps。低コストのサブネット
- **FlexRay**: タイムトリガ型、10Mbps。高信頼（x-by-wire用）
- **Automotive Ethernet**: 100BASE-T1/1000BASE-T1、ADAS/自動運転のデータ伝送に採用拡大

# 6. 安全設計（機能安全）

## 6.1 FMEA (Failure Mode and Effects Analysis)
- **ボトムアップ**: 各部品の故障モード→システム影響→重大度評価
- **RPN(Risk Priority Number)** = 発生度(O) × 影響度(S) × 検出度(D)
- RPNが高い項目から優先的に対策
- **DFMEA(設計FMEA)**: 設計段階で実施 / **PFMEA(工程FMEA)**: 製造工程で実施

## 6.2 FTA (Fault Tree Analysis)
- **トップダウン**: 望ましくない頂上事象→原因をAND/ORゲートで展開
- 定量的な故障確率の算出が可能
- **最小カットセット**: 頂上事象を引き起こす最小の基本事象の組合せ

## 6.3 IEC 61508 / ISO 26262
- **IEC 61508**: 電気/電子/プログラマブル電子系の機能安全国際規格
- **SIL(安全度水準)**: SIL1~SIL4（高いほど厳格な設計・検証要求）
- **ISO 26262**: 自動車固有の機能安全規格（IEC 61508派生）
  - ASIL(Automotive SIL): ASIL-A~D（Dが最高）
  - **HARA(Hazard Analysis and Risk Assessment)**: 危険事象を特定しASILを割り当て
- **IEC 62443**: 制御システムセキュリティ（セーフティ×セキュリティの融合）
- **Vモデル**: 要件定義↔受入テスト、設計↔結合テスト、詳細設計↔単体テストを対応させた開発プロセス

## 6.4 安全設計パターン
| パターン | 内容 | 例 |
|---|---|---|
| フェールセーフ | 故障時に安全側に倒す | 信号機→赤に固定 |
| フェールソフト | 故障時に縮退運転 | エンジン片方停止→低速飛行 |
| フォールトトレランス | 故障しても機能維持 | 冗長系による継続 |
| フォールトアボイダンス | 故障を未然に防止 | 高信頼部品の使用 |
| ウォッチドッグタイマー | ソフト暴走を検出しリセット | 一定時間内にクリアしないとリセット |
| ロックステップ | 2CPUが同一処理、比較で故障検出 | セーフティMCU(Infineon AURIX等) |

# 7. IoTシステム設計

## 7.1 エッジコンピューティング
- **原理**: データ発生源近くで処理→レイテンシ低減+帯域節約
- **エッジ vs クラウド**: リアルタイム判断はエッジ、大規模分析はクラウド
- **フォグコンピューティング**: エッジとクラウドの中間層
- **TinyML**: MCU上で機械学習推論を実行（TensorFlow Lite for Microcontrollers等）

## 7.2 OTA (Over-The-Air) 更新
- ファームウェアのリモート更新
- **A/Bパーティション**: 更新失敗時のロールバック用にダブルバンク
- **署名検証**: 不正ファームウェアの防止（コードサイニング）
- **差分更新**: 全体でなく差分のみ送信→帯域節約
- **SUIT(Software Updates for IoT)**: IETF標準のIoTファームウェア更新マニフェスト形式

## 7.3 組込みセキュリティ
- **セキュアブート**: 起動時にファームウェアの署名を検証（Root of Trust起点の信頼チェーン）
- **TEE(Trusted Execution Environment)**: 隔離された安全な実行環境（ARM TrustZone等）
- **暗号ハードウェア**: AESアクセラレータ/TRNG(真性乱数生成器)
- **IEC 62443**: 産業用制御システムのセキュリティ要件
- **PSA Certified**: ARM提唱のIoTセキュリティ認証フレームワーク

## 7.4 IoTプロトコルスタック
- **MQTT**: Pub/Subモデル、QoS 0/1/2、軽量でブローカー経由（AWS IoT Core/Azure IoT Hub対応）
- **CoAP**: UDP上のRESTライク通信、リソース制約デバイス向け
- **LwM2M**: デバイス管理プロトコル（OTA/構成管理/監視を標準化）

# 8. 組込みLinux

## 8.1 基本構成
- **ブートシーケンス**: ROM→ブートローダ(U-Boot)→カーネル→rootfs→アプリケーション
- **BSP(Board Support Package)**: ハードウェア固有の初期化コード
- **デバイスドライバ**: カーネルモジュール方式（動的ロード/アンロード）
- **デバイスツリー**: ハードウェア構成をデータ構造で記述（ドライバのHW依存を排除）
- **ビルドシステム**: Yocto Project / Buildroot でカスタムLinuxイメージを構築

## 8.2 リアルタイム拡張
- **PREEMPT_RT**: Linuxカーネルにリアルタイム性を追加するパッチ（6.xで本線統合進行中）
- **Xenomai**: デュアルカーネル方式のリアルタイム拡張
- **原理**: 標準Linuxはソフトリアルタイム。ハードリアルタイムには拡張が必要
- **選定基準**: レイテンシ要件<100μs→Xenomai/専用RTOS、<1ms→PREEMPT_RT、>1ms→標準Linux

# 9. 電源・消費電力設計

## 9.1 低消費電力設計
- **スリープモード**: Run→Sleep→Deep Sleep→Shutdown（復帰時間とのトレードオフ）
- **クロックゲーティング**: 未使用ペリフェラルのクロック停止
- **DVFS(Dynamic Voltage and Frequency Scaling)**: 負荷に応じて電圧/周波数を動的変更
- **エナジーハーベスティング**: 太陽光/振動/温度差から微小電力を回収

## 9.2 電源設計
- **LDO vs DC-DC**: LDOは低ノイズだが効率が低い、DC-DCは高効率だがスイッチングノイズ
- **デカップリングコンデンサ**: 各ICのVCC-GND間に配置（高周波ノイズ除去）
- **電源シーケンシング**: 複数電源の投入順序管理（コア→I/O→周辺の順等）
