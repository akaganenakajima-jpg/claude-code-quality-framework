## 1. 個人情報保護

### 1.1 個人情報保護法（日本）

**基本概念**:
- **個人情報**: 生存する個人に関する情報で、特定の個人を識別できるもの（氏名、生年月日等）
- **個人識別符号**: 顔認識データ、指紋、マイナンバー、運転免許番号等
- **要配慮個人情報**: 人種、信条、病歴、犯罪歴、障害等（取得に本人同意が原則必要）

**2022年改正の重要ポイント**:
- **仮名加工情報**: 他の情報と照合しない限り個人を識別できないように加工。内部分析に限り利用目的変更可
- **個人関連情報**: Cookie、IPアドレス等。第三者提供時に本人同意確認が必要
- **越境移転**: 外国の第三者への提供時に移転先国の制度情報を本人に提供

**匿名加工情報 vs 仮名加工情報**:

| 項目 | 匿名加工情報 | 仮名加工情報 |
|---|---|---|
| 個人の再識別 | 不可（不可逆） | 他情報との照合で可能 |
| 第三者提供 | 可能（届出必要） | 不可（内部利用のみ） |
| 利用目的変更 | 自由 | 変更可能（内部に限る） |
| 本人の権利 | 適用外 | 開示等請求は適用外 |
| 加工基準 | 厳格（特異値除去等） | 比較的緩い |

### 1.2 GDPR (General Data Protection Regulation)

**基本原則（第5条）**:
1. **適法性・公正性・透明性**: 合法的根拠に基づき、透明に処理
2. **目的限定**: 明確・正当な目的でのみ収集
3. **データ最小化**: 目的に必要最小限のデータ
4. **正確性**: 正確かつ最新に維持
5. **記録保存の制限**: 必要な期間のみ保持
6. **完全性・機密性**: 適切なセキュリティ措置

**データ主体の権利**:
- **アクセス権**: 自分のデータの開示請求
- **訂正権**: 不正確なデータの修正
- **消去権（忘れられる権利）**: データの削除要求
- **データポータビリティ権**: 機械読取可能な形式でデータを受け取る/他者に移転
- **異議申立権**: プロファイリング等への異議
- **自動意思決定拒否権**: 完全自動化された個人に法的効果を及ぼす意思決定を拒否

**日本 vs GDPR 比較**:

| 項目 | 日本（個人情報保護法） | EU（GDPR） |
|---|---|---|
| 適用範囲 | 日本国内の事業者 | EU域内のデータ主体に関する処理 |
| 制裁金 | 最大1億円（法人） | 最大2,000万EUR or 全世界売上4% |
| DPO | 義務なし | 一定条件で任命義務 |
| DPIA | 明文規定なし | 高リスク処理時に義務 |
| 越境移転 | 同意 or 基準適合体制 | 十分性認定 or SCC等 |

### 1.3 匿名化技術

#### k-匿名性

- **定義**: データセット内の各レコードが、少なくとも k-1 個の他のレコードと準識別子（年齢、性別、郵便番号等）が同一
- **手法**: 一般化（年齢30→30代）、抑圧（郵便番号の下桁をマスク）
- **限界**: 同質性攻撃（同じ準識別子グループの機微属性が全て同一値なら推測可能）

#### l-多様性

- **定義**: 各等価クラス（k-匿名性グループ）内の機微属性が少なくとも l 種類の値を持つ
- **限界**: 歪度攻撃（l種類でも分布が偏っていれば推測可能）

#### t-近接性

- **定義**: 各等価クラス内の機微属性の分布が、全体分布からの距離が t 以内
- **距離**: Earth Mover's Distance (EMD) を使用
- **最も強い保証だが、データの有用性が大きく低下する場合がある**

#### 差分プライバシー (Differential Privacy)

**定義**: データベースに1レコード追加/削除しても、クエリ結果の分布がほぼ変わらない
- **ε-差分プライバシー**: P(M(D₁) ∈ S) ≤ e^ε × P(M(D₂) ∈ S)
  - ε（プライバシー予算）が小さいほど強い保護
- **メカニズム**:
  - ラプラス機構: クエリ結果にラプラスノイズを加算（数値クエリ）
  - 指数メカニズム: 確率的に最適な回答を選択（カテゴリクエリ）
  - ガウスメカニズム: ガウスノイズ（(ε,δ)-差分プライバシー）
- **合成定理**: 複数クエリのε予算は加算される（プライバシー消費の管理が必要）
- **局所差分プライバシー**: データ収集前にノイズを付加（Apple, Google等が採用）

### 1.4 実務活用: プライバシー保護の実装

**プライバシー影響評価 (PIA/DPIA) のステップ**:
1. 処理内容の記述（目的、データ種類、フロー）
2. 必要性と比例性の評価（データ最小化の確認）
3. リスク特定（再識別リスク、漏洩リスク、目的外利用リスク）
4. リスク軽減策の設計（匿名化、アクセス制御、暗号化）
5. 残存リスクの受容判断

**データ最小化チェックリスト**:
- 本当にその個人情報が分析に必要か
- 集計・匿名化した状態で目的を達成できないか
- 保持期間は最小限か
- アクセス権限は必要最小限か

## 2. AI倫理

### 2.1 公平性 (Fairness)

#### 公平性の定義（相互に矛盾しうる）

| 指標 | 定義 | 例 |
|---|---|---|
| **統計的パリティ** (Demographic Parity) | P(Ŷ=1\|A=0) = P(Ŷ=1\|A=1) | 男女で合格率が同じ |
| **機会の平等** (Equal Opportunity) | P(Ŷ=1\|Y=1,A=0) = P(Ŷ=1\|Y=1,A=1) | 有資格者の合格率が同じ |
| **等しいオッズ** (Equalized Odds) | TPR, FPRが群間で等しい | 正例・負例ともに公平 |
| **予測的パリティ** (Predictive Parity) | P(Y=1\|Ŷ=1,A=0) = P(Y=1\|Ŷ=1,A=1) | 合格者の実際の成功率が同じ |
| **個人公平性** | 類似した個人は類似した結果を得る | 距離メトリクスで定義 |
| **反事実的公平性** | 保護属性を変えても結果が変わらない | 因果推論ベース |

**不可能性定理 (Kleinberg et al., 2016)**: 統計的パリティ・機会の平等・予測的パリティの3つを同時に満たすことは一般に不可能（基準率が異なる場合）

#### バイアスの種類

| バイアス | 段階 | 例 |
|---|---|---|
| **歴史的バイアス** | データ収集前 | 過去の差別が反映されたデータ |
| **表現バイアス** | データ収集 | 特定の集団がデータに過少/過多 |
| **測定バイアス** | 特徴量設計 | 代理変数（郵便番号→人種） |
| **集約バイアス** | モデリング | 一つのモデルで異質な集団を扱う |
| **評価バイアス** | 評価 | ベンチマークが特定集団に偏る |
| **展開バイアス** | デプロイ | 想定外のコンテキストでの使用 |

### 2.2 説明可能性 (Explainability)

**説明の種類**:
- **グローバル説明**: モデル全体の振る舞い（特徴量重要度、PDP）
- **ローカル説明**: 個別予測の根拠（SHAP、LIME）→ `modeling.md` §5 参照
- **反事実説明**: 「Xが○○だったら結果が変わる」（最小変更の提示）

**解釈可能なモデルの選択**:
- 高解釈性: 線形回帰、ロジスティック回帰、決定木、ルールベース、GAM
- 中解釈性: ランダムフォレスト（+SHAP）、GBDT（+SHAP）
- 低解釈性: 深層学習、アンサンブルスタッキング
- リスクの高い意思決定（医療、金融、司法）→ 解釈可能なモデル or SHAP必須

### 2.3 透明性 (Transparency)

**モデルカード (Model Cards for Model Reporting)**:
- **概要**: モデルの目的、アーキテクチャ、学習データ
- **性能**: 全体+サブグループ別の評価指標
- **制限事項**: 既知の限界、不適切な使用例
- **倫理的考慮**: バイアス評価結果、公平性指標

**データシート (Datasheets for Datasets)**:
- 動機、構成、収集プロセス、前処理
- 利用用途、配布、メンテナンス
- 倫理的・法的考慮事項

### 2.4 実務活用: バイアス検出と公平性監査

**バイアス検出パイプライン**:
1. 保護属性の特定（性別、年齢、人種、障害等）
2. データの分布確認（群間のサンプル数・基準率の差異）
3. モデル学習（保護属性を含む/含まないの両方）
4. 公平性指標の計算（SPD, EOD, DI等）
5. 不公平の検出 → 緩和策の適用

**緩和策**:
- **前処理**: データリバランス、保護属性の除去（代理変数に注意）
- **中間処理**: 制約付き最適化、敵対的学習（Adversarial Debiasing）
- **後処理**: 閾値調整（群ごとに異なる閾値）、拒否オプション分類

**ツール**:
- **Fairlearn** (Microsoft): 公平性指標の計算 + 緩和アルゴリズム
- **AI Fairness 360** (IBM): 70以上の公平性指標 + 緩和手法
- **What-If Tool** (Google): インタラクティブな公平性分析

## 3. データガバナンス

### 3.1 データオーナーシップ

**RACI マトリクス（データに関する役割）**:

| 役割 | 責任 |
|---|---|
| **データオーナー** | データの品質・セキュリティ・利用ルールの最終責任者（事業部門長） |
| **データスチュワード** | 日常の品質管理・メタデータ管理を実行（ドメイン担当者） |
| **データエンジニア** | パイプライン構築・技術的な品質保証 |
| **データサイエンティスト** | 分析・モデリング。データ利用規約を遵守 |
| **DPO (Data Protection Officer)** | プライバシー保護の監督（GDPR要件） |

### 3.2 データ利用規約・同意管理

**同意管理の原則（GDPR準拠）**:
- **自由意志**: 同意拒否でサービス利用に不利益がない
- **具体性**: 目的ごとに個別の同意
- **情報提供**: 何のデータを、何の目的で、誰が処理するか明示
- **明確性**: 積極的行為（チェックボックスON等）。事前チェック済みは不可
- **撤回可能**: いつでも同意を撤回でき、撤回は付与と同じくらい簡単

**Consent Management Platform (CMP)**:
- Cookie同意バナー、プリファレンスセンター
- 同意の記録・監査証跡の保存
- 主要ツール: OneTrust, TrustArc, Cookiebot

### 3.3 オープンデータ・ライセンス

| ライセンス | 商用利用 | 改変 | 表示義務 | 同一条件 |
|---|---|---|---|---|
| CC0 (Public Domain) | 可 | 可 | なし | なし |
| CC BY | 可 | 可 | あり | なし |
| CC BY-SA | 可 | 可 | あり | あり（ShareAlike） |
| CC BY-NC | 不可 | 可 | あり | なし |
| CC BY-ND | 可 | 不可 | あり | - |

**政府オープンデータ**:
- 日本: e-Stat（統計データ）、DATA.GO.JP（政府データカタログ）
- 利用規約: 政府標準利用規約（CC BY 4.0 互換）

**データの利用可否判断フロー**:
1. ライセンスの確認（商用利用可か、改変可か）
2. 個人情報の有無（匿名化済みか）
3. 利用規約の制限事項（APIレートリミット、再配布条件）
4. 倫理的考慮（データの取得方法が適切か）

### 3.4 実務活用: データガバナンス導入

**段階的導入ロードマップ**:
1. **Phase 1**: データインベントリ作成（何のデータがどこにあるか）
2. **Phase 2**: データ分類（機密度レベルの付与）
3. **Phase 3**: アクセス制御の実装（RBAC / ABAC）
4. **Phase 4**: データカタログの整備（メタデータ管理）
5. **Phase 5**: 品質監視の自動化（データ品質ダッシュボード）
6. **Phase 6**: 継続的改善（定期レビュー、ポリシー更新）

**データ分類の例**:

| レベル | 分類 | 例 | 取り扱い |
|---|---|---|---|
| 1 | 公開 | プレスリリース、公開統計 | 制限なし |
| 2 | 社内 | 社内報、業務マニュアル | 社外持出禁止 |
| 3 | 機密 | 顧客リスト、売上データ | アクセス制限、暗号化 |
| 4 | 極秘 | 個人情報、医療データ | 厳格なアクセス制御、監査ログ必須 |
